- 有料キー、手動サインアップ、OAuth、CAPTCHA、物理デバイスは使わない。

## 上位3候補

### 1位: CSS `text-fit`のStable実測（29点）

- 2026-06-30のChrome 150 Stable直後で時宜が強い（[Chrome 150 release notes](https://developer.chrome.com/release-notes/150)、閲覧日2026-07-13）。
- 単なる構文紹介ではなく、日本語/英語、1行/複数行、幅変更、scale limit、`@supports` fallback、computed value、zoom caveatを一つのfixtureで対照できる。
- 仕様はcomputed `font-size`が変わらないことと、viewport依存boxのpage zoom問題が未解決であることを明記している（[CSS Text Level 5](https://drafts.csswg.org/css-text-5/#text-fit-property)、2026-06-04、閲覧日2026-07-13）。「見た目は変わるのにcomputed styleは同じ」という初心者が詰まりやすい観察を記事価値にできる。
- Chrome/Edge 150以外は未対応のため、Chrome 150+ gateとfallbackを記事の中心に置ける（[Can I use](https://caniuse.com/mdn-css_properties_text-fit)、閲覧日2026-07-13）。

### 2位: webpack coreのHTML/TypeScript support（28点）

- loader/pluginを減らす実務価値が高く、CLI中心で検証しやすい。
- ただし既存記事にNode native TypeScriptとVite 8があり、webpack固有の差を丁寧に限定する必要がある。
- 公式自身が実験的・部分的と明記しているため、patch/minor差で結果が変わるリスクがある（[Webpack 5.107](https://webpack.js.org/blog/2026-05-19-webpack-5-107/)、閲覧日2026-07-13）。

### 3位: `focusgroup`のキーボード操作比較（28点）

- roving tabindexのJavaScriptを宣言的HTMLへ置き換えるため、アクセシビリティとコード削減を同時に扱える。
- Chrome 150 Stableの新機能であり国内品質記事も注目している（[Chrome 150 release notes](https://developer.chrome.com/release-notes/150)、[ミツエーリンクス](https://www.mitsue.co.jp/knowledge/blog/qc/202607/01_1326.html)、ともに閲覧日2026-07-13）。
- 一方、正しいARIA role、Tab/矢印/Home/End、disabled itemなど検証面が広く、半日では`text-fit`より停止条件の設計が難しい。

## 選定テーマ

**Chrome 150のCSS `text-fit`を、固定font-size / `clamp()` fallbackと同じfixtureで比較し、日本語・英語、container幅、複数行、scale limit、computed value、page zoomをPlaywrightとスクリーンショットで実測する。**

想定タイトル:

> Chrome 150のCSS `text-fit`を日本語見出しで試す：`clamp()`との差とzoomの注意点

選定理由:

1. Chrome 150 Stable公開から約2週間で適時性が高い（[Chrome 150 release notes](https://developer.chrome.com/release-notes/150)、2026-06-30、閲覧日2026-07-13）。
2. CSSと少量のHTML/Playwrightでローカル・無料・認証不要に成功系とfallbackを再現できる。
3. 公式仕様が、構文だけでなくcomputed値とpage zoomに関する注意点を明示しており、挙動を測る記事にできる（[CSS Text Level 5](https://drafts.csswg.org/css-text-5/#text-fit-property)、2026-06-04、閲覧日2026-07-13）。
4. 国内の既存Qiita記事はStable前の草案紹介であり、公開時点では主要ブラウザ未実装としている（[Qiita](https://qiita.com/degudegu2510/items/25bbc55b846c3a402661)、2026-06-13、閲覧日2026-07-13）。Stable版の実測、日本語と英語の対照、zoom、fallbackへ焦点を移すことで差別化できる。
5. リポジトリ内の既存記事にresponsive typography主題はなく、CSS Anchor PositioningやGap Decorationsとも機能領域が異なる。

## 半日検証アウトライン

想定4時間30分。run段階は隔離ディレクトリで行い、結果はexecution logへ記録する。

### 0. capability gate（30分）

1. OS、Node、package manager、browser executable pathを記録する。
2. system ChromeまたはChrome for Testingを使い、browser major 150以上を用意する。既存system Chrome 149を上書きせず、隔離された実行ファイルを優先する。
3. Playwrightから実ブラウザをlaunchし、`navigator.userAgent`または`browser.version()`でmajor 150以上を確認する。
4. localhost fixtureで次を評価する。

```js
CSS.supports("text-fit", "grow") &&
CSS.supports("text-fit", "shrink consistent 50%")
```

5. browser launch失敗、major不足、または`CSS.supports`がfalseなら、その事実とstderrだけを記録してrun全体を停止する。CSSの表示結果は推測しない。

### 1. 最小fixture（35分）

- localhostだけで配信する静的HTML/CSSを作る。
- 同一文字列を次の3列で表示する。
  - `font-size`固定
  - `clamp()`を使うfallback
  - `text-fit`
- 外部fontロードを避け、実際に選択されたfont family、OS、browser versionをログへ残す。
- `@supports (text-fit: grow)`でenhancementし、非対応時にも固定font-sizeが読めるようにする。

### 2. 基本挙動のmatrix（70分）

- 文字列: 短い日本語、長い日本語、短い英語、空白を含む長い英語。
- container幅: 240px / 480px / 720px。
- 値: `grow consistent`、`shrink consistent 50%`、`grow per-line`、`grow per-line-all 200%`。
- 各caseでbounding box、overflow量、line rect数、computed `font-size`、スクリーンショットを保存する。
- 仕様上computed `font-size`は変化しないため、見た目の拡縮をcomputed styleだけで判定せず、同じ環境内のgeometryと画像を併用する（[CSS Text Level 5](https://drafts.csswg.org/css-text-5/#text-fit-property)、閲覧日2026-07-13）。

### 3. 動的変更とfallback（45分）

- JavaScriptでcontainer幅を3段階に切り替え、再layout後のgeometryを記録する。
- text内容を日本語から英語へ差し替え、fitが再計算されるか観察する。
- Chrome 149または別の非対応browserを起動できる場合だけ、`@supports` fallbackが読みやすく表示されることを補助確認する。起動できなければ非対応browser結果は未検証とする。

### 4. page zoomの注意点（45分）

- browser UIの実page zoomを100%と200%へ変更できたことを、browser側の状態または独立した観測で確認してから、固定幅boxとviewport依存幅boxのスクリーンショット・geometryを比較する。
- CDPのpage scale、device scale factor、CSS `zoom`をbrowser page zoomと同一視しない。
- 自動化でpage zoom変更を証明できない場合、このsubtestは停止して「仕様上の未解決事項のみ、実測なし」と明記する。
- page zoom時にviewport依存boxの見かけ幅が変わらず、fitされた文字も大きくならない可能性は仕様のopen issueであり、期待結果として固定しない（[CSS Text Level 5](https://drafts.csswg.org/css-text-5/#text-fit-property)、閲覧日2026-07-13）。

### 5. 反復と整理（45分）

- 核心matrixを新しいbrowser contextで最低3回繰り返す。
- case ID、browser version、viewport、指定値、geometry、画像path、pass/failではなく観察結果をJSON/CSVへまとめる。
- 成功例だけでなく、scale limitで余白/overflowが残るcase、computed値が見た目を表さないcase、fallback caseを記事候補の中心にする。

## 完了条件

- Chrome/Chromium 150以上のversionと`CSS.supports`を起動後に記録できている。
- 日本語/英語、3幅、`grow` / `shrink` / 複数行mode / percentage limitの結果が一次ログと画像に対応している。
- fixed / `clamp()` fallback / `text-fit`を同一fixture・同一browserで比較している。
- computed `font-size`だけで成功判定していない。
- page zoomを実測できなければ、その範囲を明確に未検証としている。
- 対応browserが限定されるため`@supports` fallbackを含む。

## リスクと停止条件

| リスク | 対応・停止条件 |
|---|---|
| ローカルChromeが149、cached Chromiumが141 | Chrome for Testing等で150以上を隔離導入する。download、executable確認、Playwright launch、major確認のどれかが失敗したらrun全体を停止し、結果を推測しない。 |
| `text-fit`がparseできても描画実装が使えない | `CSS.supports`に加えてfixture geometryを確認する。核心caseで一切差が出ず原因を特定できない場合は成功とせず停止する。 |
| font差でpixel値が変わる | 外部fontに依存せず、実際のfont/OS/browserを記録し、絶対pixel値の普遍化を避ける。同一環境内の対照だけを主張する。 |
| computed `font-size`が変わらない | 仕様どおりの可能性があるため失敗扱いしない。geometry、line rect、スクリーンショットを一次証拠にする。 |
| page zoomを自動化できない | CDP scaleから推測しない。zoom subtestだけを未検証として止め、核心matrixと混同しない。browser launch自体が失敗した場合はrun全体を停止する。 |
| 他browser未対応 | 互換性表を事実として引用するだけで表示を推測しない。起動できたbrowserだけを実測し、`@supports` fallbackを提示する。 |
| 草案が変わる | exact browser version、access date、指定したCSS値を固定して記録する。Editor’s Draftの記述を全browserの永久仕様として一般化しない。 |
| screenshotだけでは判定が主観的 | geometry/line rect/overflowの機械値と画像をcase IDで結び付ける。ただしused font-sizeを直接取得できない場合は算出値を事実扱いしない。 |

## 結論

選定テーマは、**Chrome 150のCSS `text-fit`を固定font-size / `clamp()` fallbackと同一fixtureで比較し、日本語・英語、container幅、複数行mode、scale limit、computed value、page zoomをChrome 150+のcapability gate付きで実測するハンズオン**とする。2026-06-30のStable公開直後で時宜があり、国内でも関心が確認できる一方、Stable実機の対照測定とzoom caveatには記事余地がある。無料・認証不要・localhost完結で、新人が「CSSを指定した」から一歩進み、「どの条件で何を証拠に効いたと判断するか」を学べる。run段階ではbrowser launchとmajor 150以上を必須gateとし、失敗時は表示結果を一切推測せず停止する。
