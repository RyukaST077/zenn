## 上位候補の詳細

### 1. Node 26 の `node:ffi` で JS から libc を呼んで詰まった記録

#### 概要
Node.js 26.1 で追加された実験的 `node:ffi` を使い、C++アドオンや外部npmなしで動的ライブラリの関数（libcの`strlen`/`abs`など）をJSから直接呼ぶ。`--experimental-ffi`（＋Permission Model時は`--allow-ffi`）のゲート、型指定（`u64`戻り値がBigIntになる等）、`using`での自動closeを最小例で検証する。

#### なぜ今よさそうか
Node 26 の目玉は Temporal に集中報道されがちだが、`node:ffi` は国内の実践ログがほぼ皆無。10月にLTS入り予定でキャッチアップ需要が立ち上がる直前。experimental かつ「unsafe」明記の機能なので、詰まり所＝記事価値になる。筆者の既存作（Node 26系の gate/gotcha ログ）と地続き。

#### 新人が検証しやすい理由
公式docsに`dlopen(null, {...})`の最小例があり、macOSならシステムlibcをそのまま叩ける。CLI一発で再現でき、フラグ忘れ・型ミスマッチ・ポインタ渡しの3点で必ず一度は転ぶので「詰まった→直した」が自然に書ける。

#### 想定読者
Node.jsは書けるがFFI/ネイティブ連携は未経験の新人〜2年目。「C++アドオンは重そう」と感じている層。

#### タイトル案
1. Node 26 の `node:ffi` で JS から libc の strlen を呼ぶまでに詰まったこと
2. C++アドオンなしでネイティブ関数を叩く：Node 26 の実験的FFIを新人が試してみた
3. `--experimental-ffi` を知らずに30分溶かした話（Node 26 node:ffi 入門）

#### 検証でやること
- `--experimental-ffi` の有無で挙動がどう変わるか観察
- `dlopen(null, {...})` で `strlen`/`abs` を呼ぶ
- 戻り値の型（`u64`→BigInt）と引数のポインタ渡しでの詰まり所を記録
- `using` による自動 `dlclose` と、明示 close の違い
- Permission Model（`--permission`）下で `--allow-ffi` なしだと弾かれることを確認

#### 検証手順
1. `node -v` で 26.1+ を確認（無ければ nvm 等で導入）
2. フラグ無しで `import { dlopen } from 'node:ffi'` を実行し、エラーメッセージを記録
3. `--experimental-ffi` 付きで最小 `strlen` 呼び出しを成功させる
4. 戻り値・引数の型を故意に間違えて挙動（クラッシュ/誤値）を観察
5. `--permission --allow-ffi` の有無を比較し、Permission Model の効果を確認

#### 記事構成案
1. はじめに（Node 26＝Temporalの陰に隠れたFFI）
2. なぜ試すのか（C++アドオンとの違い）
3. 事前に調べたこと（experimental / unsafe 警告）
4. 環境構築（Nodeバージョン確認）
5. 実際に試したこと（strlen/absを呼ぶ）
6. 詰まった点（フラグ忘れ・BigInt戻り値・ポインタ渡し）
7. Permission Model と `--allow-ffi`
8. 触ってみて分かったこと（安全性の注意）
9. どんな人に向いていそうか
10. まとめ

#### 差別化ポイント
Temporal一色の Node 26 記事群の中で、国内ほぼ未検証の FFI を新人目線の「詰まりログ」で扱う。過去記事の gate/gotcha シリーズと同系統で読者導線もある。

#### 参考リンク
- https://nodejs.org/api/ffi.html
- https://nodejs.org/en/blog/release/v26.1.0
- https://versionlog.com/nodejs/26/

#### 注意点
- experimental かつ unsafe。誤ったシグネチャでプロセスクラッシュし得るので、記事にはクラッシュ例も「危険」明示で載せる
- macOSでの `dlopen(null)` 挙動を実機確認し、Linux差異があれば明記
- APIが将来変わり得る旨を書き添える

### 2. Bun 1.3.13 の `bun test --changed`/`--parallel` を fixture で試す

#### 概要
Bun 1.3.13 でテストランナーに追加された `--parallel`（Nワーカーへ分散）、`--isolate`（ファイル毎にクリーンなglobal）、`--shard=M/N`（CI分割）、`--changed`（git差分の影響テストのみ）を、小さなfixtureテスト群で挙動確認する。特に `--changed` の import グラフ解析が「どこまで賢く絞るか」を実験する。

#### なぜ今よさそうか
Bun 1.3系はテスト機能とビルトインAPIが毎パッチで拡大中。過去記事は Bun.SQL(SQLite) のみで、テストランナーは国内未検証。Vitest/Jest との住み分けを気にする読者に刺さる。すべてローカル＆git自前で完結し、AI単独実行に最適。

#### 新人が検証しやすい理由
`bun test` に fixture を数本置いてフラグを足すだけ。`--changed` は 1ファイル変更→再実行で「絞られる/絞られない」を観察でき、結果が視覚的に分かりやすい。

#### 想定読者
JestやVitestを使ってきて、Bunのテストランナーが実用になるか知りたい新人〜2年目。

#### タイトル案
1. Bun 1.3.13 の `bun test --changed` はどこまで賢くテストを絞るのか試してみた
2. `--parallel`/`--isolate`/`--shard`/`--changed`：Bunのテスト新フラグを新人が全部触ってみた
3. Vitestしか知らない新人が `bun test --changed` の import グラフ解析を検証した

#### 検証でやること
- 依存関係のある fixture テスト（util.ts → a.test.ts / b.test.ts）を作る
- `--changed` で util.ts だけ変更→どのテストが走るか観察
- `--parallel=N` でワーカー数と実行時間の変化を記録
- `--isolate` 有無でグローバル状態の分離を確認
- `--shard=1/3` 等でファイル分割の分配ルールを観察

#### 検証手順
1. `bun -v` で 1.3.13+ を確認
2. 依存ありの最小テスト群を作成し `bun test` で全緑
3. `--changed` を git 差分あり/なしで比較
4. `--parallel` のワーカー数を変えて実行時間を計測
5. `--isolate`/`--shard` の挙動をログに残す

#### 記事構成案
1. はじめに（Bunのテストが充実してきた）
2. なぜ試すのか（Vitest/Jestとの比較観点）
3. 事前に調べたこと（各フラグの公式説明）

### 推奨テーマ
**Node 26 の `node:ffi` で JS から libc を呼んで詰まった記録**（候補#1）

### 理由
- **トレンド性**: Node 26 は現行・10月LTS入り予定で追い風。だが FFI は Temporal の陰で国内実践ログがほぼ空白＝先行者利益が大きい。
- **新人でも検証できる**: 公式docsの最小例（`dlopen(null,{strlen})`）をそのまま動かせ、macOSでlibcを直接叩ける。1日以内で再現可能。
- **Zenn読者に刺さる**: 「C++アドオンなしでネイティブ関数を呼べる」は驚きがあり、`--experimental-ffi`/`--allow-ffi`/BigInt戻り値/ポインタ渡しという“詰まり所”が読者の実益になる。
- **半日〜1日で完結**: CLIとJSファイル数本のみ。外部依存・認証・課金なし＝AIエージェント単独で最後まで検証可能。
- **完成イメージ**: 「フラグ忘れで即エラー→つけたら動く→型を間違えて誤値/クラッシュ→正しく直す→Permission Modelで弾かれる」を時系列ログ＋コマンド出力で追う、筆者お馴染みの gotcha ログ。

4. 型・引数を故意に誤り、挙動（誤値/クラッシュ）を安全に記録
5. `--permission --allow-ffi` 有無で Permission Model の効果を比較

「Node 26 の `node:ffi` で JS から libc の strlen を呼ぶまでに詰まったこと」

### 書く前に用意するもの
