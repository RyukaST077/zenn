### 採点メモ

1. **Deno 2.9テスト機能**: 2026-06-25公開で、主要機能を単一CLI・単一題材で成功と意図的失敗の両方にできる（I1〜I3、2026-07-11閲覧）。国内記事D1は変更点を列挙する速報であり、実行ログ中心の記事には差別化余地がある。
2. **Node test randomization**: 外部依存なしで「普段は通るが順序を変えると落ちる」テストを作れ、seed再現まで完結する（I4, I5、2026-07-11閲覧）。ただしNode.js 26系が必要なため適時性は高いがCurrent系の検証になる。
3. **Navigation API**: 2026年にBaseline Newly availableとなり、ブラウザ自動化でURL・history entry・DOMを数値的に確認できる（I7, I8、2026-07-11閲覧）。既存CSS記事とは違うが、ブラウザ3種導入が必要なら準備時間が増える。
4. **Deno lockfile移行**: 2.9固有で差別化しやすいが、npm/pnpmの両方を扱うと依存ダウンロードと差分分析が膨らむ（I1、2026-07-11閲覧）。
5. **Temporal**: 固定日時なら決定的で実務的だが、国内外に入門解説が多く、境界条件テストに絞らないとAPI紹介になりやすい（I6、2026-07-11閲覧）。
6. **TypeScript 7**: 調査日の3日前に正式版が出た最有力級の話題だが、国内ですでに速報と詳説があり（D2, D3、2026-07-11閲覧）、このリポジトリのoxlint記事でも互換性問題に触れている。実測規模を工夫しないと一般的な「10倍速」の再掲になりやすい。
7. **Deno supply-chain設定**: 実用性は高いが、パッケージ公開時刻やregistry応答という外部状態に結果が左右されやすく、再現性を固定する設計が難しい（I1、2026-07-11閲覧）。

## 上位3候補

### 1. Deno 2.9の組み込みテスト機能を小さなCLIで一巡する

- **記事の問い**: 文字列整形CLIを題材に、`Deno.test.each`、`t.assertSnapshot()`、retry/repeats、`--related`、coverage threshold、shardを追加ライブラリなしでどこまで再現できるか。
- **記事価値**: Deno 2.9はこれらを同時に組み込みテストランナーへ追加した（I1〜I3、2026-07-11閲覧）。機能カタログではなく、正常系、snapshot差分、flaky表示、coverage不足という観測可能な結果を並べられる。
- **差別化**: 既存のVitest記事はブラウザ描画と画像差分が主題。今回はCLI出力、テーブル駆動、影響テスト選択、品質ゲートが主題で、ブラウザもGUIも使わない。
- **主なリスク**: 全機能を欲張ると記事が散漫になる。snapshot、table test、retry、coverageを必須にし、`--related`とshardは時間が残った場合の追加検証に固定する。

### 2. Node.js 26.1で順序依存テストを発見・再現する

- **記事の問い**: 共有stateを持つわざと悪いテストが、通常順では通って`--test-randomize`で落ち、出力seedで同じ順序を再現できるか。
- **記事価値**: 失敗原因と修正前後が明確で、新人がテスト独立性を理解しやすい。公式はファイルと同一ファイル内のqueued testをランダム化し、seedを再利用できると説明している（I5、2026-07-11閲覧）。
- **主なリスク**: sequentialに`await`したsubtestはランダム化されないため、最初に公式どおりのeligibleな構造を作る必要がある（I5、2026-07-11閲覧）。

### 3. Navigation APIとHistory APIで最小SPAを比較する

- **記事の問い**: 同じ2画面SPAで、リンククリック、プログラム遷移、戻る・進む、遷移キャンセルを両APIで実装すると、コードと観測結果がどう変わるか。
- **記事価値**: Navigation APIは2026年1月からBaseline Newly availableで、SPA向けに設計されたHistory API後継として説明されている（I7, I8、2026-07-11閲覧）。Playwright等でURL、DOM、entry数を自動検証できる。
- **主なリスク**: ブラウザ版差と同一origin制約が結果に影響する。ローカルHTTP配信に固定し、利用できるブラウザが1種だけならクロスブラウザ一般化をしない。

## 選定テーマ

**Deno 2.9の組み込みテストランナーだけで、文字列整形CLIにsnapshot・table test・retry・coverage gateを導入し、成功と失敗の出力を記録する。**

### 想定タイトル

`Deno 2.9のテスト機能を一巡してみた：snapshot・test.each・retry・coverage gate`

### 選定理由

1. Deno 2.9は2026-06-25公開で、調査基準日の約2週間前という適時性がある（[Deno 2.9](https://deno.com/blog/v2.9)、2026-07-11閲覧）。
2. `t.assertSnapshot()`、`Deno.test.each`、retry/repeats、coverage thresholdは、1つの小さなCLIと決定的な入力だけで確認できる（[Deno 2.9](https://deno.com/blog/v2.9)、2026-07-11閲覧）。
3. Deno CLIとローカルファイルだけを使う計画なので、外部アカウント、APIキー、OAuth、物理デバイス、GUIを必要としない。
4. 国内のD1はリリース変更点の速報であり、snapshot更新前後、flaky表示、coverage不足の非ゼロ終了を一次ログで見せる記事には差別化余地がある。
5. 既存Vitest記事のビジュアルリグレッションとは対象も実行環境も異なり、リポジトリ内の実質重複がない。
6. 「Jest/Vitestを置き換えられる」と一般化せず、小さなCLIで観測した範囲だけを結論にできるため、新人向けでも主張を安全に限定できる。

### 記事で答える問い

- `Deno.test.each`は各ケースを独立したテスト名として表示し、個別filterできるか。
- `t.assertSnapshot()`は初回生成、意図しない差分、更新、古いentry整理をどのように表示するか。
- 失敗後のretryで通ったテストは、公式説明どおり`flaky`として残るか。
- coverage thresholdを下回ると非ゼロ終了し、どの指標が不足したか分かるか。
- `--related`と`--shard`は、小規模な複数モジュール・複数テストファイルでも対象集合を説明可能な形で絞れるか。

## 半日検証アウトライン

### 0. 範囲と環境を固定する（20分）

- 隔離したrun directoryだけを使い、リポジトリ既存ファイルを変更しない。
- 実際に導入できたDeno 2.9.xのexact version、OS、CPU architectureを記録する。
- 追加テストフレームワーク、Webサーバー、ブラウザ、外部APIは使わない。
- Git stateを変更しない。`--changed`はGit履歴に依存するため必須範囲から外し、影響テスト選択は明示ファイルを渡せる`--related`で確認する。

### 1. 決定的な文字列整形CLIを作る（40分）

- `src/normalize.ts`: 空白正規化と大小文字変換。
- `src/render.ts`: `{title, tags}`を固定順のテキストへ変換。
- `main.ts`: 固定JSON入力を受けて出力する薄いCLI。
- 日時、乱数、ネットワーク、locale依存を入れず、同じ入力から同じ出力になるようにする。

### 2. 通常assertと`Deno.test.each`を確認する（40分）

- 正常入力、空文字、日本語、余分な空白の4ケースをtable testにする。
- 実行結果で各ケースが独立したテストとして表示されることを記録する。
- `--filter`で1ケースだけ選べるかを確認する。
- 1ケースだけ期待値を壊し、diffとexit codeを記録してから元へ戻す。

### 3. snapshotの生成・失敗・更新を確認する（50分）

- 複数フィールドを持つ整形結果を`t.assertSnapshot()`で固定する。
- `deno test --update-snapshots`で初回生成し、`.snap`の保存場所と内容を記録する。
- rendererの出力を意図的に変え、通常実行がdiff付きで失敗することを確認する。
- 更新後に再度通ることを確認し、snapshot更新は正しさの証明ではなく人間のdiff確認が必要だと明記する（[Deno snapshot docs](https://docs.deno.com/runtime/test/snapshots/)、2026-07-11閲覧）。

### 4. retry/repeatsの表示を確認する（35分）

- テストプロセス内の明示counterで「初回だけ失敗、2回目は成功」する専用テストを作る。
- retryなしでは失敗、`retry: 1`では最終成功だが`flaky`表示が残るか記録する。
- 決定的に成功するテストへ`--repeats=3`を指定し、実行回数とsummaryを記録する。
- retryを本物の不安定テストの隠蔽策として推奨しない。

### 5. coverage thresholdを品質ゲートとして確認する（45分）

- 意図的に未実行branchを1つ残し、coverageを採取する。
- 高すぎるthresholdで非ゼロ終了することと不足metricを記録する。
- 現実的なthresholdへ下げて成功させるのではなく、追加テストでbranchを通し、同じthresholdを満たす修正前後を比較する。
- line、branch、functionのうち実際に安定して取得できた指標だけを記事に使う。

### 6. `--related`とshardを追加確認する（40分・時間がある場合）

- `normalize.ts`と`render.ts`に対応する別々のテストファイルを用意する。
- `deno test --related=src/normalize.ts`で対象が絞られるかを、実行されたtest名で確認する。
- `--shard=1/2`と`--shard=2/2`の対象集合を記録し、和集合が全テストファイルと一致するか確認する。
- 小規模すぎて分割価値が見えない場合は、機能紹介に留めず「この規模では利点を測れない」と記録する。

### 7. 証拠を整理する（30分）

- 全コマンド、stdout/stderr、exit code、exact versionをexecution logへ残す。
- 成功だけでなく、snapshot差分、table test失敗、flaky表示、coverage不足を残す。
- 公式の説明と観測結果を分け、未検証機能へ結論を広げない。

## 完了条件

- Deno 2.9.xのexact versionが記録されている。
- 通常assertと`Deno.test.each`の成功・意図的失敗が各1件以上ある。
- snapshotの生成、差分失敗、更新後成功が記録されている。
- retryで通過したケースのsummaryが記録され、`flaky`表示の有無を断定できる。
- coverage thresholdの失敗と、テスト追加後の成功が同じthresholdで記録されている。
- すべての必須検証がローカル・無料・認証なしで完了している。
- 記事では実測した小規模CLIの範囲だけを主張し、他テストランナーの全面代替とは書かない。

## リスクと停止条件

| リスク | 対応 | 停止条件 |
|---|---|---|
| Deno 2.9.xを公式配布元から導入できない | 公式install docsとrelease artifactの範囲だけで再試行し、非公式バイナリは使わない。 | 公式手段でversion確認まで到達できなければ実践を中止する。 |
| snapshotに時刻・乱数・環境依存値が混ざる | 入力を固定し、必要なら公式docsのcustom serializer方針に従って正規化する。 | 正規化しても同一入力のsnapshotが連続2回一致しなければsnapshot評価を中止する。 |
| retry用テストが実行順や内部実装に依存する | counterを専用ファイルに隔離し、並列テストと共有しない。 | retryなし/ありの差を2回続けて再現できなければ`flaky`挙動を記事の結論にしない。 |
| coverage出力が環境差で変わる | exact versionを固定し、絶対値より同一環境の修正前後とexit codeを比較する。 | 同一コード・同一コマンドでmetricが不安定ならcoverage率の数値比較を取りやめる。 |
| `--related`が小規模構成で全テストを選ぶ | 依存方向が明確な2モジュールへ分け、実行test名を証拠にする。 | 期待どおり絞れなければ観測事実だけを残し、効率化を主張しない。 |
| 機能数が多く半日を超える | snapshot、table test、retry、coverageを必須、`--related`とshardを任意にする。 | 必須4項目のいずれかが未完のまま6時間を超えたら追加検証を打ち切る。 |
| 秘密情報・環境情報の混入 | 固定ダミーデータだけを使い、環境変数一覧やホームパスをログへ出さない。 | credential、token、個人情報を要求する手順に遭遇したら即時停止する。 |

## 結論

選定テーマは、**Deno 2.9の組み込みテストランナーでsnapshot・table test・retry・coverage gateを小さなCLIへ導入する検証**とする。直近リリースとして適時性があり、ローカル・無料・GUIなしで成功と失敗の両方を短時間に記録できる。既存記事との主題重複もなく、公式機能一覧の要約ではなく「どの出力が出て、どこで注意が必要だったか」を一次ログで示せる点が、初心者向けZenn記事として最も強い。
