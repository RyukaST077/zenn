## 上位候補の詳細

### 1. Node 26.5 の `--experimental-import-text` で `.txt` をimportして詰まった記録

#### 概要
Node.js v26.5.0（2026-07-08リリース）で追加された text imports を検証する。`import msg from './message.txt' with { type: 'text' }` のように import attributes を付けて `.txt` を文字列として読み込む機能で、`--experimental-import-text` フラグが必須、`with { type:'text' }` も必須、公開されるのは default export のみ、という制約を実際に踏みながら確認する。

#### なぜ今よさそうか
- 調査のわずか5日前（2026-07-08）にStableのCurrent版へ入ったばかりの機能で、国内の検証記事がまだほとんど無い。
- Bunは以前から無フラグ対応、Denoは `--unstable-raw-imports`。「Nodeは慎重にフラグ付き」という各ランタイムの温度差も比較の切り口になる。
- import attributes（`with { type }`）はES2025でJSONが安定化しており、textはその延長。仕様の流れを新人が理解する良い入口。

#### 新人が検証しやすい理由
必要なのは Node 26.5 のインストールと数行のファイルだけ。ネットワーク・課金・アカウント一切不要。フラグや属性を外すと明確にエラーになるので「詰まりどころ」を意図的に再現でき、検証ログ記事に向く。

#### 想定読者
ESM/importは書けるが import attributes を意識したことがない新人〜実務2年目。

#### タイトル案
1. `node app.mjs` で `.txt` を import してみたら、まず属性で怒られた話（Node 26.5 text imports）
2. 新人がNode 26.5の text imports を触って、フラグと `with { type:'text' }` で詰まった記録
3. JSON importは知ってた。text importは何が違うのか小さく試してみた

#### 検証でやること
- `--experimental-import-text` の有無で挙動がどう変わるか
- `with { type:'text' }` を省いた/間違えたときのエラー全文
- `fs.readFileSync` との書き味・取得結果の差
- JSON import（`type:'json'`）との対比、default export のみである確認

#### 検証手順
1. `node -v` で26.5系であることを確認（無ければ入手）
2. `message.txt` と `main.mjs` を用意し、`import msg ... with { type:'text' }` を書く
3. フラグ無しで実行→エラー確認、フラグ付きで実行→成功確認
4. 属性を外す/`type:'json'`に変える等で失敗パターンを収集
5. 同じ内容を `fs.readFileSync` でも読み、コード量・型・実行結果を比較

#### 記事構成案
1. はじめに / 2. なぜtext importsを試すのか / 3. 事前に調べたこと（import attributesとは） / 4. 環境構築（Node 26.5） / 5. 実際に試したこと / 6. 詰まった点（フラグ・属性エラー） / 7. 触ってみて分かったこと / 8. fs/JSON importと比べて感じたこと / 9. どんな人に向いていそうか / 10. まとめ

#### 差別化ポイント
既存の国内記事はJSON importの解説が中心。「Stable直後のtext importを実機で踏んで、失敗メッセージまで残す」検証ログはほぼ無い。

### 2. `crypto.randomUUIDv7()` を使い、v4と「ソートできるID」の違いを実測してみた

#### 概要
Node v26.1 で追加された `crypto.randomUUIDv7()` を試す。UUIDv7は先頭48bitがミリ秒Unixタイムスタンプで、生成順にソート可能。v4（完全ランダム）と並べてソート・タイムスタンプ抽出・DBのB-treeインデックス親和性という観点を比較する。

#### なぜ今よさそうか
- 標準ライブラリだけでUUIDv7が生成できるようになり、これまで `uuid`/`uuidv7` パッケージに頼っていた作業が不要に。
- PostgresやBunも同機能を持ち「ランタイム/DB横断でUUIDv7が当たり前になる」流れの中にある。
- 「なぜv7がDBで嬉しいのか」を手を動かして体感するのは新人に学びが大きい。

#### 新人が検証しやすい理由
標準 `node:crypto` のみで完結。生成→配列ソート→先頭bitからのタイムスタンプ復元まで数十行。無料・ローカル・数十分。

#### 想定読者
DB主キーやIDの設計を最近意識し始めた新人〜2年目。

#### タイトル案
1. Node標準の `crypto.randomUUIDv7()` を試して、v4との「並び順」の違いを見てみた
2. 新人が「時刻でソートできるUUID」を初めて生成してみた（UUIDv7 in Node 26）
3. `uuid`パッケージを入れずにUUIDv7を作れるようになったので比較してみた

#### 検証でやること
- v4/v7を大量生成して文字列ソートし、生成順との一致を比較
- v7先頭48bitからタイムスタンプを取り出し実時刻と突き合わせ
- `disableEntropyCache` オプションの有無で挙動確認
- 連続生成時の単調増加性の観察

#### 検証手順
1. Node 26.1+ を確認
2. `randomUUIDv7()` と `randomUUID()` を各1万件生成
3. 文字列ソート結果と生成順の相関を数値化
4. v7の先頭からミリ秒を復元して `Date` と比較
5. 結果を表・簡単なグラフ（テキスト）でまとめる

#### 記事構成案
テンプレート標準構成に準拠（はじめに→なぜ→事前調査→環境→試した→詰まった→分かった→比較→向き→まとめ）。

#### 差別化ポイント
「UUIDv7とは」の解説記事は多いが、Node標準実装を使い、v4と定量比較して“ソート可能性”を実データで見せる新人検証ログは少ない。

#### 参考リンク
- https://github.com/nodejs/node/commit/b267f6bca3
- https://nodejs.org/api/crypto.html
- https://www.infoq.com/news/2026/07/nodejs-26-temporal/

#### 注意点
UUIDv7はタイムスタンプを含むため、生成時刻が推測可能である点（プライバシー観点）に触れる。Node 26はCurrent（LTSは2026年10月〜）である旨を明記。

### 3. Bun 1.3で既定になった isolated install を、phantom dependency をわざと踏んで確かめた

#### 概要
Bun 1.3 では workspace利用時のデフォルトが isolated install になった。宣言していない依存（phantom dependency）へアクセスできなくなる挙動を、小さなmonorepo fixtureでわざと再現し、hoisted（`--linker=hoisted`）との違い、`bun why` での依存チェーン確認まで行う。

#### なぜ今よさそうか
- 「知らないうちに既定が変わって既存コードが壊れた」という報告が国内外で相次いでいる、生きたトラブル題材。
- pnpm由来の設計思想（phantom dep排除）を、実際に壊して理解できる。

#### 新人が検証しやすい理由
Bunをインストールし小さな2パッケージworkspaceを作るだけ。全てローカル・無料。壊れる/直るが明確に観察できる。

#### 想定読者
monorepo・workspaceに触れ始めた新人〜2年目。npm/pnpmの node_modules 構造に曖昧さを感じている人。

#### タイトル案
1. Bun 1.3でworkspaceの既定が変わってた—phantom dependencyをわざと踏んで確かめた
2. 新人が「宣言してない依存が使えない」を体験してisolated installを理解した話
3. `bun why` と isolated install で、node_modules の中身を初めて意識した

#### 検証でやること
- 宣言外importが isolated で失敗・hoisted で成功することの確認
- `node_modules/.bun/` + symlink 構造の観察
- `bun why <pkg>` の出力読み解き
- 既定変更で壊れやすいツール（例: Biome等）の挙動メモ

#### 検証手順
1. Bun 1.3系をインストール（`bun --version`）
2. workspace付き最小monorepoを作成
3. 宣言していないパッケージをimportして isolated で失敗を確認
4. `--linker=hoisted` / bunfig.toml で切り替え差分を観察
5. `bun why` で依存経路を確認しログ化

#### 記事構成案
テンプレート標準構成に準拠。

#### 差別化ポイント
国内記事は「デフォルトが変わった」報告や構造解説どまり。phantom depを意図的に踏んで失敗→原理理解まで通す検証ログは希少。
