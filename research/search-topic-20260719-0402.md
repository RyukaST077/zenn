### 1. Bun.Imageでsharpなしに画像リサイズ/変換して書き比べてみた

#### 概要
Bun 1.3.14 で入った組み込み画像処理API `Bun.Image` を、これまで定番だった `sharp` と同じ処理（読み込み→リサイズ→webp/avif変換→保存）で書き比べ、コード量・依存・処理時間を並べる検証ログ。

#### なぜ今よさそうか
- Bun.Image は 2026年5月の直近リリースで、公式が「native addon不要・依存ゼロで sharp 並み/以上」と主張しており話題性が高い。
- sharp は native ビルドで詰まりがちな定番のハマり所。そこを「pure Bunで消せるか」は読者の関心が強い。
- 過去記事は Bun.SQL（DB）で、画像処理は未検証。同じBunでも**別機能・別ユースケース**で重複しない。

#### 新人が検証しやすい理由
- `bun` を入れて画像1枚を用意すれば試せる。チェーンAPIが直感的（`.resize().webp().write()`）。
- 追加の native ビルドが不要なので環境構築でつまずきにくい。

#### 想定読者
Node/Bunで画像リサイズをsharpでやってきた人、sharpのビルドで困った経験のある新人。

#### タイトル案
1. Bun.Imageを初めて触って、sharpと同じ画像処理を書き比べてみた
2. sharpしか知らない新人がBun.Imageで画像リサイズを試してみた
3. Bun 1.3.14のBun.Imageで、依存ゼロの画像変換がどこまでできるか試した

#### 検証でやること
- 同一画像に対する resize→webp/avif 変換を Bun.Image と sharp の両方で実装
- コード量・必要な依存（node_modules差）・出力ファイルサイズ・処理時間を比較
- `metadata()` での寸法取得、fitモード（inside/fill）やquality指定の挙動確認

#### 検証手順
1. `bun init` で最小プロジェクトを作り、サンプル画像を配置
2. Bun.Image でリサイズ＋webp/avif 変換して保存（`.resize().webp().write()`）
3. 同じ処理を `sharp` で実装し、`bun add sharp` の導入コスト（ビルド有無）を記録
4. 出力サイズ・処理時間（`performance.now()` を複数回）・コード行数を表にする
5. HEIC/AVIF などOS依存フォーマットや大きい画像でのエラー/差異を記録

#### 記事構成案
1. はじめに
2. なぜBun.Imageを試すのか（sharpのビルド問題）
3. 事前に調べたこと（対応フォーマット・依存ゼロの理由）
4. 環境構築
5. 実際に試したこと（Bun.Image / sharp 両実装）
6. 詰まった点（フォーマット非対応・OS差・大画像）
7. 触ってみて分かったこと（速度・サイズ・コード量）
8. sharpと比べて感じたこと
9. どんな人に向いていそうか
10. まとめ

#### 差別化ポイント
「Bun＝速い」ではなく、**sharpのnativeビルド地獄を消せるか**という新人の実利点に絞る。処理時間だけでなく依存・コード量・つまずきを並べる。

#### 参考リンク
- Bun.Image 公式ドキュメント: https://bun.com/docs/runtime/image
- Bun v1.3.14 リリースブログ: https://bun.com/blog/bun-v1.3.14

#### 注意点
公式の「70x」等は metadata 読み取り等の限定条件。記事では**自分の実測値**を主にし、公式値は引用として区別する。OS依存フォーマット（HEIC/AVIF）は環境で結果が変わる点を明記。

### 2. TS7のtsgoとtscで同じリポジトリの型チェック時間を測ってみた

#### 概要
2026年7月にGAした TypeScript 7（Go製ネイティブコンパイラ）を、`npx tsgo --noEmit` と従来 `tsc --noEmit` で同一リポジトリに走らせ、型チェック時間と診断（エラー）の一致を実測する。

#### なぜ今よさそうか
- TS7がGA直後で「10x速い」が話題。だが**自分のコードで測る**記事は価値が高い。
- 過去記事の「tsc bin衝突ログ」は導入時の衝突が主題で、**速度・診断一致は未検証**＝切り口が違う。

#### 新人が検証しやすい理由
- `npm i -D @typescript/native-preview` で `tsgo` を併走できる。既存の `tsc` を壊さず比較可能。
- 小〜中規模のサンプル/OSSを clone して測るだけ。

#### 想定読者
CIの型チェックが遅くて悩む人、TS7を試したいが安全に比較したい新人。

#### タイトル案
1. TypeScript 7のtsgoが本当に速いのか、手元のコードで測ってみた
2. tscしか使ったことない新人がtsgoと型チェック時間を比べてみた
3. TS7 GA後、tsgoとtscを併走させて診断が一致するか確かめた

#### 検証でやること
- 同一リポジトリで `tsc --noEmit` と `tsgo --noEmit` の所要時間を複数回計測
- 出力される診断（エラー件数・内容）が一致するか差分チェック
- 規模の違う複数プロジェクト（小/中）で傾向を見る

#### 検証手順
1. サンプル/OSSを clone し、`tsc --noEmit` の時間とエラーを記録
2. `npm i -D @typescript/native-preview` → `npx tsgo --noEmit` を同条件で計測
3. 各3回ずつ測り中央値を取る／診断差分を `diff` で確認
4. 型が複雑な箇所で挙動差が出ないか確認
5. 「速いが安定API(7.1)前、eslint連携は未対応」等の制約を整理

#### 記事構成案
1. はじめに 2. なぜ試すのか 3. 事前に調べたこと（tsgoの位置づけ） 4. 環境構築 5. 実際に試したこと 6. 詰まった点 7. 分かったこと（速度/診断一致） 8. tscと比べて 9. 向いている人 10. まとめ

#### 差別化ポイント

### 3. cross-document View TransitionsをPlaywrightで確認してみた

#### 概要
`@view-transition { navigation: auto; }` だけで、MPA（複数HTML）間の遷移をCSSのみでアニメーションさせる cross-document View Transitions を、2枚の静的HTMLで作り、Playwright（Chromium）で遷移をスクリーンショット確認する。

#### なぜ今よさそうか
- Chrome/Safari 18.2+ で実用段階、View Transitions自体は2025-10にBaseline入り。SPA前提だった機能がMPAで使える点が新しい。
- 過去のCSS記事（Anchor Positioning成功／Gap Decorations・focusgroup gate）とは**別機能**で、成功寄りの検証にできる。

#### 新人が検証しやすい理由
- ビルド不要の静的HTML2枚＋CSS数行。`python -m http.server` 等で配信しPlaywrightで開く。

#### 想定読者
JSフレームワーク無しでページ遷移アニメを入れたいフロント新人。

#### タイトル案
1. JSなしでページ遷移アニメ、cross-document View Transitionsを試してみた
2. 静的HTML2枚で@view-transitionを動かしてPlaywrightで確認した
3. MPAでもView Transitionsが効くのか新人が試してみた

#### 検証でやること
- 同一オリジンの2ページに `@view-transition{navigation:auto}` を付与
- `view-transition-name` で要素を対応付けて遷移をアニメ化
- Playwrightで遷移前後をスクショ、効かない条件（別オリジン/戻る等）を確認

#### 検証手順
1. `pageA.html` / `pageB.html` と共通CSSを用意
2. ローカルHTTPサーバで配信（file://では動かない点に注意）
3. Playwright(Chromium)でリンク遷移させ、途中/前後をキャプチャ
4. `view-transition-name` の有無で見え方の差を比較
5. 4秒タイムアウトやFirefox未対応などの制約を記録

#### 記事構成案
1. はじめに 2. なぜ試すのか 3. 事前に調べたこと 4. 環境構築 5. 実際に試したこと 6. 詰まった点（file:///オリジン/戻る） 7. 分かったこと 8. SPA版と比べて 9. 向いている人 10. まとめ

### 4. Node 26のPermission ModelでSQLite拡張ロードが止まる挙動を確かめた

#### 概要
Node 26 の Permission Model（`--permission`）を有効にした状態で `node:sqlite` を使い、「fs制限があってもsqlite経由でファイルに触れてしまう」点や「実行時ロード拡張(loadable extension)がロードできない」挙動を、最小コードで再現・記録する。

#### なぜ今よさそうか
- 26.5 で SQLite（3.53.3）更新と権限まわりの挙動が明文化。セキュリティ境界の具体例は需要が高い。
- 過去の `npm allowScripts`（security）とはツールも境界も違い、gate系の得意フォーマットで書ける。

#### 新人が検証しやすい理由
- 外部依存なし。標準の `node:sqlite` と `--permission` フラグの有無で挙動差を出すだけ。

#### 想定読者
Node標準SQLiteとPermission Modelを使い始める新人、セキュリティ境界を知りたい人。

#### タイトル案
1. Node 26のPermission Modelで、SQLite経由のファイルアクセスを確かめた
2. `--permission`下でSQLite拡張ロードが止まる挙動を新人が試した
3. Node標準SQLite×権限モデルの境界を最小コードで確認した

#### 検証でやること
- `--permission --allow-fs-read=...` 有無での `node:sqlite` の読み書き挙動比較
- loadable extension のロードが拒否されることの再現とエラー確認
- 「fs制限=完全な隔離ではない」ドキュメント記載の裏取り

#### 検証手順
1. 最小DBを作る `node:sqlite` スクリプトを用意
2. `--permission` なし→動作確認、ありで挙動変化を記録
3. 拡張ロードAPIを呼びエラーメッセージを記録
4. `--allow-fs-*` の粒度を変えて挙動差を表にする
5. 公式ドキュメントの注意書きと実挙動を突き合わせる

#### 記事構成案
1. はじめに 2. なぜ試すのか 3. 事前に調べたこと 4. 環境構築 5. 実際に試したこと 6. 詰まった点 7. 分かったこと（境界の実際） 8. 期待と比べて 9. 向いている人 10. まとめ

#### 差別化ポイント

## 最初に試すべき1本

### 推奨テーマ
**Bun.Imageでsharpなしに画像リサイズ/変換して書き比べてみた**

### 理由
- **トレンド性**: Bun 1.3.14 の直近目玉機能で、「依存ゼロで sharp 代替」は複数の海外メディア・公式が推している。
- **新人でも検証できる**: `bun` と画像1枚だけ。チェーンAPIが直感的で、環境構築で詰まりにくい。
- **Zenn読者に刺さる**: sharp の native ビルド地獄は日本語圏でも定番のハマり所。そこを消せるかは実利的で共感を得やすい。
- **半日〜1日で検証可**: 実装＋計測＋比較表まで1〜2時間の最小検証が可能。
- **過去記事と重複しない**: Bun記事は Bun.SQL のみで、画像処理は未検証。同じBunでも別機能・別ユースケースで差分が明確。

### 完成イメージ
Bun.Image と sharp の**同一処理コードを並べ、依存差・出力サイズ・処理時間・つまずきを表にした「書き比べ検証ログ」**。過去の `bun-sql-sqlite-crud-try` / `node26-map-getorinsert-...`（書き比べ系）と同じ読み味で、新機能を1本に収める。

### 最小検証プラン
1. `bun init` で最小プロジェクト＋サンプル画像を用意
2. Bun.Image で resize→webp/avif 変換して保存
3. 同処理を `sharp` で実装（`bun add sharp` の導入コストも記録）
4. 出力サイズ・処理時間（複数回）・コード行数・依存差を比較表に
5. 非対応フォーマット/大画像で出るエラー・差異を「詰まった点」として記録

### 想定タイトル
「Bun.Imageを初めて触って、sharpと同じ画像処理を書き比べてみた」

### 書く前に用意するもの
- Bun 最新（1.3.14 以降）
- 比較用のサンプル画像（JPEG/PNG、可能なら大きめ1枚）
- 計測用の最小スクリプトと結果記録用メモ（実行ログ・スクショ）

### 記事化するときの注意
- 専門家として断定しすぎない
- 「新人が試した範囲」を明示する（対応フォーマット・OS依存を限定と書く）
- うまくいった点だけでなく詰まった点（非対応/大画像/OS差）を書く
- 公式情報（bun.com/docs/runtime/image）へのリンクを入れる
- 実行ログ・スクリーンショット・両実装コードを残す
- 公式の「70x」等は自分の実測と区別して引用する
