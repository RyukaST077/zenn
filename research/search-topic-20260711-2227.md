## 除外候補と理由

- Deno Desktop: 目玉機能だが、OS固有WebView、installer、アイコンなどGUI・OS差が結果を左右しやすく、半日の再現可能なCLI記事という既定条件から外す。実験機能であることは[Deno 2.9公式](https://deno.com/blog/v2.9)（閲覧: 2026-07-11）にも示される。
- Deno 2.9のsnapshot / parameterized test / retry / coverage: `research/search-topic-20260711-0148.md`で選定済みのため除外する。
- TypeScript 7、Node VFS、テスト順ランダム化、Vite 8、Vitest 4、Biome、oxlint、Playwright passkeys、CSS Anchor Positioning、CSS Gap Decorations: 既存記事と実質重複するため除外する。
- Baseline Alerts: 通知設定にGitHub sign-inが必要とされるため、認証不要という条件から外す（[web.dev](https://web.dev/blog/baseline-alerts)、公開: 2026-06-25、閲覧: 2026-07-11）。
- Deno Deploy、KV Backup、クラウドbenchmarks: signup、外部service、credentialまたは課金条件が入り得るため除外する。
- Fresh View Transitions: 有望だがブラウザ表示の証拠が中心になり、過去のCSS・Vitestブラウザ記事より差別化が弱い。公式は未対応ブラウザではanimationなしで動くと説明している（[Fresh公式](https://usefresh.dev/docs/advanced/view-transitions)、閲覧: 2026-07-11）。

小さな3-package monorepoで明示dependencyとphantom dependencyを作り、通常の`node_modules`探索とpackage mapの挙動を比較する。直近性と差別化は高いが、Node.js 26.4以上と実験flagが必要である（[Node.js公式](https://nodejs.org/api/packages.html#package-maps)、閲覧: 2026-07-11）。

## 選定テーマ

**Deno 2.9の`deno install`で既存Nodeプロジェクトを移行し、npm / pnpmのlockfileから解決済みversion・integrity・workspace構成がどこまで引き継がれるか確認する。**

想定タイトル:

> npm/pnpmのlockfileをDeno 2.9へ持っていったら、本当に同じ依存関係になったか確かめた

選定理由:

- Deno 2.9は2026-06-25公開で、調査時点の適時性が高い（[公式発表](https://deno.com/blog/v2.9)、閲覧: 2026-07-11）。
- 公式が4種類のlockfileからresolved versionとintegrityをseedすると具体的に主張しており、ファイル差分とCLI出力で成否を判定できる（同上）。
- 国内では機能紹介が既に出ている一方、固定version、workspace、frozen install、lifecycle scriptまで一続きにした実測記事は今回確認した国内ソースにはなく、差別化できる（[JSer.info](https://jser.info/2026/07/05/deno-2.9-vite-8.1-es2026/)、[ラーゲイト](https://www.ragate.co.jp/media/developer_blog/34lmbdb-xu)、いずれも閲覧: 2026-07-11）。
- Node.js、Deno 2.9、npm / pnpmの公開packageだけで実行でき、認証、クラウド、GUI、課金を必要としない。
- Denoのテストランナーを扱った過去調査とは主題が異なり、既存記事にもpackage manager migrationはない。

## 半日検証アウトライン

所要目安は4〜6時間。repository外の隔離run directoryを使い、exact version、全command、stdout / stderr、exit code、生成file一覧を記録する。

### 0. 環境ゲート（20分）

- OS、architecture、`node --version`、`npm --version`、`pnpm --version`、`deno --version`を記録する。
- Deno 2.9.xであることを必須にし、異なるmajor/minorなら停止する。
- registry、private URL、token、環境変数全量は記録しない。

### 1. 同一fixtureを作る（45分）

- `package.json`にpure JavaScriptのruntime dependency 2個、dev dependency 1個、`test` scriptを置く。
- direct dependency、transitive dependency、optional dependencyがtreeに現れる小構成を選ぶ。native addonとinstall scriptは、この段階では避ける。
- npm版とpnpm版を別directoryに複製し、sourceと`package.json`のhashが同じことを記録する。

### 2. 元package managerのbaselineを固定する（45分）

- npm版で`npm ci`、pnpm版で`pnpm install --frozen-lockfile`を実行する。
- direct / transitive dependencyのversion、lockfile上のintegrity、test結果を抽出する。
- `node_modules`を削除してもlockfileとsourceが残る状態を移行開始点にする。

### 3. Deno 2.9へ初回移行する（60分）

- `deno.lock`がない状態で各fixtureへ`deno install`を実行し、seed message、exit code、生成fileを記録する。
- 元lockfileのresolved version / integrityと`deno.lock`を機械的に比較する。format差を同一視せず、packageごとの対応表を作る。
- `deno task test`を実行し、元package manager時と同じtestが通るか確認する。

### 4. frozen installと再現性を確認する（40分）

- 生成済み`deno.lock`で`deno install --frozen`または`deno ci`を実行する。公式のstrict install挙動は[Deno dependency docs](https://docs.deno.com/runtime/packages/)（閲覧: 2026-07-11）を根拠にする。
- `package.json`のversion rangeを意図的に変更し、frozen状態で非ゼロ終了するかを記録する。
- 変更を戻し、再度installとtestが成功することを確認する。

### 5. pnpm workspace境界を確認する（60分）

- 2 packageだけのworkspaceと`pnpm-workspace.yaml`を用意し、workspace package間依存を1本作る。
- Denoが設定移行を提案・実行した場合、変更された`package.json` / `deno.json`をdiffで記録し、再実行後にtaskが通るか確認する。公式仕様は[Deno 2.9発表](https://deno.com/blog/v2.9)（閲覧: 2026-07-11）に限定する。
- `catalog`は基本catalogを1つだけ試し、`catalogs`追加検証は時間超過時に切る。

### 6. lifecycle scriptの安全境界（35分）

- 外部通信やnative buildをしない自作local packageに、marker fileを作る無害な`postinstall`を置く。
- Denoの通常installで自動実行されないことを確認する。明示承認を試す場合もrun directory内のmarkerだけを対象にする。
- Deno公式がlifecycle scriptをopt-inとする説明と、実測結果を分けて記載する（[公式docs](https://docs.deno.com/runtime/packages/)、閲覧: 2026-07-11）。

### 7. 証拠整理（40分）

- npm / pnpm baselineとDeno移行後のversion、integrity、test、workspace、script挙動を表にする。
- 「完全互換」「そのまま移行できる」と一般化せず、今回のexact versionとfixtureで一致した範囲を結論にする。
- 元lockfile、生成`deno.lock`、設定diff、command出力、exit codeをexecution logへ時系列で残す。

## 完了条件

- npmとpnpmの2 fixtureでbaselineの依存version、integrity、test結果を記録できた。
- Deno 2.9初回installのseed成否と、生成された`deno.lock`を記録できた。
- direct / transitive dependencyのversionが元lockfileと一致したかをpackage単位で判定できた。
- 移行後に同じtestを`deno task`で実行し、結果を記録できた。
- frozen installの成功と、意図的なmanifest不整合時の失敗を確認できた。
- pnpm workspaceまたはlifecycle script境界の少なくとも一方を、公式仕様と実行ログの両方で確認できた。
- すべてローカル・無料・認証不要で完結した。

## リスクと停止条件

- **Deno 2.9を用意できない:** 対象機能を検証できないため停止し、versionだけを失敗証拠として残す。別versionの挙動から成功を推測しない。
- **npm registryへ接続できない:** 隔離cacheで1回だけ再試行する。取得不能なら停止し、既存global cacheだけの結果を新規install成功として扱わない。
- **lockfileのseed messageが出ない、または再解決が疑われる:** exact versionとintegrityを比較し、不一致なら移行成功と書かない。fixtureをdirect dependency 1個に一度だけ単純化し、それでも原因不明なら停止する。
- **npm / pnpm baselineが一致しない:** package managerごとの正常な解決差であり得るため、相互に同一であることは要求しない。それぞれの元lockfileとDeno移行後だけを比較する。
- **pnpm workspace自動移行が設定を予期せず破壊する:** run directoryのcopyだけでdiffを取り、追加修正を一度に限定する。repositoryや実プロジェクトへ適用しない。
- **install scriptがrun directory外へ書こうとする、または外部通信を要求する:** 即時停止する。marker fixture以外のscript承認は行わない。
- **integrity表現を対応付けられない:** hash algorithmと値を生データのまま提示し、「保持された」と断定しない。
- **時間超過:** 6時間を超える場合、追加のYarn / Bun fixtureと複数catalogを打ち切る。npm、pnpm、frozen install、最低1つの境界検証を優先する。
- **秘密情報混入:** token、cookie、private registry、private hostname、user名を含む絶対path、環境変数全量をartifactへ保存しない。混入時は保存前に伏せる。

## 結論

最初に試すテーマは、**Deno 2.9の`deno install`でnpm / pnpmのlockfileを`deno.lock`へseedし、解決済みversion・integrity・test・workspace・frozen installの境界を確認する検証**とする。2026-06-25公開の直近機能で、国内外にリリース関心がありながら、既存記事にはない依存グラフ移行の実測へ差別化できる。認証やGUIを使わず、成功だけでなく不一致・strict install失敗・script非実行もCLI証拠として残せるため、新人向けの半日ハンズオンに適している。
