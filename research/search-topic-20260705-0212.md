## 上位候補の詳細

### 1. `node app.ts`が普通に動く時代を新人が試す（合計28点 / 最優先）

#### 概要
Node 24 LTS で type stripping が既定になり、`tsx` や `ts-node` を入れなくても `node app.ts` が動く。
新人が「本当にビルドなしで動くのか」「どこまで動いてどこで壊れるのか」を実測する検証ログ。

#### なぜ今よいか
Node.js 公式が「greenfieldの新標準」と位置づけ、22.18+で既定化。移行の意思決定が各現場で発生中。
一次情報（Node.js公式docs / Learn）が明快で、新人でも一次ソースに当たれる。

#### 新人が検証しやすい理由
必要なのは Node 24 と `.ts` ファイルだけ。無料・認証不要・数分でセットアップ完了。

#### 想定読者
tsx/ts-nodeを使ってきた新人〜2年目。「ビルドステップって結局要るの？」に答えたい層。

#### タイトル案
- 「`node app.ts` が普通に動く時代になったので、tsx を卒業できるか新人が試した」
- 「Node 24 のネイティブTypeScript、どこまで動いてどこで詰まる？を全部試した」

#### 検証でやること
- Node 24 で `node app.ts` を実行（フラグなし）
- `enum` / `namespace` / パラメータプロパティ / import aliases が **動かない** ことを実際に踏む
- 型エラーは検出されないことを確認 → `tsc --noEmit` を別途回す運用を検証
- `tsx` 実行との起動速度・挙動差を比較
- `--experimental-strip-types` 系フラグの現状を確認

#### 記事構成案
1. はじめに（tsx/ts-nodeを入れてた前提）
2. Node 24 で `node app.ts` を動かす（最小例）
3. 詰まった点集（enum/namespace/型エラー非検出）とエラー全文
4. tsx との比較（起動速度・カバー範囲）
5. 結局どう運用する？（実行=node / 型チェック=tsc --noEmit）
6. まとめ：新人が今から始めるなら

#### 差別化ポイント
「概念解説」ではなく **壊れるケースをエラー全文つきで踏む** 検証ログ。新人が最初に踏む地雷を先回り。

#### 参考リンク
- Node.js 公式 `Modules: TypeScript`（typescript.html）
- Node.js Learn「Running TypeScript Natively」

#### 注意点
Nodeのバージョンで挙動が変わる。記事に検証したNodeバージョンを必ず明記する。

### 2. React Compiler v1.0でuseMemo/useCallbackを全部消してみた（合計27点）

#### 概要
Vite+ReactにReact Compiler v1.0を入れ、codemodで useMemo/useCallback/React.memo を除去。
React Profilerで再レンダリング回数を before/after 実測し、「消して大丈夫だったもの・壊れたもの」を記録する。

#### なぜ今よいか
v1.0が安定し、95%のケースで手動メモ化が不要に。Metaで初期表示12%改善の実績。無料・アカウント不要。

#### 新人が検証しやすい理由
`npm create vite` ベースにBabelプラグイン追加のみ。Profiler/ESLintルールで結果が可視化される。

#### 想定読者
useMemo/useCallbackを「おまじない」で書いてきた新人。

#### タイトル案
- 「React Compiler v1.0を入れてuseMemo/useCallbackを全部消してみた（Profilerで実測）」
- 「『もうuseMemoいらない』は本当か、新人が再レンダリング数を測って確かめた」

#### 検証でやること
- Vite+ReactにReact Compiler導入＋ESLint（react-hooks）有効化
- 小さなサンプル（リスト＋フィルタ）でProfilerの再レンダリング数を before/after 比較
- codemodでメモ化フックを除去
- LogRocket記事の「壊れた例」を再現（React Hook Formの`watch()`、Chart.js的なコールバックのstale参照、`"use no memo"`ディレクティブ回避）
- Playwrightでクリック挙動のスクショ取得

#### 記事構成案
1. はじめに（useMemoを何となく書いてた話）
2. 導入手順（Vite/Babel/ESLint）
3. Profilerで再レンダリングを測る（before/after）
4. 消してみた → 壊れた箇所と回避（`"use no memo"`）
5. Memoバッジ=最適化成功ではない、という落とし穴
6. まとめ：新人が今すぐ消していいか

#### 差別化ポイント
「解説記事」でなく **実測値＋壊れた再現手順**。Profilerのスクショと数値を出す。

#### 参考リンク
- react.dev「React Compiler v1.0」
- LogRocket「React Compiler memoization: what actually broke」

#### 注意点
Compilerの効果は元コードのメモ化状況に依存。恣意的に効果を盛らず、素直なサンプルで測る。
