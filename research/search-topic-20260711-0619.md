- **Deno 2.9 `deno desktop`**: 公式発表ではネイティブデスクトップアプリを対象とするが、GUI 動作確認が主になり、利用可能なヘッドレス環境だけで完結する保証がないため除外（[Deno 2.9](https://deno.com/blog/v2.9)、2026-06-25、アクセス日 2026-07-11）。
- **WebMCP を本番相当で検証**: Chrome 149 の Origin Trial という段階で、ブラウザ側エージェントや試験設定への依存が大きいため、半日での安定再現を優先して除外（[Chrome at I/O 2026](https://developer.chrome.com/blog/chrome-at-io26)、2026-05-19、アクセス日 2026-07-11）。
- **HTML-in-Canvas**: Origin Trial であり、安定版の標準機能をローカル検証する今回の条件に合わない（[Chrome at I/O 2026](https://developer.chrome.com/blog/chrome-at-io26)、2026-05-19、アクセス日 2026-07-11）。
- **GitHub SHA-1 brownout 対応**: 2026-07-14 の brownout は適時性が高いが、古い TLS/Git 環境を安全に再現するにはコンテナや暗号設定の作り込みが必要で新人向け半日検証として重い（[GitHub Changelog](https://github.blog/changelog/2026-04-20-sunsetting-sha-1-in-https-on-github/)、2026-04-20、アクセス日 2026-07-11）。

## 上位 3 候補

### 1. CSS Gap Decorations を従来手法と比較

- Chrome/Edge 149 で安定提供された直後で適時性が高い（[Chrome 公式](https://developer.chrome.com/blog/gap-decorations-stable)、2026-05-15、アクセス日 2026-07-11）。
- 静的 HTML/CSS と Playwright のヘッドレス Chromium だけで、スクリーンショット、computed style、要素座標を自動記録できる。
- 日本語解説は存在するため、構文紹介だけでなく「border/nth-child 回避策とのコード量・レスポンシブ時の挙動・座標不変・非対応時フォールバック」の観測ログで差別化する。

### 2. Deno 2.9 の既存 lockfile 移行

- 公式は npm/pnpm/yarn/Bun lockfile を `deno install` が直接読めると説明しており、Node プロジェクト移行の実用テーマになる（[Deno 2.9](https://deno.com/blog/v2.9)、2026-06-25、アクセス日 2026-07-11）。
- 小さな npm プロジェクトを複製し、npm/pnpm lockfile ごとの解決結果と生成差分を CLI で比較できる。
- ただし Deno の導入と registry 通信に依存し、CSS の単一 HTML より外部要因が多い。

### 3. CSS `@function` のローカル検証

- Chrome の 2026 UI 更新では `@function` を Chrome 139 以降の機能として紹介し、`if()` や style queries との組合せ例を示している（[Chrome Web UI 2026](https://developer.chrome.com/blog/new-in-web-ui-io26)、2026-07-01、アクセス日 2026-07-11）。
- CSS だけで入力・ローカル変数・戻り値を持つ関数を作り、同じ計算を custom property/Sass 風記述と比較できる。
- Chrome 偏重かつ実務の定番用途がまだ固まっていないため、Gap Decorations より有用性を低く評価した。

## 選定テーマ

**CSS Gap Decorations で Grid/Flex の区切り線を引き、border + `nth-child` 回避策と比較する**

想定タイトル:

> Grid/Flex の区切り線を `row-rule` で引いてみた — border 回避策はもう不要？

選定理由:

1. 2026-05 に Chrome/Edge 149 で利用可能になった直近機能である（[Chrome 公式](https://developer.chrome.com/blog/gap-decorations-stable)、2026-05-15、アクセス日 2026-07-11）。
2. 従来の border、背景、疑似要素という具体的な困りごとを、短い CSS で置換できるため新人にも価値が伝わりやすい。
3. 完全ローカル・無料・認証不要で、ヘッドレスブラウザによる再現とスクリーンショット取得が可能。
4. 既存記事の CSS Anchor Positioning と重複しない。
5. 現状は Firefox/Safari 未対応と記録されているため、成功例だけでなく progressive enhancement と `@supports` の必要性まで記事にできる（[MDN browser-compat-data](https://chromium.googlesource.com/external/github.com/mdn/browser-compat-data/+/1c8d7073c8173fdc9c716517b7a21dc459afe3f0/css/properties/row-rule.json)、アクセス日 2026-07-11）。

## 半日検証アウトライン

### 0. 事前ゲート（20分）

1. 隔離した run ディレクトリで Node、Playwright、同梱 Chromium のバージョンを記録する。
2. `CSS.supports("row-rule", "1px solid red")` と `CSS.supports("column-rule", "1px solid red")` を評価する。
3. どちらも `false` なら、Chrome/Chromium 149+ を安全な run-local cache へ導入できる場合のみ1回試す。導入できなければ停止し、未検証を成功扱いしない。

### 1. ベースライン（40分）

1. 3列から1列へ変化する responsive Grid と、wrap する Flex の静的 HTML を作る。
2. 従来版として item border + `nth-child` で区切り線を実装する。
3. 幅 960px / 480px のスクリーンショット、CSS 行数、不要な末尾線の有無を記録する。

### 2. Gap Decorations（70分）

1. 同じ markup に `row-rule`、`column-rule`、`rule` を適用する。
2. `repeat()` で線幅・色を交互にし、`rule-break: intersection`、`rule-inset`、`rule-visibility-items: between` を個別ケースで確認する。
3. 各ケースで screenshot と `getComputedStyle()` を保存する。
4. 装飾前後の container/item の `getBoundingClientRect()` を比較し、装飾がレイアウト寸法へ影響しないか実測する。仕様上の期待は [MDN CSS gaps](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Gaps)（アクセス日 2026-07-11）と照合する。

### 3. フォールバックとエンジン差（50分）

1. `@supports (row-rule: 1px solid)` 内に新実装、外側に border ベースのフォールバックを置く。
2. 利用可能なら Playwright Firefox/WebKit で同一ページを開き、未対応時にも内容・レイアウトが失われないことを確認する。
3. ブラウザバイナリが未導入なら追加ダウンロードを無制限に繰り返さず、Chromium の feature detection とフォールバック CSS の静的確認までを明記する。

### 4. 整理（60分）

1. 従来版と新実装の CSS 行数、selector 数、viewport 変更時の不具合、座標差、ブラウザ対応を表にする。
2. 「Chrome/Edge では便利」「クロスブラウザでは progressive enhancement が必要」を観測範囲に限定して結論化する。
3. コマンド、標準出力、スクリーンショット、環境バージョンを execution log に残す。

## リスクと停止条件

- **ブラウザ版不足**: `CSS.supports` が false で Chrome/Chromium 149+ を1回の安全な導入でも用意できなければ停止。フラグで無理に成功扱いしない。
- **仕様変動**: Working Draft であり、古い `*-outset` と現行 `*-inset` が混在し得る。実行ブラウザと W3C/Chrome 公式の現行構文が一致しない場合は、その差を失敗として記録して停止または範囲縮小する。
- **クロスブラウザ未対応**: Firefox/Safari の成功を完了条件にしない。未対応を既知の制約として記録し、`@supports` フォールバックが内容を壊さないことを完了条件にする。
- **画像だけの判定**: スクリーンショット目視のみで成功判定せず、`CSS.supports`、computed style、bounding rect を併用する。取得値が矛盾した場合は断定を止める。
- **ネットワーク障害**: 既存ブラウザで検証できず、依存・ブラウザ取得が失敗した場合は再試行を1回までとし、環境や共有 cache を破壊しない。
- **時間超過**: 2時間時点で最小 Grid の `row-rule` / `column-rule` が描画できなければ高度な `repeat()` や animation を切り、原因が確定しなければ検証自体を停止する。
- **秘密情報**: 環境ログは必要なバージョンだけを記録し、ユーザー名、ホームパス、cookie、token、環境変数一覧を成果物へ残さない。
