### 1. ローカルMCPサーバーをTypeScriptで初めて作り、MCP Inspectorで叩いてみた

#### 概要
`@modelcontextprotocol/sdk` を使い、stdioで動く小さなMCPサーバー（tool を1〜2個）をTypeScriptで作る。動作確認は `npx @modelcontextprotocol/inspector` のWeb UI（:6274）からtool callを直接叩く。SDK v2ベータでは stateless化・パッケージ分割・inputSchemaのStandard Schema対応（Zod v4/ArkType等）といった大きめの変更が入っており、その新しい書き味を触る。

#### なぜ今よさそうか
- MCPは開発ツール連携の事実上の標準になりつつあり、国内外で実践記事の需要が高い。
- **2026-07-28に仕様RCが予定**され、SDK v2ベータが出ている「まさに今」のタイミング。
- LLMの鍵が無くてもInspectorでtool呼び出しを検証でき、AIエージェント単独・完全ローカルで完結する。

#### 新人が検証しやすい理由
公式手順で「stdioサーバー＋toolを追加→Inspectorで叩く」が約30分の中核。JSON-RPCの往復をInspector上で目視でき、新人でも「作った→動いた」を実感しやすい。

#### 想定読者
MCPを名前だけ知っている新人〜2年目。「まず自分の手で最小のMCPサーバーを動かしたい」層。

#### タイトル案
1. MCPサーバーを初めてTypeScriptで作って、Inspectorで叩いてみた
2. 鍵なしで試すMCP入門 — 最小のstdioサーバーをローカルで動かす
3. MCP TS SDK v2ベータの新しい書き味（stateless / Standard Schema）を新人が触ってみた

#### 検証でやること
- SDK v2ベータで最小のstdioサーバーを作成（tool を1〜2個）
- MCP Inspectorからtool callを実行し、入出力を確認
- inputSchemaをStandard Schema（Zod/ArkType）で書いてバリデーションを確認

### 最小検証プラン
1. TS環境を作り、MCP TS SDK v2ベータを**バージョン固定**で導入
2. stdioで動く最小サーバーを実装し、副作用のない `add` tool を1つ定義
3. inputSchemaを Standard Schema（Zod v4 か ArkType）で書く
4. `npx @modelcontextprotocol/inspector` を起動し、UIからtool callを実行してスクショ
5. 不正入力でバリデーションエラーを確認し、v1系との書き味の差・詰まった点を記録

### 想定タイトル
「MCPサーバーを初めてTypeScriptで作って、Inspectorで叩いてみた（鍵なし・ローカル完結）」

### 書く前に用意するもの
- Node.js（LTS/Current）とTypeScript、`@modelcontextprotocol/sdk`（v2ベータ、版を固定）
- `@modelcontextprotocol/inspector`（`npx` 実行、常駐不要）
- スクリーンショット保存先（`images/<slug>/`）

### 記事化するときの注意
- 専門家として断定しすぎない
- 「新人が試した範囲」を明示する
- うまくいった点だけでなく詰まった点（ベータのバージョン差・stateless周り）を書く
- 公式情報へのリンクを入れる
- 実行ログ、スクリーンショット、コードを残す
- ベータ仕様は変わりうるため「検証時点の版」を必ず明記する
