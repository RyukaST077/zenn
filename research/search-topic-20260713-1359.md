## 上位3候補

### 1位: pnpm生成package mapsの`standard` / `loose`比較

同じworkspaceに「宣言済みdependency」「rootにhoistされて偶然読めるphantom dependency」「同一packageの異なるversion」を置く。通常Node、pnpmの`loose` map、`standard` mapを比較し、成功 / `MODULE_NOT_FOUND`、resolved path / version、生成map差分を記録する。Node公式の手書きJSON sampleを再演するだけでなく、package managerがdependency graphから生成するmapが実際に何を拒否するかへ踏み込める（[Node.js](https://nodejs.org/api/packages.html#package-maps)、[pnpm 11.8](https://github.com/pnpm/pnpm/releases/tag/v11.8.0)、閲覧: 2026-07-13）。

### 2位: `permission.drop()`の「dropしたのに読める」を境界で説明

configをopenする前後、directory grantとfile reference、wildcard grant、open済みfile descriptorを小さなfixtureで比較する。`drop()`はfuture checkだけに効くため、「既存handleは閉じる必要がある」を失敗例で伝えられる（[Node.js Permissions](https://nodejs.org/api/permissions.html)、閲覧: 2026-07-13）。実用性は高いが、日本語解説が既にexact-grantとopen resourceまで扱うため2位とした（[Webtech Watcher](https://note.com/webtech_watcher/n/nad3d7c4e2e4e)、閲覧: 2026-07-13）。

### 3位: ECMAScript 2026 support matrix

正式承認されたeditionを対象に、Node 22 / 24 / 26とlaunchできたbrowserだけで固定test suiteを実行し、syntax / API availability / edge caseを比較する（[Ecma International](https://ecma-international.org/news/ecma-international-approves-new-standards-14/)、閲覧: 2026-07-13）。時宜は最上位だが、複数featureの紹介に散りやすく、browser engine差を記事の結論へ一般化しない設計が必要になる。

## 選定テーマ

**Node.js 26.4+のexperimental package mapsを、pnpm 11.8+が生成する`standard` / `loose` mapで比較する。同じ小規模workspaceに宣言済みdependencyとphantom dependencyを作り、通常の`node_modules`探索では成功するimportが`standard` mapで拒否されるか、`loose`では互換挙動へ戻るかを、生成JSON・resolved version・終了codeで検証する。**

想定title:

> pnpmでは読めた依存がNode.js 26で消える？ package mapsのstandard / looseを試す

### 選定理由

1. Node.js 26.4.0は2026-06-24公開で、調査日の約3週間前にpackage mapsを追加した（[release](https://nodejs.org/en/blog/release/v26.4.0)、閲覧: 2026-07-13）。
2. Node公式はpackage mapsの用途としてmonorepo、phantom dependency拒否、filesystem非依存のstatic resolutionを明記しており、記事の問いと成功 / 失敗条件が仕様に直結する（[Packages docs](https://nodejs.org/api/packages.html#package-maps)、閲覧: 2026-07-13）。
3. pnpm 11.8とYarnがmap生成・flag注入・`standard` / `loose`を設定として持つため、手書きconfiguration紹介ではなくpackage managerを含む現実的な移行判断を扱える（[pnpm](https://github.com/pnpm/pnpm/releases/tag/v11.8.0)、[Yarn](https://yarnpkg.com/configuration/yarnrc/#nodeExperimentalPackageMap)、閲覧: 2026-07-13）。
4. Node.js、pnpm、local workspaceだけで完結し、account、token、cloud、GUI、browser、物理device、課金を必要としない。
5. 既存Node記事はTypeScript実行、test order、VFS、text importであり、package resolution graphとphantom dependencyを扱う記事はない。
6. experimental featureなので採用推奨ではなく、「standardへ切り替えるとどのundeclared importが壊れるかを先に見つける」という対照実験に限定できる。

## 記事で答える問い

- rootへhoistされたpackageを子workspaceが`package.json`へ宣言せずimportした場合、通常Node、`loose` map、`standard` mapの終了codeとerrorはどう変わるか。
- 宣言済みdependencyとrelative / builtin importは3条件すべてで同じ結果になるか。
- 同名packageの異なるversionを2つのworkspaceから読む場合、package mapはどのversionへ解決し、`import.meta.resolve()`または`require.resolve()`のpathはどう変わるか。
- pnpm生成`node_modules/.package-map.json`のどのentryがphantom dependency拒否を表現しているか。
- `standard`で壊れたfixtureをdependency宣言追加で直すと、再install後にmapと実行結果がどう変わるか。
- experimental flagを外した通常実行へ戻す手順と、現段階で本番採用を一般化できない境界は何か。

想定4時間30分〜6時間。repository本体ではなく新しい隔離run directoryを使い、command、stdout / stderr、終了code、生成file hashをexecution logへ残す。

### 0. capability / version gate（30分）

1. OS、architecture、既定`node --version`、`npm --version`を記録する。環境変数全量やcredentialは出力しない。
2. Node.js `26.5.x`または26.4以上をisolated toolchainとして取得し、`node --help`に`--experimental-package-map`があることを確認する。取得不能、major / minor不足、flag不存在なら停止し、挙動を推測しない。
3. pnpm `11.8.0`以上をproject-localまたはpackage runnerで取得し、exact versionを記録する。`node-experimental-package-map`またはmap生成が利用不能なら停止する。
4. 選定テーマはbrowserを使わない。追加でbrowser検証へ広げない。

### 1. workspace fixture（45分）

1. `apps/app-a`、`apps/app-b`、`packages/shared`の3 workspaceを作る。
2. 外部packageはpure JavaScript・install scriptなしの小さいpackage 2つに固定する。可能なら同一packageの2 versionをaliasまたは別workspace dependencyとして固定する。
3. `app-a`はdependencyを正しく宣言し、`app-b`はrootまたはsibling由来で見えてしまうpackageを意図的に未宣言にする。
4. 各entrypointはpackage名、version、`import.meta.resolve()` / `require.resolve()`結果をJSON 1行で出す。期待値をcodeへ埋め込まず、観測結果として保存する。
5. source treeと`package.json`のhashを記録する。

### 2. mapなしbaseline（35分）

1. pnpm install後、通常のNodeで各workspaceを実行する。
2. declared / phantom / relative / builtin importについて終了code、resolved path、versionを保存する。
3. phantom importがbaselineでも失敗するlayoutならfixture不成立として調整する。ただし結果を作るためにmanual symlinkは置かず、package managerのsupported hoisting設定だけを使う。

### 3. `loose` package map（45分）

1. pnpmのpackage map生成を有効にし、typeを`loose`へ固定する。生成された`node_modules/.package-map.json`のpath、size、hash、top-level keyを記録する。
2. pnpm script経由とNodeへflagを明示した直接実行の両方を試し、同じmapが使われたかcommand lineと出力で確認する。
3. baselineと同じmatrixを実行し、成功 / error、resolved versionを表にする。

### 4. `standard` package map（55分）

1. typeだけ`standard`へ変更し、clean installまたは公式手順でmapを再生成する。
2. map JSON diffを取り、appごとのdependencies mappingを抽出する。全文を記事へ貼らず、再現に必要な最小部分だけ引用する。
3. declared dependencyが成功し、undeclared dependencyがnon-zeroと`MODULE_NOT_FOUND`になるかを確認する。Node公式のalgorithmは未宣言bare specifierを`MODULE_NOT_FOUND`にすると定義するが、実測を先取りしない（[Node.js docs](https://nodejs.org/api/packages.html#package-maps)、閲覧: 2026-07-13）。
4. relative、absolute、`node:` builtinはmap対象外としてbaselineと同じか確認する。

### 5. 修正前後の対照（40分）

1. `app-b/package.json`へ不足dependencyを正しく追加する。
2. install / map生成をやり直し、map diff、resolved version、終了codeの変化を記録する。
3. 単にrootへdependencyを追加するだけでは`standard`で許可されるのか、consumer自身の宣言が必要かを観測する。
4. 修正後に全workspace testを再実行する。

### 6. 重複versionとmodule boundary（45分、時間切れ時は省略可）

1. 同一packageの2 versionを別consumerへ宣言し、各consumerが期待するversionへ解決するか確認する。
2. ESMを必須、CommonJSは同じfixtureで短い追加caseにする。両者で結論が異なる場合は別々に記録する。
3. Node docsの複数package ID / 同一URLに関するadvanced caseへは広げない。半日を超えるため別記事候補とする。

### 7. 証拠整理（40分）

1. `no map` / `loose` / `standard` / `standard after fix`を列にし、declared、phantom、relative、builtin、duplicate versionの結果を表にする。
2. exact Node / pnpm version、configuration、map hash、全command、stdout / stderr、終了codeをexecution logへ残す。
3. 「package mapsは安全なsandbox」「pnpmの通常installは常にphantom dependencyを許す」「experimental APIが将来も同じ」と一般化しない。
4. 記事は現行exact versionで観測したdependency resolutionだけを結論にする。

## 完了条件

- Node.js 26.4+とpnpm 11.8+のexact versionが記録されている。
- mapなし、`loose`、`standard`の3条件で同一fixtureを実行している。
- 少なくともdeclared dependency 1件、phantom dependency 1件、relative import 1件、builtin import 1件を終了code付きで比較している。
- 生成package mapとconfigurationのdiffが保存されている。
- phantom dependencyを正しく宣言して直すbefore / afterがある。
- 公式仕様と実測結果を分離し、unsupported / untested条件を明記している。

## riskと停止条件

- Node.js 26.4+を隔離取得できない、または`--experimental-package-map`が存在しない: 停止する。
- pnpm 11.8+でmap生成 / flag注入設定を確認できない: 手書きmapだけの代替記事へすり替えず停止する。
- package managerが生成するJSON formatまたは設定名がexact versionで変わっている: current official release / docsを再確認し、根拠を確定できなければ停止する。
- baselineでphantom dependencyが見えず、supported hoisting設定でも再現できない: 「拒否された」と推測せずfixture不成立として停止または候補2へ戻す。
- experimental featureのcrash、hang、nondeterministic resolutionが出る: timeoutと最小reproductionを記録し、成功結果を補完しない。
- public package取得にnetworkが必要だがregistryへ到達できない: local packageだけへ勝手にscope変更せず停止する。最初からcache hitを成功条件にしない。
- 生成mapへlocal absolute pathが含まれる場合、記事ではplaceholderへredactし、execution logにはworkspace外のsecret pathや環境変数を残さない。
- browser検証へ拡張する場合: `danger-full-access`ではreal browser / Playwrightを許可するが、browser launchとfeature gateが通らなければ停止し、screenやDOM結果を推測しない。今回の計画にはbrowser工程を含めない。
- 半日を超えた場合: duplicate version / CommonJS追加caseを切り、core matrixと修正before / afterを優先する。

## 最終判断

Node.js package mapsは、直近release、package-manager integration、客観的なsuccess / failure、既存記事との差の4点が揃う。特に`standard` / `loose`を同一fixtureで比較すれば、「新機能を動かした」だけでなく、undeclared importを移行前に見つける実務的な記事にできる。experimentalであることをversion gateと停止条件で明示したうえで、次stageのpractice plan対象として採用する。
