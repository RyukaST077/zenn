### 1. Vitest 4のブラウザモードで「見た目が崩れたら落ちる」テストを新人が初めて書いてみた（合計27点 / 最優先）

#### 概要
Vitest 4で安定化したBrowser Modeと、新しい`toMatchScreenshot`（ビジュアルリグレッション）を使い、小さなReactコンポーネントに対して「実ブラウザでのスクショ比較テスト」を書く。JSDOMとの違い、初回ベースライン生成、意図的に見た目を壊したときにテストが落ちる様子までを検証ログにする。

#### なぜ今よさそうか
- Vitest 4でBrowser Modeが正式安定、`toMatchScreenshot`・`toBeInViewport`・Playwright Tracesが追加された一次情報がある
- テストを「実ブラウザで動かす」流れが2026の標準化フェーズ。JSDOM卒業の話題
- スクリーンショットが自然に大量に得られるため、Zenn記事として画像で説明しやすい

#### 新人が検証しやすい理由
公式の最小構成（`vitest` + `@vitest/browser` + `@vitest/browser-playwright`）で始められる。テスト対象は自作の小コンポーネント1つで十分。ヘッドレスChromiumで完結する。

#### 想定読者
Vitest/Jestは触ったことがあるが、ブラウザモード・ビジュアルリグレッションは未経験の新人〜2年目。

#### タイトル案
1. Vitest 4のブラウザモードで「見た目が崩れたら落ちる」テストを初めて書いてみた
2. JSDOMしか知らなかった新人が、Vitest 4の`toMatchScreenshot`を試してみた
3. Vitest 4でビジュアルリグレッションテストを新人が動かすまで（詰まった点つき）

#### 検証でやること
- Vitest 4 + Browser Mode + Playwright provider の最小セットアップ
- 小コンポーネントのブラウザテスト（レンダリング／`toBeInViewport`）
- `toMatchScreenshot` で初回ベースライン生成 → わざとCSSを変えて差分検出→落ちるのを確認

#### 検証手順
1. `pnpm create vite` で最小React、`vitest`/`@vitest/browser`/`@vitest/browser-playwright` を追加
2. `vitest.config` に browser設定（provider: playwright, headless）
3. コンポーネントの描画テストを1本書いて実行
4. `toMatchScreenshot` でベースライン生成、再実行で緑を確認
5. CSSを1px変える等で意図的に差分を作り、赤になる／差分画像が出るのを確認・スクショ

#### 記事構成案
1. はじめに 2. なぜブラウザモードを試すのか 3. 事前に調べたこと（v3との違い/provider別パッケージ化） 4. 環境構築 5. 実際に書いたテスト 6. 詰まった点（provider未インストール・ベースライン差分・CI差異） 7. 触ってみて分かったこと 8. JSDOMと比べて感じたこと 9. どんな人に向くか 10. まとめ

#### 差別化ポイント
「速度比較」ではなく「新機能を新人が最小で動かし、意図的に壊して落とす」検証ログ。日本語での`toMatchScreenshot`実践例はまだ少ない。過去記事（Vite8）とは領域（テスト）が別。

#### 参考リンク
- Vitest公式blog「Vitest 4.0 is out!」 / VoidZero「Announcing Vitest 4.0」
- Vitest Browser Mode Guide

#### 注意点
- OS/フォント差でスクショが揺れやすい → 同一環境で撮る前提を明記
- provider別パッケージ化などv4破壊的変更を踏まえる

### 2. TypeScriptで初めて自作MCPサーバーを作り、Inspectorだけで動作確認してみた（合計26点）

#### 概要
`@modelcontextprotocol/sdk` + zod で、小さなツール（例: 簡単な計算やローカルファイル要約）を1つ持つMCPサーバーをTypeScriptで実装。AIツールに繋ぐ前に、`npx @modelcontextprotocol/inspector --cli` のヘッドレスCLIで `tools/list`・`tools/call` を叩いて動作確認する。「AIクライアント無し・認証無しでどこまで検証できるか」に焦点を当てる。

#### なぜ今よさそうか
- MCPは月間9,700万DL規模、主要AIツールが全対応の一次情報。「自作サーバー」需要が急増
- Inspectorに `--cli` のスクリプタブルモードがあり、AIエージェント単独・ヘッドレスで検証できる

#### 新人が検証しやすい理由
公式チュートリアルが充実。STDIO最小サーバーはファイル1つ＋数十行。`--cli`で単発呼び出しできるので対話UIやClaude Desktop登録（人手）が不要。

#### 想定読者
MCPという言葉は知っているが自作はしたことがない新人〜2年目。

#### タイトル案
1. TypeScriptで初めてMCPサーバーを作って、Inspectorだけで動作確認してみた
2. AIツールに繋ぐ前に、自作MCPサーバーをCLIだけで検証してみた
3. MCP入門：新人が最小のtoolを1つ実装して呼び出すまで（詰まった点つき）

#### 検証でやること
- SDK＋zodで最小STDIOサーバー（tool 1つ、resource 1つ）を実装
- `--cli`で `tools/list` / `tools/call` を叩いて結果確認
- STDIO運用の落とし穴（`console.log`禁止・絶対パス）を体験

#### 検証手順
1. npm init（`type: module`）、`@modelcontextprotocol/sdk` と `zod` を導入
2. tool（入力zodスキーマ付き）を1つ定義したサーバーを書く
3. `tsc` でビルド
4. `npx @modelcontextprotocol/inspector --cli node build/index.js --method tools/list`
5. `--method tools/call --tool-name <name> --tool-arg key=value` で実行・出力確認

#### 記事構成案
1. はじめに 2. なぜMCPを自作するのか 3. 事前に調べたこと（tools/resources/prompts） 4. 環境構築 5. 実装したtool 6. 詰まった点（stdoutログ破損・相対パス・zodスキーマ） 7. Inspector CLIで分かったこと 8. どんな人に向くか 9. まとめ

#### 差別化ポイント
多くの入門記事は「Claude Desktopに登録して動かす（人手）」で終わるが、本記事は **AIクライアント無し・CLIだけで検証** に振る。新人が最初に踏むstdout破損などの詰まりを一次情報として残す。

#### 参考リンク
- MCP公式docs「Build an MCP server」 / TypeScript SDK / Inspector GitHub

#### 注意点
- Claude Desktop連携は人手が要るため記事の検証範囲外と明記
- `DANGEROUSLY_OMIT_AUTH`はローカル検証限定・危険性に触れる
