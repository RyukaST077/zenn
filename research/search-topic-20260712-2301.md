## 除外候補と理由

- **Deno 2.9 `deno desktop`**: 2.9ではexperimentalと公式に明記され、GUI/WebView/OS固有差が核心になる。`danger-full-access`なので実ブラウザやGUIの使用自体は許可されるが、半日で安定した新人向け結論を出すにはcapability riskが高いため除外した（[Deno 2.9](https://deno.com/blog/v2.9)、閲覧: 2026-07-12）。
- **Deno Deploy / Deno Sandbox**: クラウドサービス側の利用条件やアカウントが関わり、登録不要・ローカル完結の既定制約から除外した（[Deno Blog](https://deno.com/blog)、閲覧: 2026-07-12）。
- **Storybook 10.4 agentic setupそのもの**: agent環境と対象アプリの複雑さに結果が依存し、同じ入力での決定的な再現が難しい。agentが利用するサービスによっては認証や契約条件も加わるため、change-aware reviewだけを候補に残した（[Storybook 10.4](https://storybook.js.org/blog/storybook-10-4/)、閲覧: 2026-07-12）。
- **Chrome 150 `text-fit`、画像対応`light-dark()`**: ローカル事前観測のChromeは149で、必要versionを満たす実ブラウザをrun段階で確保できる保証がない。ブラウザlaunchまたはcapability gate失敗時に推測で続行できないため今回は除外した（[What’s new in web UI](https://developer.chrome.com/blog/new-in-web-ui-io26?hl=en)、閲覧: 2026-07-12）。
- **CSS Gap Decorations、npm 12 `allowScripts`、Node.js VFS、`node:test --test-randomize`、Playwright passkey、TypeScript 7 side-by-side**: 既存記事と主題が重複するため除外した。

## 上位3候補

### 1位: Deno 2.9のnpm lockfile移行を対照実験する

- 強み: 同一fixtureについてnpm baseline、Deno初回install、再install、`deno ci`負例を並べられ、version・integrity・exit codeという客観的な証拠が残る。
- 差別化: 国内の週刊Denoは実装を紹介しているが、このリポジトリにはDeno記事がなく、実際の`package-lock.json`から何が維持されるかを測る余地がある。
- 実行条件: Deno 2.9.x、Node/npm、ローカルfixture、npm registryへの通常のpackage取得。無料・認証不要。ブラウザ不要。

### 2位: CSS scroll-triggered animationsとIntersectionObserverを比較する

- 強み: JavaScript削減、entry/exit action、複数要素での`trigger-scope`、`prefers-reduced-motion`を視覚とevent logの両方で示せる。
- 弱み: Chrome系中心の機能で、実ブラウザlaunch成功と`CSS.supports()`相当のcapability確認が必要。どちらかが失敗すれば推測せず停止する必要がある（[Chrome公式](https://developer.chrome.com/blog/scroll-triggered-animations?hl=ja)、閲覧: 2026-07-12）。

### 3位: Deno 2.9 built-in snapshot testingを試す

- 強み: 追加test libraryなしでsnapshot作成、mismatch diff、`--update-snapshots`を短時間で再現できる（[Deno 2.9](https://deno.com/blog/v2.9)、閲覧: 2026-07-12）。
- 弱み: fixtureが単純すぎると公式サンプルの再演になりやすい。custom serializerや不安定値の正規化まで広げると半日枠を超えやすい。

## 選定テーマ

**小さなnpm/Node.jsアプリをfixture化し、Deno 2.9の`deno install`が既存`package-lock.json`から生成する`deno.lock`について、直接・推移依存の解決バージョンとintegrityが維持されるかをnpm baselineと比較する。続けてNode実行、`deno task`、`deno ci`の成功条件とlockfile drift時の失敗条件を記録する。**

想定タイトル:

> npmプロジェクトをDeno 2.9へ2コマンド移行：package-lockのversionとintegrityは本当に引き継がれるか

### 選定理由

1. **適時性**: Deno 2.9は2026-06-25公開で、他package managerのlockfileから`deno.lock`をseedする機能が同releaseの主要項目として公式発表されている（[Deno 2.9](https://deno.com/blog/v2.9)、閲覧: 2026-07-12）。
2. **実行可能性**: CLI、ローカルfixture、公開npm packageだけで検証でき、アカウント、APIキー、OAuth、GUI、物理デバイスを必要としない。現在のローカルDenoは2.8.3なので、run段階では2.9.x取得を最初のcapability gateにする。
3. **記事価値**: 「移行できた」という主観ではなく、npmとDenoのlockfileからversion・integrityを抽出して比較し、`deno ci`の意図的な失敗まで示せる（[lockfile docs](https://docs.deno.com/examples/dependency_lockfile_tutorial/)、閲覧: 2026-07-12）。
4. **実務有用性**: 公式はNode runtimeを維持したままDenoをpackage managerとして使う小さな移行と、`deno.lock`を削除してnpmへ戻す可逆性を示しているため、全面移行を決めていない新人にも判断材料を出せる（[Migrate from npm](https://docs.deno.com/runtime/migrate/migrate_from_npm/)、閲覧: 2026-07-12）。
5. **国内関心と差別化**: 週刊Denoがlockfile seed実装を関連PR付きで紹介している一方、既存記事群にはDenoの実測記事がない（[週刊Deno](https://uki00a.github.io/deno-weekly/articles/2026/06/21.html)、閲覧: 2026-07-12）。

### 記事で答える問い

- `package-lock.json`しかないnpm fixtureでDeno 2.9の`deno install`を初回実行すると、どのメッセージとファイル差分が生じるか。
- 直接依存と推移依存について、npm側と生成された`deno.lock`のversion・integrityは一致するか。
- 同じsourceを`node`、npm script、`deno task`で実行したとき、出力とtest結果は一致するか。
- `package.json`だけを変更してlockfileを更新しない場合、`deno ci`は非0で停止し、どのdriftを報告するか。
- Deno移行を取り消してnpm baselineへ戻したとき、同じtestを再実行できるか。

## 半日検証アウトライン

### 0. version / capability gate（20分）

- 専用run directoryを作り、OS、CPU architecture、Node、npm、Denoのversionと実行ファイルpathを記録する。
- Deno `2.9.x`を安全に取得または選択し、`deno --version`で確認する。2.9.xを取得できなければ、2.8.3の挙動から推測せず停止する。
- npm registryへの取得確認を、選定した無害な公開packageのmetadata取得で行う。DNS/TLS/registry accessが失敗したら再試行は1回までとし、それでも失敗なら停止する。
- 選定テーマはブラウザ不要。補助UIを追加する場合でも、実ブラウザまたはPlaywright launchに失敗したらUI結果を推測せず、CLI核心だけに戻すか、UIが必須になった時点で停止する。

### 1. npm baseline fixture（45分）

- `package.json`、小さなESM entry、Node標準testを持つfixtureを作る。
- direct dependencyが1件以上、transitive dependencyが1件以上になる、install script不要のpackage/versionを明示的に固定する。
- `npm install --package-lock-only`または`npm install`で`package-lock.json`を生成し、`npm ci`、npm script、`node --test`のcommand、exit code、stdout/stderrを記録する。
- `package-lock.json`からpackage名、version、resolved、integrityを決定的なscriptで抽出し、baseline JSONとして保存する。

### 2. Deno 2.9初回移行（55分）

- fixtureを複製し、`package-lock.json`は残したまま`deno.lock`と`node_modules`がないことを確認する。
- Deno 2.9で`deno install`を一度だけ実行し、stdout/stderr、exit code、生成・変更ファイル一覧を保存する。
- `Seeded deno.lock from package-lock.json`相当の観測は実出力に存在する場合だけ記載する。文言が異なる、または出ない場合はそのまま記録する。
- 生成された`deno.lock`からdirect/transitive dependencyのversionとintegrityを抽出し、npm baselineとの一致・不一致を機械比較する。

### 3. 実行互換と再現性（50分）

- Denoが生成した`node_modules`を使い、`node`、`npm test`、`deno task test`を同じfixtureで実行する。
- stdoutの意味的な値、test件数、exit codeを比較する。実行時間は補助情報に留め、少数回の値から性能優劣を断定しない。
- `node_modules`を除去し、`deno ci`で再構築後に同じtestを再実行する。
- 2回目のinstallでlockfileが不用意に変わらないか、hashとdiffで確認する。

### 4. 負例と可逆性（55分）

- branch A: `package.json`の依存rangeだけを変更し、`deno.lock`を更新せず`deno ci`を実行する。非0 exitとdrift messageを記録する。
- branch B: `deno.lock`を退避し、元の`package-lock.json`を使って`npm ci`へ戻し、baseline testを再実行する。
- lifecycle scriptを必要とするpackageやnative addonは追加せず、script approvalとABI差は今回の未検証範囲として明記する。
- `deno.lock`だけで完全offlineになるとは主張せず、必要なら`--cached-only`を補助確認に留める。lockfileとvendorの役割の違いは公式説明に従う（[Supply chain management](https://docs.deno.com/runtime/packages/supply_chain/)、閲覧: 2026-07-12）。

### 5. 証拠整理（50分）

- 全command、cwd、開始終了時刻、exit code、stdout/stderrをexecution logへ残す。
- npm baseline / Deno seed / `deno ci` / npm rollbackの4列で、version、integrity、test、lockfile差分を比較表にする。
- 公式仕様、ローカル観測、編集上の解釈を別欄に分ける。
- 成功、不一致、未確認を明示し、Deno 2.9以外のversionへ一般化しない。

### 完了条件

- Deno 2.9.xのversion gateと取得方法が記録されている。
- npm baselineのdirect/transitive dependencyについてversionとintegrityを抽出できている。
- Deno初回installの実出力、生成ファイル、`deno.lock`抽出結果がある。
- npmとDenoのversion・integrity比較が機械的にpass/failを返している。
- Node/npm script/`deno task`/`deno ci`のexit codeとtest結果が記録されている。
- lockfile drift負例が非0で止まるかを実測し、止まらなければその事実を失敗として残している。
- npmへのrollback後にbaseline testを再実行している。

## リスクと停止条件

- **Deno version不足**: Deno 2.9.xを利用できなければ停止する。手元2.8.3や公式記事から2.9の結果を推測しない。
- **registry access失敗**: npm registryへの取得が2回失敗したら停止する。既存cacheだけの成功をonline初回移行の証拠に置き換えない。
- **fixtureの不適合**: 選んだpackageがinstall script、native build、認証付きregistryを要求したら、そのpackageを捨ててinstall script不要のfixtureへ一度だけ差し替える。再度同種の問題が出たら停止する。
- **lockfile format差**: parserが想定formatを扱えなければraw JSON/YAMLを壊して手修正せず、schemaを読み直して抽出scriptを直す。2時間以内にdirect/transitive双方を抽出できなければ核心未達として停止する。
- **integrity比較の誤り**: npmとDenoでfield表現が異なる場合、文字列一致だけで「依存が変わった」と断定しない。package/versionとhash algorithm/valueを正規化できた範囲だけ判定し、できない項目は未確認にする。
- **既存workspace汚染**: runは専用directoryで行い、このrepositoryの既存`package-lock.json`、`node_modules`、記事をfixtureとして直接変更しない。
- **ブラウザcapability**: CLIテーマなのでブラウザを必須にしない。追加の実ブラウザ/Playwright検証を行う場合、launch失敗時は結果を推測せず停止し、過去runのbrowser結果を流用しない。
- **時間超過**: 4時間15分でintegrity比較と`deno ci`負例まで完了しない場合、minimum dependency ageやsnapshot testへ広げない。核心2項目を未確認のまま成功記事へ進めない。
- **過剰な一般化**: 1 fixtureの成功から「すべてのnpm projectが無修正で移行できる」と書かない。native addon、lifecycle script、private registry、workspaceは未検証範囲として残す。

## 結論

選定テーマは、**Deno 2.9の`deno install`で既存npm fixtureの`package-lock.json`から`deno.lock`を生成し、direct/transitive dependencyのversionとintegrity、Node/npm/Denoでのtest、`deno ci`のdrift失敗、npmへのrollbackを同一fixtureで対照検証するハンズオン**とする。2026-06-25公開の新機能で時宜性があり、国内コミュニティでも実装が追跡されている。CLIだけで無料・認証不要・ローカル中心に検証でき、既存記事にはDeno主題がないため差別化も高い。run段階ではDeno 2.9.xとregistry accessをcapability gateにし、満たせなければ結果を推測せず停止する。
