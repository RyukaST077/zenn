## 上位候補の詳細

### 1. `using`/`Symbol.dispose` を try/finally と書き比べてみた

#### 概要
ES2026 で標準化された明示的資源管理（`using` / `await using` / `Symbol.dispose` / `Symbol.asyncDispose`）を、従来の `try/finally` による後始末と同じ題材で書き比べる。Node 26（既定ランタイム）でフラグなしに動く。

#### なぜ今よさそうか
ES2026 入りで Node 24+・TS 5.2+ が対応済み。解説記事は増えたが「同じ処理を try/finally と using で書いて挙動差を観察した」検証ログはまだ手薄。過去記事の「Temporal を Date と書き比べ」「Map.getOrInsert を書き比べ」と同じ勝ちパターンで、機能だけ差し替えられる。

#### 新人が検証しやすい理由
ファイルハンドル・一時ディレクトリ・DBコネクション擬似オブジェクトなど、`Symbol.dispose` を実装した小さなクラスを自作すれば完結。外部依存ゼロ。

#### 想定読者
try/finally で後始末を書いている新人〜2年目。

#### タイトル案
1. Node 26 の `using` を try/finally と同じ処理で書き比べてみた
2. `Symbol.dispose` で後始末を自動化：新人が try/finally から乗り換えてみた
3. `using`/`await using` の解放順序を実際に動かして確かめた

#### 検証でやること
- `Symbol.dispose` を実装した自作リソースクラスで解放タイミングを観察
- 複数 `using` の LIFO（後入れ先出し）解放順を確認
- 例外時・early return 時にも解放されるかを確認
- `await using` と `Symbol.asyncDispose` の非同期解放を確認

#### 検証手順
1. `node -v` で 26 系を確認（TS を挟むなら tsc/tsx バージョンも記録）
2. `Symbol.dispose` を持つクラスを定義し `using r = new R()` で解放ログを出す
3. `using` を2つ並べて解放順（LIFO）を観察
4. `throw` / `return` を挟んで解放されるか確認
5. `await using` + `Symbol.asyncDispose` で非同期解放を確認、try/finally 版と比較

#### 記事構成案
1. はじめに 2. なぜ試すのか 3. 事前に調べたこと 4. 環境構築 5. try/finally 版 6. using 版に書き換え 7. 解放順・例外時の挙動 8. 詰まった点 9. 使いどころ 10. まとめ

#### 差別化ポイント
解説ではなく「同じ題材の書き比べ＋解放順の実測ログ」。過去のNode 26書き比べ路線の延長で、読者の信頼を積み上げやすい。

#### 参考リンク
- MDN Symbol.dispose / Total TypeScript "using" / nodejs.org v26 release notes

#### 注意点
TS 経由なら down-compile 挙動が混ざるので、まず素の Node 26 で確認してから TS を足す。

### 2. Node 26 の `randomUUIDv7` で時刻順にソートできるUUIDを試す

#### 概要
Node 26.1.0 で追加された `crypto.randomUUIDv7()` を使い、生成した UUID が時刻順（先頭48bitがms精度タイムスタンプ）でソートできることを実測。`randomUUID()`（v4）との差を比較する。

#### なぜ今よさそうか
DB主キーの断片化対策として UUIDv7 が注目される中、外部ライブラリなしで標準生成できるようになった一次情報が新しい。

#### 新人が検証しやすい理由
生成してソートするだけ。SQLite（標準の node:sqlite でも可）に挿入してインデックス順を眺める程度まで広げられる。

#### 想定読者
UUID主キーの並び順・断片化が気になる新人〜2年目。

#### タイトル案
1. Node 26 の `randomUUIDv7` で「並ぶUUID」を試してみた
2. v4 と v7 を並べて、UUIDが時刻順にソートされるか確かめた
3. `crypto.randomUUIDv7()` を DBキー目線で触ってみた

#### 検証でやること
- v7 を連続生成して文字列ソートと生成順が一致するか確認
- v4 と比較してソート性の差を観察
- `disableEntropyCache` の有無で挙動が変わるか確認

#### 検証手順
1. Node 26 系を確認 2. v7 を100件生成→配列を sort→生成順と一致するか 3. v4 で同じことをして崩れるのを確認 4. `{disableEntropyCache:true}` 版を比較 5. node:sqlite に挿入して並びを確認（任意）

#### 記事構成案
1. はじめに 2. UUIDv7とは 3. 環境 4. 生成して観察 5. v4との比較 6. 詰まった点 7. DBキー目線の所感 8. まとめ

#### 差別化ポイント
「標準APIだけで・実測ソート結果つき」。ライブラリ紹介で終わらない検証ログ。

#### 参考リンク
- nodejs.org crypto ドキュメント / commit b267f6bca3

#### 注意点
UUIDv7 は乱数部があるため「同一ms内の厳密順序は保証されない」点を明記する。

### 3. MPA View Transitions を Playwright で撮って確かめた

#### 概要
`@view-transition { navigation: auto }` による複数ページ（MPA）遷移アニメーションを、localhost の2ページ構成で実装し、Playwright で遷移前後をスクリーンショット観察する。

#### なぜ今よさそうか
Interop/Baseline 2026 で SPA だけでなく MPA 遷移が主要エンジンに到達。過去のCSS記事（Anchor/Gap/focusgroup）とは別機能で、差別化しやすい。

#### 新人が検証しやすい理由
静的HTML 2枚 + わずかなCSSで再現でき、Playwright で自動操作・撮影まで完結。

#### 想定読者
ページ遷移演出を JS フレームワークなしで入れたい新人。

#### タイトル案
1. MPA の View Transitions を素のHTML2枚で試してみた
2. `@view-transition` のページ遷移を Playwright で撮って確かめた
3. JSなしのページ遷移アニメを新人が触ってみた

#### 検証でやること

- Playwright（Chromium）で遷移を発火しスクショ

#### 記事構成案
