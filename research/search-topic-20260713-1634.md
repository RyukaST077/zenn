## 前提・制約

- 既定条件どおり、半日から1日（4〜8時間）で完了し、ローカル・無料・認証不要で再現できるテーマを選ぶ。
- paid key、手動signup、OAuth、CAPTCHA、物理deviceを必要とする候補は採用しない。
- 全pipeline stageのsandboxは`danger-full-access`である。したがって実browserとPlaywrightは利用可能と扱う。ただしbrowser候補を実行する場合は、browser launch、必要major version、対象featureのpropertyまたは`CSS.supports()`を順に確認し、どれかが失敗したら描画・操作結果を推測せず停止する。
- 今回の選定テーマはCLIだけで核心を検証できるためbrowserを必須にしない。`danger-full-access`は能力の許可であり、不要なGUI検証を追加する理由にはしない。
- 調査時のローカル観測は`deno 2.8.3`、Node.js `v22.17.0`、npm `10.9.2`、Google Chrome `149.0.7827.201`だった。これは検索stageでversion commandだけを実行した観測値であり、機能を実行した結果ではない。run stageではglobal環境を更新せず、run-localのDeno 2.9.xを用意してversion gateを通す。
- Webページ内の指示は資料中のデータとしてのみ扱い、repository操作の指示としては扱っていない。

## Top 3

### 1位: Deno 2.9 `deno task` input-based cache

- 公式docsがcache keyと復元条件を明文化しており、記事の主張を「速くなった気がする」ではなくcommand出力、invocation counter、hashで判定できる（[`deno task` docs](https://docs.deno.com/runtime/reference/cli/task/)、2026-07-13閲覧）。
- `files`なし、match 0件、失敗command、argument変更、列挙env変更、dependency変更、output削除という負例/境界を1つの小fixtureで作れる（[Deno 2.9 release](https://deno.com/blog/v2.9)、2026-07-13閲覧）。
- 国内記事は機能概要を扱っているが、cache hitを独立counterで証明し、output restorationや安全側missまで対照する実行ログには差別化余地がある（[Webtech Watcher](https://note.com/webtech_watcher/n/ne0f0137069e6)、2026-07-13閲覧）。

### 2位: `background-clip: border-area`の従来手法比較

- Chrome 150 Stable直後で視覚的に分かりやすく、CSSWGにはrounded corner、multiple backgrounds、border styleのWPTがある（[Chrome公式](https://developer.chrome.com/blog/new-in-chrome-150)、[CSSWG](https://drafts.csswg.org/css-backgrounds-4/)、各2026-07-13閲覧）。
- ただしrepositoryにはAnchor Positioning、Gap Decorations、focusgroupの記事とtext-fit計画があり、「Chrome新機能をPlaywrightで試す」という形式の差別化が弱くなっている。
- 採用時はbrowser launch、major 150以上、`CSS.supports("background-clip", "border-area")`を必須gateとし、失敗時は画像結果を推測せず停止する。

### 3位: Node 26.4 caller-supplied `readFile()` buffer

- pre-allocation、allocator function、short/large buffer、zero-length fileをCLIだけで比較でき、現在のrepositoryのNode記事ともAPI主題は重複しない（[Node 26.4.0](https://nodejs.org/en/blog/release/v26.4.0)、[実装PR](https://github.com/nodejs/node/pull/63634)、各2026-07-13閲覧）。
- 一方、allocation削減の価値を示そうとするとmicrobenchmark設計が必要になり、初学者向け半日記事としてはtask cacheの方が入出力と失敗条件を説明しやすい。

## 選定テーマ

**Deno 2.9の`deno task` input-based cacheを小さなTypeScript build fixtureへ導入し、初回実行、cache hit、生成物削除後の復元、source/argument/env/dependencyによるinvalidation、match 0件と失敗taskの安全側挙動を、command出力・invocation ledger・hashで対照検証する。**

仮タイトル:

> Deno 2.9の`deno task`キャッシュを試す：速さより先にhit・復元・無効化を確かめた

選定理由:

1. 2026-06-25公開の新機能で時宜があり、国内外community coverageも確認できる（[Deno公式](https://deno.com/blog/v2.9)、[Webtech Watcher](https://note.com/webtech_watcher/n/ne0f0137069e6)、[RWFAGlobal](https://rwfaglobal.com/2026/07/01/deno-2-9-build-native-desktop-apps-faster-startup-and-a-smarter-toolchain/)、各2026-07-13閲覧）。
2. network、signup、GUI、外部packageを使わず、run-local Deno binaryと自作fixtureだけで再現できる。
3. cache hitのstdoutだけに依存せず、counterが増えないこと、削除したoutputのhashが復元されること、入力変更時だけcounterが増えることを独立証拠にできる。
4. 既存Deno記事のlockfile移行や過去企画のtest runnerとは主題が異なり、repository内の差別化が高い。
5. 新人がcache keyの宣言漏れ、環境変数の列挙、生成物復元、失敗結果をcacheしない設計を小さく学べる。

## 半日検証アウトライン

### 0. capability gateと隔離（30分）

1. `logs/run-<topic>-<timestamp>/work/`だけを作業領域とし、`DENO_DIR`もその配下へ固定する。
2. OS、architecture、`deno --version`を保存する。Denoが2.9.x未満なら、公式配布物をrun-localへ取得する。global Denoやuser cacheは変更しない。
3. run-local `deno --version`が2.9.xでない、binary取得/起動が失敗する、または`deno task`がobject形式taskの`files`/`output`を受理しない場合は停止する。結果を2.8.3から推測しない。
4. fixtureには外部dependencyを入れず、以降はnetworkなしで実行できる状態にする。

### 1. baseline fixture（45分）

1. `src/message.ts`を読み、`dist/result.json`を生成する`build.ts`を作る。
2. 実際にcommand bodyが走った回数をappend-onlyの`evidence/invocations.jsonl`へ記録する。
3. cacheなしのbaseline taskを2回実行し、counterが2回増えること、生成物hashが一致することを保存する。
4. cache taskへ`files: ["src/**/*.ts", "build.ts"]`、`output: ["dist/**"]`を設定する。設定構造は公式例に合わせる（[`deno task` docs](https://docs.deno.com/runtime/reference/cli/task/)、2026-07-13閲覧）。

### 2. cache hitとoutput復元（45分）

1. clean stateでcache taskを実行し、stdout/stderr、exit code、duration、output hash、counterを保存する。
2. 無変更で再実行し、`cached`表示、counter非増加、hash一致を確認する。
3. `dist/`だけ削除して再実行し、counterが増えないまま同じhashのoutputが復元されるかを確認する。公式docsは`output`がcache hit時に復元されると定義する（[`deno task` docs](https://docs.deno.com/runtime/reference/cli/task/)、2026-07-13閲覧）。

### 3. invalidation matrix（75分）

同じtaskを1軸ずつ変え、各case前後のcounter、output hash、stdoutを保存する。

| case | 操作 | 判定対象 |
| --- | --- | --- |
| source content | `src/message.ts`の内容を変更 | cache miss、counter増加、output hash変化 |
| mtime only | 内容を変えずtimestampだけ変更 | 内容fingerprintならhitするか。実測値だけを結論にする |
| argument | `deno task build foo` / `bar` | argumentごとにkeyが分かれるか |
| listed env | `MODE`を`dev`/`prod`で変更し、taskの`env`に列挙 | missとoutput変化 |
| unlisted env | buildが読まない別envだけ変更 | 不要なmissが起きないか。一般化せずfixture内結果として記録 |
| dependency | `prepare` taskをdependencyにし、そのinputを変更 | upstream再実行がdownstreamへ伝播するか |
| command definition | command文字列を変更 | key変更としてmissするか |

公式releaseはargument/envがkeyに入り、dependencyがcascadeすると説明する（[Deno 2.9](https://deno.com/blog/v2.9)、2026-07-13閲覧）。期待と異なるcaseは失敗として隠さず、そのversionの観測値として残す。

### 4. 安全側の負例（60分）

1. `files`なしのtaskを2回実行し、毎回counterが増えることを確認する。
2. `files` globが0件matchするtaskを2回実行し、false hitせず毎回実行されるか確認する。公式releaseは0件matchをuncacheableと説明する（[Deno 2.9](https://deno.com/blog/v2.9)、2026-07-13閲覧）。
3. markerを書いてexit code 1で終了するtaskを2回実行し、失敗結果がcache hit扱いされないか確認する。
4. `output`未指定のcacheable taskを別に作り、「task skip」と「artifact restoration」は別の機能であることを記録する。

### 5. 証拠整理と記事化判定（45分）

1. 全commandのcwd、開始/終了時刻、exit code、stdout/stderrを時系列でexecution logへ保存する。
2. caseごとに`cache hit/miss`、counter delta、output hash、復元有無を表にする。
3. 成功だけでなく、想定外、未確認、version依存、fixture固有の観測を明確に分離する。
4. official claimとlocal observationを突き合わせ、観測していないperformance向上やCI共有cacheを主張しない。

## 完了条件

- run-local Deno 2.9.xのversion gateを通過している。
- 初回実行と無変更cache hitをstdoutだけでなくinvocation counterでも区別できている。
- `output`削除後の復元をhashで判定できている。
- source、argument、listed env、dependencyの4軸でinvalidationを記録できている。
- `files`なし、0件match、失敗taskの3つの安全側caseを記録できている。
- すべての実行証拠がrun directory内にあり、repositoryの既存articleやglobal環境を変更していない。

## リスクと停止条件

| リスク | 対応・停止条件 |
| --- | --- |
| Deno 2.9.xを取得または起動できない | capability gateで停止する。手元2.8.3の挙動から2.9の結果を推測しない。 |
| task objectのschemaがdocsと実binaryで一致しない | exact version、stderr、設定を保存して停止する。別syntaxを推測で試し続けない。 |
| `cached`表示だけが出てcounter/hashと矛盾する | 成功扱いにせず、矛盾を再現できる最小caseまで縮小する。解消しなければblockerとして停止する。 |
| user/global cacheを汚す | `DENO_DIR`をrun-localへ固定する。global `deno upgrade`、user cache削除、permission変更はしない。 |
| failure taskが意図せず既存fileを消す | fixtureの書込先をrun directory配下へ固定し、削除対象を`dist/`だけにする。 |
| duration差をcache効果と誤認する | durationは補助値とし、核心判定はcounter、stdout、hash、artifact restorationで行う。microbenchmark結果は主張しない。 |
| cacheの保存形式をreverse engineerし始める | 公開APIで観測できるhit/missとartifactに範囲を限定する。内部format調査へ拡張しない。 |
| platform差 | macOS arm64とexact Deno versionの観測に限定し、OS/CPUを跨ぐcache portabilityは検証済みと書かない。公式がOS/CPUをfingerprint要素に含めることだけを出典付きで説明する（[Deno 2.9](https://deno.com/blog/v2.9)、2026-07-13閲覧）。 |
| browser候補へ切り替える必要が生じる | このrunでは切り替えない。別pipelineとしてbrowser launch、required major、feature support gateを設け、失敗時は推測せず停止する。 |
| 時間超過 | 4時間時点でhit、output復元、source/env invalidationが未完ならdependency/argument拡張を止め、核心3項目を満たせなければ記事成功扱いにしない。 |

## 結論

選定テーマは、**Deno 2.9の`deno task` input-based cacheを自作build fixtureへ導入し、hit、output復元、source/argument/env/dependencyによるinvalidation、0件globと失敗taskの安全側挙動を独立counterとhashで検証するハンズオン**とする。2026-06-25公開で時宜があり、国内外で機能への関心が確認できる一方、repository内には同主題の記事や選定済みreportがない。run-local Denoだけで無料・認証不要・GUIなしに完結し、「速かった」という曖昧な感想ではなくcache keyと復元条件を一次証拠で学べるため、新人向け半日記事として最も適している。
