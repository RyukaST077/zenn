## 上位候補の詳細

### 1. Node 26 の `Map.getOrInsert` と `Iterator.concat` で「get-or-set定型」を書き比べてみた

#### 概要
Node 26 が同梱する V8 14.6 で、`Map.prototype.getOrInsert` / `getOrInsertComputed`（WeakMapにも）と `Iterator.concat` が既定で使えるようになった。誰もが書く「キーがあれば取得、なければ入れて取得」という定型と、配列を作らずイテレータを連結する処理を、旧来の書き方と新APIで書き比べる。

#### なぜ今よさそうか
- Node 26（2026-05リリース）の既定機能で、フラグ不要。すぐ使える一次情報が揃っている。
- get-or-set定型は集計・グルーピング・キャッシュで頻出。実務直結で刺さる。

#### 新人が検証しやすい理由
`node` 一発で動く。追加依存なし。数十行のスクリプトで完結する。

#### 想定読者
Map/配列でグルーピングや集計を書いたことがある新人〜実務2年目。

#### タイトル案
1. Node 26 の `Map.getOrInsert` で「あれば取得、なければ入れる」定型を消してみた
2. `Iterator.concat` で配列を作らずにイテレータを連結してみた（Node 26）
3. Node 26で既定になったV8 14.6の新メソッドを旧来の書き方と書き比べてみた

#### 検証でやること
- `getOrInsert` / `getOrInsertComputed` と `??=`・分岐での旧来実装の書き比べ
- `getOrInsertComputed` のコールバックが「キー未存在時のみ」呼ばれることの確認（副作用ログで実証）
- `Iterator.concat` で複数イテレータを遅延連結し、`.map/.filter/.take` を中間配列なしで繋ぐ
- 大きめ入力で中間配列版とメモリ/挙動を軽く比較

#### 検証手順
1. `node -v` で26系を確認（無ければnvm等で用意）
2. グルーピング集計を旧来（`if`/`??=`）で実装
3. `getOrInsert` / `getOrInsertComputed` で書き換え、出力が一致することを確認
4. コールバック呼び出し回数をカウントして遅延評価を実証
5. `Iterator.concat` で複数ソースを連結し遅延パイプラインを組む

#### 記事構成案
1. はじめに 2. なぜ試すか 3. 事前に調べたこと 4. 環境構築 5. 実際に試したこと（書き比べ） 6. 詰まった点 7. 分かったこと 8. 既存の書き方と比べて 9. どんな人向きか 10. まとめ

#### 差別化ポイント
既存のNode 26記事群（Temporal/ffi/text imports/package maps）が触れていない「V8のコレクション新メソッド」に絞る。技術解説ではなく「定型コードがどれだけ短くなるか」の実測に寄せる。

- https://maximov.by/nodejs-26-complete-guide.html

#### 注意点
- APIはV8/Node版に依存。動かした `node -v` と日付を明記する。
- 「速い」と断定せず、簡易計測であることを明示する。

### 2. Bun 1.3.14 の `Bun.Image` で画像リサイズを依存ゼロで試し、sharpと比べてみた

#### 概要
Bun 1.3.14 で追加された組み込み画像処理API `Bun.Image` を使い、リサイズ・回転・WebP変換をローカルで試す。従来 sharp を入れていた処理を依存ゼロで書き、出力とざっくりした速度を比較する。

#### なぜ今よさそうか
- 2026年のBunアップデートの目玉の一つで新しい。
- 画像リサイズ/サムネ生成は個人開発でも頻出。「依存ゼロでできる」は刺さる。

#### 新人が検証しやすい理由
`bun` でスクリプトを走らせ、生成画像を確認するだけ。チェーンAPIが直感的。

#### 想定読者
画像アップロード/サムネ生成に sharp を使ったことがある、または興味がある新人。

#### タイトル案
1. Bun 1.3.14 の `Bun.Image` でサムネ生成を依存ゼロにしてみた
2. sharpを卒業できる? `Bun.Image` でリサイズ→WebP変換を試した
3. `Bun.Image` のチェーンAPIで画像処理を書いてみた（新人検証ログ）

#### 検証でやること
- サンプル画像を用意し `.resize().webp().write()` でサムネ生成
- `.rotate()/.flip()/.modulate()` などチェーンを試す
- `.metadata()` で出力サイズ/形式を確認、`.placeholder()` も試す
- 同じ処理を sharp でも実装し、出力サイズと処理時間をざっくり比較

#### 検証手順
1. Bunを1.3.14以上に更新（`bun --version`）
2. テスト画像を生成/用意
3. `Bun.Image` でリサイズ→WebP変換を実装し出力確認
4. `npm i sharp` で同処理を書き、出力と時間を比較
5. Playwrightで生成画像をHTMLに並べてスクショ（記事用素材）

### 3. CSS `contrast-color()` で文字色の白黒自動化をJSなしで試してみた

#### 概要
2026年4月にBaseline化した CSS `contrast-color()` を使い、背景色に応じて白/黒の文字色を自動選択するUIをJSなしで作る。`@supports` フォールバックも試す。

#### なぜ今よさそうか
- Baseline Newly available（2026-04, Chrome147/FF146/Safari26）で「今使える」段階。
- アクセシビリティ（WCAGコントラスト）を手作業のペア管理なしで担保できる。

#### 新人が検証しやすい理由
HTMLとCSSだけ。Playwrightで computed color を取得しスクショで確認できる（このリポジトリのPlaywright運用と相性◎）。

#### 想定読者
テーマカラーやバッジ配色でコントラストに悩んだことのあるフロント新人。

#### タイトル案
1. CSS `contrast-color()` で文字色の白黒を自動化してみた（JSなし）
2. 背景色に応じた文字色、もう手計算しない — `contrast-color()` を試した
3. `contrast-color()` はどこまで使える? Baseline化した新関数を新人が検証

#### 検証でやること
- 複数の背景色のカードに `color: contrast-color(var(--bg))` を適用
- `@supports (color: contrast-color(red))` でフォールバック
- Playwrightで各要素の `getComputedStyle().color` を取得し、白/黒判定を確認
- ライト/ダークやグラデ背景など境界ケースを試す

#### 検証手順
1. カード群のHTML/CSSを作成
2. 背景色を段階的に変え文字色の切り替わりを確認
3. `@supports` フォールバックを追加
4. Playwrightで computed color を取得しアサート＋スクショ
5. うまくいかない配色（中間トーン）を探して記録

#### 記事構成案
（テンプレ標準10節）

#### 差別化ポイント
既出のCSS記事（anchor positioning / gap decorations / focusgroup）とは別機能。Playwrightでcomputed値まで検証する「動作確認込み」で差別化。

#### 参考リンク
- https://developer.mozilla.org/en-US/docs/Web/CSS/color_value/contrast-color
