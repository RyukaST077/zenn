## 上位候補の詳細

### 1. Bun 1.3 の Bun.SQL で SQLite を触ってみた

#### 概要
Bun 1.3 で追加された `Bun.SQL` は MySQL/PostgreSQL/SQLite を**同一API**で扱える内蔵DBクライアント。中でも SQLite は追加依存・サーバ不要で、`bun run` だけで CRUD が書ける。新人がテーブル作成→INSERT→SELECT→パラメータバインドまでを一通り試し、従来の `bun:sqlite` や Node の別ドライバとの書き味の違いをまとめる。

#### なぜ今よさそうか
- 1.3 の目玉機能で、国内（Publickey1）・海外（InfoQ, LogRocket）ともにニュース化直後。
- 既存記事に **Bun が1本も無い**＝リポジトリ内で完全な新規領域。
- SQLite ならサービス不要・完全ローカルで AIエージェント単独完結。

#### 新人が検証しやすい理由
`bun install` 後は数行のスクリプトで動く。DBサーバ構築が不要で、詰まりどころ（型・タグ付きテンプレート・トランザクション）も小さく切り出せる。

#### 想定読者
Bun を名前だけ知っている新人／Node で DB を触った経験が浅い人。

#### タイトル案
1. Bun 1.3 の Bun.SQL で SQLite を初めて触ってみた
2. 「1つのAPIで3つのDB」Bun.SQL を新人が SQLite で試してみた
3. Bun.SQL の SQLite を触って詰まったところと分かったこと

#### 検証でやること
- Bun 1.3 系のバージョン固定インストールと確認
- `Bun.SQL` で SQLite に接続しテーブル作成
- INSERT / SELECT / パラメータバインド / トランザクション
- 旧 `bun:sqlite` との書き味・戻り値の違いを比較

#### 検証手順
1. Bun をインストールし `bun --version` を記録（バージョン固定）
2. `sqlite://` で接続する最小スクリプトを書く
3. CRUD を実行し結果とエラーを全文記録
4. パラメータバインド・トランザクションを試す
5. 旧APIとの差分・詰まった点を整理

#### 記事構成案
1. はじめに 2. なぜBun.SQLを試すのか 3. 事前に調べたこと 4. 環境構築 5. 実際に試したこと 6. 詰まった点 7. 触ってみて分かったこと 8. 既存（bun:sqlite / Node）と比べて 9. どんな人に向くか 10. まとめ

#### 差別化ポイント
「統一API」の触れ込みを **SQLite単体で新人が最短検証**し、旧APIとの差分を一次ログで示す。既存の和文は旧 `bun:sqlite` 中心で、1.3の Bun.SQL 新人検証はまだ薄い。

#### 参考リンク
- Bun公式blog: https://bun.com/blog/bun-v1.3
- Bun SQLite docs: https://bun.com/docs/runtime/sqlite
- Publickey1: https://www.publickey1.jp/blog/25/javascriptbunbunsqlmysqlposgrersqlsqlite.html
- InfoQ: https://www.infoq.com/news/2026/01/bun-v3-1-release/

#### 注意点
MySQL/PostgreSQL部分はサーバが要るため **SQLiteに絞る**。バージョン差で API が動く/動かないがあるので `bun --version` を必ず明記。断定せず「試した範囲」を強調。

### 2. Bun 1.3 の isolated installs 既定化で phantom 依存が消えるか試す

#### 概要
Bun 1.3 で workspace の `isolated` インストールが既定化。pnpm風の厳格分離で、宣言していない依存（phantom dependency）を import できなくなる。小さな monorepo fixture を作り、hoisted と isolated で phantom import の可否がどう変わるかを実験する。

#### なぜ今よさそうか
既定化が HN で議論になったばかり。「なぜ壊れる/なぜ守られる」を新人目線で図解できる。

#### 新人が検証しやすい理由
依存を1つ用意して「宣言せず import」する fixture を作るだけ。完全ローカルで再現可能。

#### 想定読者
pnpm/npm の依存解決に不安がある新人〜2年目。

#### タイトル案
1. Bun 1.3 の isolated installs で phantom 依存が本当に消えるか試した
2. 「宣言してない依存」を import して Bun の isolated 既定化を検証してみた
3. pnpm風 isolated installs を新人が hoisted と比べてみた

#### 検証でやること
- 最小 monorepo fixture 作成
- 未宣言依存を import するコードを用意
- `--linker=hoisted` と `isolated` で挙動比較
- `bun why` で依存チェーンを確認

#### 検証手順
1. workspace 付き fixture を作成
2. 未宣言 import を書いて hoisted で動くことを確認
3. isolated（既定）で失敗する様子とエラー全文を記録
4. 正しく宣言して解消
5. `bun why` の出力で違いを可視化

#### 記事構成案
1. はじめに 2. なぜ試すか 3. phantom依存とは 4. 環境構築 5. 実際の挙動 6. 詰まった点 7. 分かったこと 8. pnpm/npmとの比較 9. 向く人 10. まとめ

#### 差別化ポイント
概念解説でなく **fixtureで再現した一次ログ**。`bun why` を絡めて依存の見える化まで行う。

#### 参考リンク
- Bun isolated installs docs: https://bun.com/docs/pm/isolated-installs
- HN: https://news.ycombinator.com/item?id=44611467
- Bun公式blog: https://bun.com/blog/bun-v1.3

#### 注意点
OS依存の既知バグ（Windowsのisolatedバグ）があったため、検証OSを明記。既定挙動はバージョンで変わりうるので固定。
