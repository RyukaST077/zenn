## 上位候補の詳細

### 1. ESLint/oxlintしか知らない新人がBiomeの「型認識lint」を試してみた

#### 概要
Biomeが stable で提供する「tscに依存しない型認識リンティング（type-aware linting）」を、小さなTypeScriptプロジェクトで実際に動かし、型情報がないと検出できないバグをどこまで拾えるか検証する。

#### なぜ今よさそうか
- ESLint/Prettier置換として Biome・oxlint の比較が国内外で活発。型認識は「Biomeにあってoxlintに無い」明確な差別化点で、話題の核心。
- 過去記事で oxlint を扱っているため、同じ筆者が「型認識という違い」を語れる自然な続き。

#### 新人が検証しやすい理由
- `npm i -D @biomejs/biome` と `biome init` だけで開始でき、CLI出力がそのまま素材になる。
- 型認識ルールが「効くコード / 効かないコード」を自作して比較でき、学びが可視化しやすい。

#### 想定読者
ESLint/Prettier または oxlint を使っていて、Biomeの型認識lintが実務で使えるか気になる新人〜2年目。

#### タイトル案
1. ESLint/oxlintしか知らない新人がBiomeの「型認識lint」を試してみた
2. tscなしで型を見るlint？ Biomeの type-aware linting を触ってみた
3. oxlintに無くてBiomeにある「型認識lint」を新人が検証してみた

#### 検証でやること
- Biomeを最新版で導入し `biome init`
- 型認識ルールを有効化して、型依存のバグ（誤ったPromise扱い等）を検出できるか確認
- oxlint/ESLintでは拾えない例と対比

#### 検証手順
1. 小さなTS/Reactサンプルを用意（またはoxlint記事の題材を流用）
2. `npm i -D @biomejs/biome` → `npx biome init`
3. 型認識ルールを設定で有効化
4. わざと型依存のバグを仕込み `npx biome check` で検出を確認
5. 出力ログ・検出/未検出の差をスクショで記録

#### 記事構成案
1. はじめに / 2. なぜBiomeの型認識lintを試すのか / 3. 事前に調べたこと（oxlintとの違い） / 4. 環境構築 / 5. 実際に試したこと / 6. 詰まった点 / 7. 触ってみて分かったこと / 8. oxlint・ESLintと比べて感じたこと / 9. どんな人に向いていそうか / 10. まとめ

#### 差別化ポイント
既存のBiome紹介は「速い・1ツール完結」中心。本記事は「型認識という機能軸」に絞り、oxlint経験者目線で検出できる/できないを具体例で見せる。

#### 参考リンク
- Biome Roadmap 2026: https://biomejs.dev/blog/roadmap-2026/
- Migrate from ESLint and Prettier: https://biomejs.dev/guides/migrate-eslint-prettier/

#### 注意点
- 型認識ルールの正式名・有効化方法は公式ドキュメントで最新を確認（実験扱いのルールが混在する可能性）。
- 断定を避け「新人が試した範囲」を明示。

### 2. Deno 2.8 の `deno audit fix` で脆弱依存を自動修正させてみた

#### 概要
Deno 2.8（2026-05）で追加された依存監査コマンド群（`deno audit` / `deno audit fix` / `deno why`）を、npm依存を含む小さなプロジェクトで動かし、脆弱性検出→自動修正→依存経路の説明までを検証する。

#### なぜ今よさそうか
- 「過去最大のマイナー」と公式が言う新機能で、日本語の一次検証記事がまだ薄い＝先行できる。
- `npm audit fix` 相当をランタイム内蔵で完結する体験は実用価値が高い。

#### 新人が検証しやすい理由
- すべてローカルCLIで完結し、認証・課金不要。
- 「検出された脆弱性が fix でどう変わるか」がログで明快に残る。

#### 想定読者
Denoを触り始めた新人、npm audit系の運用に興味がある2年目。

#### タイトル案
1. Deno 2.8 の `deno audit fix` で脆弱な依存を自動修正させてみた
2. `npm audit fix` との違いは？ Deno内蔵の依存監査を新人が試した
3. `deno why` と `deno audit` で依存の脆弱性を追ってみた

#### 検証でやること
- 既知の脆弱バージョンを含むnpm依存を入れる
- `deno audit` で検出、`deno audit fix` で自動修正を確認
- `deno why` で依存経路を可視化

#### 検証手順
1. Deno 2.8+ を導入しバージョン確認
2. 意図的に古い（脆弱な）npmパッケージを追加
3. `deno audit` で結果を記録
4. `deno audit fix` を実行し lock/バージョン差分を確認
5. `deno why <pkg>` で依存経路を出力・スクショ

#### 記事構成案
1. はじめに / 2. なぜ試すのか / 3. 事前に調べたこと（npm auditとの対比） / 4. 環境構築 / 5. 実際に試したこと / 6. 詰まった点 / 7. 分かったこと / 8. npm auditと比べて感じたこと / 9. 向いている人 / 10. まとめ

#### 差別化ポイント
Deno 2.8紹介は英語ニュース中心。実際にログを取り、`npm audit fix` との挙動差を新人目線で具体化する。

#### 参考リンク
- Deno 2.8: https://deno.com/blog/v2.8
- Release v2.8.0: https://github.com/denoland/deno/releases/tag/v2.8.0

#### 注意点
- 脆弱パッケージの選定は再現できる公開パッケージに限定。修正できないケース（制約違反）も記録すると価値が高い。

### 3. ネイティブmasonry `display: grid-lanes` を新人が試して詰まった点

#### 概要
CSSのネイティブmasonryレイアウト `display: grid-lanes` を試し、カードギャラリーを組む。ブラウザ支持が発展途上なため「どこで動いてどこで詰まるか」を含めて検証ログにする。

#### なぜ今よさそうか
- 「ついにネイティブmasonry」とフロント界隈で話題。Safari先行・他ブラウザ実験段階という“今しか書けない過渡期”のログになる。

#### 新人が検証しやすい理由
- 数行のCSSで最小サンプルが作れる。JSフレームワーク不要。

#### 想定読者
CSSレイアウトに興味があるフロント新人、masonryをJSライブラリで妥協してきた人。

#### タイトル案
1. ネイティブmasonry `display: grid-lanes` を新人が試して詰まった点
2. JSなしでmasonry？ `grid-lanes` を触ってみた（対応状況の壁つき）
3. `display: grid-lanes` を Playwright で動かして確認してみた

#### 検証でやること
- `display: grid-lanes` でカード列を組む最小HTML/CSS
- Playwrightで各エンジン（Chromium/WebKit）＋実験フラグでの表示差を確認・スクショ

#### 検証手順
1. カード群の最小HTML/CSSを作成
2. `display: grid-lanes` + `grid-template-columns` + `gap` を適用
3. Playwrightで表示、スクリーンショット取得
4. 未対応エンジンでの挙動（フォールバック）を確認
5. フラグ有効化（`--enable-experimental-web-platform-features`等）で差を記録

#### 記事構成案
1. はじめに / 2. なぜ試すのか / 3. 事前に調べた対応状況 / 4. 環境構築 / 5. 実際に試したこと / 6. 詰まった点（対応ブラウザ） / 7. 分かったこと / 8. JSライブラリと比べて / 9. 向いている場面 / 10. まとめ

#### 差別化ポイント
解説記事は多いが「Playwrightで実際にエンジン差を撮って確認した検証ログ」は少ない。過渡期の生の挙動が価値。

#### 参考リンク
- CSS-Tricks: https://css-tricks.com/masonry-layout-is-now-grid-lanes/
- MDN Masonry: https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_grid_layout/Masonry_layout
- ICS MEDIA: https://ics.media/en/entry/260611/

#### 注意点
- **足切り注意**: Playwright同梱のWebKit/ChromiumがSafari 26.4相当の実装を持つとは限らない。着手前に「Playwrightのブラウザで実際に描画できるか」を最初の30分で確認し、無理ならフラグ運用や「非対応の記録」に方針転換する。

- **トレンド性**: ESLint/Prettier置換（Biome vs oxlint）は国内外で継続的に話題。型認識lintはその論争の核心。
- **新人でも検証できる**: `npm i -D @biomejs/biome` と `biome init` で開始、CLI出力がそのまま素材。

### 最小検証プラン

2. `npm i -D @biomejs/biome` → `npx biome init`
3. 型認識ルールを有効化（公式ドキュメントで最新手順を確認）
4. 型依存バグを仕込み `npx biome check` で検出可否を確認・ログ保存
5. oxlint/ESLintで同じコードを検査し、検出差をスクショで比較
