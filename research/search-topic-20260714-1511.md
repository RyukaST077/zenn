## 上位候補の詳細

### 1. Node 26のTemporalを既定で触り、Dateと書き比べて詰まる

#### 概要
Node.js 26 で `Temporal` がフラグ不要（既定有効）になった。`Temporal.Now` / `PlainDate` / `ZonedDateTime` / `Duration` を使い、`Date` で書いていた「日付の加算」「タイムゾーン変換」「月末計算」を書き比べ、新人が最初に詰まるポイントを検証ログにする。

#### なぜ今よさそうか
- v26 の最大の目玉で各媒体が反復的に特集＝キャッチアップ需要が高い。
- 過去記事は Node 26 の text imports / package maps を扱ったが、Temporal は**未着手**。
- `Date` の落とし穴（可変・暗黙UTC変換・月加算の破綻）は新人読者に刺さる普遍テーマ。

#### 新人が検証しやすい理由
`node --version` で 26 系を確認し、数十行のスクリプトを走らせるだけ。ビルドもサーバも不要で、フラグも不要。

#### 想定読者
- Node 26 で Temporal を試したい新人〜実務2年目
- `Date` の日時計算に苦しんだことがある人

#### タイトル案
1. Node 26でTemporalが既定になったので、Dateと同じ処理を書き比べてみた
2. 新人がNode 26のTemporalを触って、Dateとの違いに詰まった点まとめ
3. 「月末＋1か月」でDateが壊れる問題をNode 26のTemporalで確かめてみた

#### 検証でやること
- `Temporal.Now.plainDateTimeISO()` など各型の生成を確認
- `Date` と `Temporal` で「1か月加算」「TZ跨ぎ」「差分(Duration)」を書き比べ
- ISO 8601 パース・イミュータブル性・`Date`との相互変換で詰まる点を記録

#### 検証手順
1. Node 26 を用意し `node -e "console.log(process.version)"` で確認
2. `Date` 版のスクリプトで月末＋1か月・タイムゾーン変換を書き、挙動をログ
3. 同じ処理を `Temporal`（`PlainDate.add` / `ZonedDateTime` / `Duration`）で書き直す
4. 相互変換（`Temporal.Instant.fromEpochMilliseconds` ⇔ `Date`）と型の違いを確認
5. 詰まった点（メソッド名・不変性・TZ指定必須など）を整理

#### 記事構成案
1. はじめに 2. なぜTemporalを試すのか 3. 事前に調べたこと（既定有効・型一覧）4. 環境（Node 26確認）5. Dateで書いた処理 6. Temporalで書き直し 7. 詰まった点 8. Dateと比べて分かったこと 9. どんな人に向くか 10. まとめ

#### 差別化ポイント
既存の解説記事は「Temporalとは」の紹介が多い。本記事は**Dateとの書き比べと詰まりログ**に絞る。

#### 参考リンク
- Node.js v26.0.0 リリース: https://nodejs.org/en/blog/release/v26.0.0
- InfoQ: Node.js 26 Temporal: https://www.infoq.com/news/2026/07/nodejs-26-temporal/

#### 注意点
- 「Dateは不要」と断定しない（共存が前提）。あくまで新人が試した範囲として書く。

### 2. Deno 2.9の`test.each`と`t.assertSnapshot`・`--changed`を初めて試す

#### 概要
Deno 2.9 でテストランナーが刷新され、`Deno.test.each`（パラメタライズ）、コンテキストの `t.assertSnapshot`（外部import不要のスナップショット）、`deno test --changed`/`--related`（変更起点で関連テストのみ実行）が入った。新人が小さなテストを書いて挙動を確かめる。

#### なぜ今よさそうか
- 2.9 の主要更新で、外部ユーティリティ不要のDX改善。
- 過去記事は Deno 2.9 の task cache / lockfile を扱ったが、**テストランナーは別機能で未着手**。
- ローカルCLIだけで完結し、AI単独実行に最適。

#### 新人が検証しやすい理由
`deno test` だけで動く。スナップショットは `__snapshots__` に自動生成、`-u` で更新できる。

#### 想定読者
- Deno でテストを書き始めた新人
- パラメタライズ/スナップショットを標準機能で使いたい人

#### タイトル案
1. Deno 2.9の新テスト機能（test.each・assertSnapshot・--changed）を試してみた
2. 新人がDeno 2.9でスナップショットテストを外部依存なしで書いてみた
3. `deno test --changed`は本当に関連テストだけ動くのか確かめてみた

#### 検証でやること
- `Deno.test.each` で入力表からテストを生成
- `t.assertSnapshot` で `__snapshots__` 生成と `-u` 更新を確認
- ファイルを変更して `--changed` / `--related` の絞り込みを検証

#### 検証手順
1. Deno 2.9 を用意（`deno --version`）
2. `Deno.test.each` で複数ケースのテストを書く
3. `t.assertSnapshot` を使い、初回生成→意図変更→`-u`更新の流れを確認
4. 依存する関数を変更し `deno test --changed` の実行対象を観察
5. 実行対象数や生成物で「効いたか」を確認（実行時間ではなく回数/生成物で判定）

#### 記事構成案
1. はじめに 2. なぜ試すのか 3. 事前に調べたこと 4. 環境 5. test.each 6. assertSnapshot 7. --changed 8. 詰まった点 9. 既存(node:test等)と比べて 10. まとめ

#### 差別化ポイント
過去のDeno 2.9記事（task cache/lockfile）とは別機能。**テストDXの新規3機能に限定**し、切り口で重複回避。

#### 参考リンク

### 3. 標準APIだけでポスト量子暗号(ML-KEM/ML-DSA)を新人が試してみた

#### 概要
Web Crypto / `node:crypto` に NIST標準のポスト量子アルゴリズム（ML-KEM=鍵カプセル化、ML-DSA=署名）が入り始めた。新人が「鍵生成→カプセル化/デカプセル化」「署名→検証」を短いコードで動かし、既存のRSA/ECDSAと使い勝手を比べる。

#### なぜ今よさそうか
- PQC移行は2026年の大きなテーマで、標準API化が進行中。
- ライブラリ不要・完全ローカルで、AI単独実行に向く。
- 過去記事に暗号テーマは無く、差別化が明確。

#### 新人が検証しやすい理由
鍵生成・署名・検証は各数行。概念は難しいが「動かす」だけなら半日で足りる。

#### 想定読者
- 暗号は未経験だがPQCという言葉を最近よく聞く新人

#### タイトル案
1. 標準APIだけでポスト量子暗号(ML-KEM/ML-DSA)を新人が動かしてみた
2. RSAしか知らない新人がML-KEMの鍵カプセル化を試してみた
3. Node/DenoでML-DSA署名を書いて、ECDSAと比べてみた

#### 検証でやること
- ML-KEM で鍵生成→`encapsulate`/`decapsulate`し共有鍵一致を確認
- ML-DSA で署名→検証（改ざん時にfalseになるか）
- ML-KEM-512/768/1024 で鍵/暗号文サイズを比較

#### 検証手順
1. 対応ランタイムを確認（Node 26系 or Deno 2.9）
2. ML-KEM の鍵生成・カプセル化・デカプセル化で共有鍵の一致を確認
3. ML-DSA の署名・検証、改ざんデータで検証失敗を確認
4. パラメータ違い（512/768/1024）で鍵/署名サイズをログ
5. 既存(RSA/ECDSA)との書き味・サイズ差を整理

#### 記事構成案
1. はじめに 2. なぜPQCを試すのか 3. 事前に調べたこと（NIST標準名）4. 環境 5. ML-KEM 6. ML-DSA 7. 詰まった点（API名/対応バージョン）8. 既存暗号と比べて 9. 向く人 10. まとめ

#### 差別化ポイント
概念解説ではなく、**新人が標準APIで最小コードを動かした検証ログ**に徹する。

## 最初に試すべき1本

### 推奨テーマ
**Node 26のTemporalを既定で触り、Dateと同じ処理を書き比べて詰まる**

### 理由
- **トレンド性**: Node 26 最大の目玉で、直近の公式・二次情報が反復的に取り上げている。
- **新人でも検証できる理由**: フラグ不要・既定有効。数十行のスクリプトだけで、ビルドもサーバも不要。
- **Zenn読者に刺さる理由**: `Date` の落とし穴（月加算・TZ・可変性）は誰もが踏む普遍的な痛み。「書き比べ」は再現しやすく実用的。
- **半日〜1日で検証できるか**: 生成・加算・TZ変換・相互変換の4点に絞れば半日で十分。
- **記事の完成イメージ**: 「Dateで壊れる例 → Temporalで書き直し → 詰まった点」の検証ログ。コード＋実行結果を素直に並べれば成立する。

### 最小検証プラン
1. Node 26 を確認（`node -e "console.log(process.version)"`）
2. `Date` で「月末＋1か月」「TZ跨ぎ変換」を書き、壊れ方をログ
3. 同処理を `Temporal`（`PlainDate.add` / `ZonedDateTime` / `Duration`）で書き直す
4. `Date` ⇔ `Temporal` 相互変換と不変性を確認
5. 詰まった点（メソッド名・TZ必須・型の使い分け）を整理して記事素材化

### 想定タイトル
「Node 26でTemporalが既定になったので、Dateと同じ処理を書き比べてみた」

### 書く前に用意するもの
- Node.js 26 系のローカル環境（`node --version` で確認）
- 検証用の小さな `.mjs`/`.ts` スクリプト一式
- 実行ログ（`Date`版と`Temporal`版の出力の対比）

### 記事化するときの注意
- 専門家として断定しすぎない（「Dateはもう不要」等はNG。共存前提）
- 「新人が試した範囲」を明示する

- 公式情報（nodejs.org v26.0.0）へのリンクを入れる
