## 上位3候補

### 1. `node:vfs`で実ディスクを汚さないテストfixture

- `node:vfs`はNode.js 26.4.0で追加された実験機能で、`--experimental-vfs`が必要である（[Node.js VFS docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。
- defaultのMemoryProviderはMap-backed treeでsymlinkとwatchを支え、`setReadOnly()`後は書き込みが`EROFS`になるため、成功系だけでなく明確な失敗系も作れる（[Node.js VFS docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。
- 実ディスクの一時directoryと同じconfig loader testをVFSへ差し替え、fixture作成、隔離、read-only、watch、API差を短いCLIログで比較できる。

### 2. Package Mapsでphantom dependencyを防ぐ

- Package MapsはNode.js 26.4.0追加の実験機能で、静的JSON tableからbare specifierを解決し、undeclared dependencyへのアクセス防止を用途に挙げる（[Node.js package docs](https://nodejs.org/api/packages.html#package-maps)、閲覧日2026-07-11）。
- 2 packageの最小monorepoで「通常のhoistでは偶然読める」「mapでは宣言なしが失敗」「宣言追加で成功」を観測でき、実務的な物語が明確である。
- module resolutionとpackage IDの説明が必要で、VFSより新人向けの前提知識とfixture量が増える。

### 3. Reporting APIでCSP違反をlocalhost収集

- Reporting APIは2026年3月からBaseline Newly availableで、CSP等のbrowser reportを共通形式で扱う（[MDN](https://developer.mozilla.org/en-US/docs/Web/API/Reporting_API)、閲覧日2026-07-11）。
- localhostのreport endpointと違反を起こすHTMLを用意し、header、POST body、enforce/report-onlyを確認できる。
- report送信はbrowser実装とtimingの影響を受け、既存記事でもbrowser起動失敗が発生しているため、Node CLIだけで完結する1位より停止リスクが高い。

## 選定テーマ

**Node.js 26.4+の実験的`node:vfs`を使い、実ディスクを汚さないconfig loaderのテストfixtureを作って、MemoryProvider・read-only・watch・RealFSProviderの境界を確認する。**

想定タイトル:

> Node.jsに標準VFSが来たので、`tmp`なしのファイルテストを作ってみた

### 選定理由

1. `node:vfs`は2026-06-24公開のNode.js 26.4.0で追加された直近機能である（[Node.js 26.4.0](https://nodejs.org/en/blog/release/v26.4.0)、閲覧日2026-07-11）。
2. `node:fs`に似たsync/callback/promise APIをMemoryProvider上で使え、tests、fixtures、embedded assetsが公式用途に挙げられる（[Node.js VFS docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。
3. Node.js本体と数個の`.mjs`だけで検証でき、クラウド、認証、ブラウザ、物理device、外部実データが不要である。
4. 国内ではリリース概要が出ている一方、fixtureの隔離、read-only化、watch、RealFSProviderとの比較を一続きで実測する余地がある（[Webtech Watcher](https://note.com/webtech_watcher/n/ncafb7148a832)、閲覧日2026-07-11）。
5. Stability 1かつsecurity sandboxではないという重要な制約を、成功例だけでなく失敗例と停止条件を含めて説明できる（[Node.js VFS docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。
6. 既存Node.js記事のTypeScript実行・テスト順ランダム化とは主題が重複しない。

## 半日検証アウトライン

### 0. 環境ゲート（20分）

- 隔離run directoryでNode.js 26.4.0以上を用意し、`node --version`、OS、architectureを記録する。
- `node --experimental-vfs -e "import('node:vfs').then(() => console.log('ok'))"`がexit code 0になることを必須gateにする。
- flagなしでのimportも別processで実行し、実際のerrorとexit codeを記録する。`--experimental-vfs`必須の根拠は[公式docs](https://nodejs.org/api/vfs.html)（閲覧日2026-07-11）とする。

### 1. 比較対象の小さなconfig loaderを作る（35分）

- `readFile`と`stat`を持つ`fsLike`を受け取り、JSON configを読み、必須keyを検証する関数を作る。
- hostの`node:fs/promises`を渡すbaseline testを一時directoryで実行し、成功、missing file、invalid JSONを記録する。
- 実在のuser fileやrepository fileはfixtureにせず、架空データだけを使う。

### 2. MemoryProviderへ差し替える（45分）

- `vfs.create()`で独立した2 instanceを作り、片方に`/config.json`を書いてconfig loaderを通す。
- もう片方から同pathが見えないことを確認し、instanceごとの隔離をassertする。
- sync APIと`vfs.promises`の少なくとも各1操作を実行し、host側にfixture fileが作られていないことを検査する。

### 3. 失敗系とread-onlyを確認する（35分）

- missing file、invalid JSON、directoryをfileとして読むケースを用意し、error codeとmessageを記録する。
- `MemoryProvider`へseedを書いた後に`setReadOnly()`を呼び、readは成功しwriteが`EROFS`になることをassertする。不可逆のread-only化は[公式docs](https://nodejs.org/api/vfs.html)（閲覧日2026-07-11）に従う。

### 4. symlinkとwatchを確認する（45分）

- MemoryProvider内でtarget fileとsymlinkを作り、`readlink`、`realpath`、symlink経由readを確認する。
- `watch`を開始してwrite/renameを行い、eventを上限5秒で収集してwatcherを必ずcloseする。
- event名や回数を一般化せず、実測値と公式の`supportsWatch === true`だけを分けて記録する（[Node.js VFS docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。

### 5. RealFSProviderとの境界を比較する（40分）

- run directory配下に専用rootを1つ作り、`new RealFSProvider(root)`から`/config.json`を書いて、host側の対応fileだけが作られることを確認する。
- `..`やroot外symlinkを使う脱出ケースを試し、拒否されたerrorを記録する。ただし公式docsが明記する通りsecurity sandboxとは表現しない（[Node.js VFS docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。

### 6. テスト比較と証拠整理（50分）

- 同じconfig loader suiteをhost fsとVFSで実行し、pass/fail件数、実行時間、作成file数を最低5回記録する。
- 速度差は小さなfixtureでの参考値とし、性能優位を一般化しない。
- command、stdout、stderr、exit code、fixture tree、Node versionをexecution logへ時系列で残す。

## 完了条件

- Node.js 26.4.0以上で`node:vfs`をflag付きでloadでき、flagなしの挙動も記録できた。
- MemoryProvider上でconfig loaderの成功、missing file、invalid JSONをテストできた。
- 2つのVFS instanceが同pathで状態を共有しないことを確認できた。
- read-only化後のread成功とwriteの`EROFS`を確認できた。
- symlinkまたはwatchの少なくとも一方を公式仕様と実行ログの両方で確認できた。
- RealFSProviderが専用rootへ対応付ける挙動とroot脱出拒否を確認できた。
- host fs比較を含め、すべてローカル・無料・認証不要で完結した。

## リスクと停止条件

- **Node.js 26.4未満しか利用できない、または`--experimental-vfs`でloadできない:** 対象機能を検証できないため停止し、versionとerrorを証拠として残す。別機能へすり替えない。
- **閲覧時docsと実行versionのAPIが一致しない:** 実行versionに対応するversioned docsを確認し、`create()`とMemoryProviderの最小read/writeが不可能なら停止する。実験APIのため差分を成功扱いしない（[公式docs](https://nodejs.org/api/vfs.html)、閲覧日2026-07-11）。
- **watch eventが5秒以内に届かない:** watcherをcloseして当該項目を未確認と記録する。待機を無制限に延ばさず、symlinkとread-onlyの検証を続ける。
- **RealFSProviderのroot外にfileが作られた疑いがある:** 即時停止し、run directory内のfixtureだけを削除対象候補として記録する。repositoryや外部pathは変更しない。
- **root脱出入力が拒否されない:** security issueの可能性として停止し、追加の攻撃的試行は行わない。VFSをsandboxと説明しない。
- **実ディスク比較で速度差が安定しない:** 性能結論を捨て、API差・隔離性・失敗系だけを記事候補の根拠にする。
- **秘密情報、資格情報、private hostname、外部実データが必要になる:** 即時停止する。

## 結論

最初に試すテーマは、**Node.js 26.4+の`node:vfs`でconfig loaderのテストfixtureをMemoryProviderへ置き、実ディスク版との違い、隔離、read-only、symlink/watch、RealFSProviderの境界を確認する検証**とする。2026-06-24に追加された直近の標準実験機能で、追加package・認証・GUIなしに成功系と失敗系を短いCLIログへ残せる。既存記事との主題重複もなく、公式が警告する「sandboxではない」という境界まで扱えるため、新人向けの実践記事として価値が高い。
