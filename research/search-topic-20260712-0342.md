## 選定テーマ

**npm v12でinstall scriptが明示許可制になった挙動を、自作の無害なdependencyで確認し、`approve-scripts`によるversion固定承認、更新時の再審査、`npm ci`での再現まで検証する。**

想定タイトル:

> npm v12でpostinstallが動かない？ allowScriptsの承認・更新・CIを安全なfixtureで試した

選定理由:

- npm v12は2026-07-08に一般提供され`latest`になったばかりで、調査時点の適時性が最も高い（[GitHub公式](https://github.blog/changelog/2026-07-08-npm-install-time-security-and-gat-bypass2fa-deprecation/)、閲覧: 2026-07-12）。
- 公式が`allowScripts`の既定off、暗黙の`node-gyp rebuild`も含む対象範囲、`approve-scripts` / `deny-scripts`による移行を明示しており、観測可能な問いが具体的である（[予告記事](https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/)、閲覧: 2026-07-12）。
- 国内Zennでは防御策の解説が既にあり、関心が確認できる一方、安全なlocal fixtureを使った承認前後・version更新・CI再現の一続きの実測に差別化余地がある（[国内Zenn記事](https://zenn.dev/vtf/articles/npm-v12-supply-chain-defense)、閲覧: 2026-07-12）。
- npm CLI、Node.js、自作local dependencyだけで実行でき、account、token、cloud、GUI、課金を必要としない。
- 既存記事と過去選定テーマのいずれとも主題が重ならず、新人が「installが成功したのに必要な生成物がない」状態を避ける実務的な記事にできる。

## 半日検証アウトライン

所要目安は4〜6時間。repository本体ではなく新しい隔離run directoryを使い、全command、stdout / stderr、exit code、fixture hash、生成file一覧を時系列で記録する。

### 0. 環境とversion gate（25分）

- OS、architecture、`node --version`、既定の`npm --version`を記録する。環境変数全量は出力しない。
- package runnerまたは隔離prefixでnpm 11.18.xとnpm 12.0.xをexact versionで用意し、各CLIの`--version`を記録する。repositoryやglobal npmは変更しない。
- npm 12が取得不能、または`npm@12`が`latest`であることを公式発表とCLIの両方で確認できない場合は停止する（[GA発表](https://github.blog/changelog/2026-07-08-npm-install-time-security-and-gat-bypass2fa-deprecation/)、閲覧: 2026-07-12）。

### 1. 無害なfixtureを作る（40分）

- app packageと、local file dependency `fixture-installer`を作る。
- dependencyの`postinstall`はrun directory内に固定文字列のmarker fileを1個作るだけにし、外部通信、shell download、native build、環境変数参照を禁止する。
- dependencyのruntime exportはmarkerの有無を返すだけにし、install終了コードと実際の生成物を別々に判定できるようにする。
- source一覧とhashを保存し、npm 11 / 12で同一fixtureを使う。

### 2. npm 11と12のbaseline比較（50分）

- 各versionでclean cache / clean `node_modules`からinstallし、終了コード、warning、marker生成、lockfile差分、runtime testを記録する。
- npm 11ではwarning preview、npm 12ではdefault-denyが期待されるが、結果を先取りせず出力とmarkerだけで判定する（[npm v12予告](https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/)、閲覧: 2026-07-12）。
- installが0終了でもmarkerがなければ「script実行成功」と扱わない。

### 3. pending確認・承認・deny（60分）

- `npm approve-scripts --allow-scripts-pending`のhuman-readable出力と、利用可能ならJSON出力を保存する。
- `fixture-installer`だけを承認し、`package.json`の`allowScripts`差分がname-onlyかversion固定かを確認する。version固定が既定であることは[npm公式docs](https://docs.npmjs.com/cli/v11/commands/npm-approve-scripts/)（閲覧: 2026-07-12）を根拠にする。
- clean install後にmarkerとruntime testを再確認する。
- fixtureを複製した別packageを`deny-scripts`で明示denyし、blanket approvalの対象にならないことを確認する。

### 4. version更新時の再審査（55分）

- local dependencyのversionを`1.0.0`から`1.0.1`へ変更し、marker内容もversion付き固定文字列へ変える。
- lockfileを更新してclean installし、旧versionの固定承認が新versionへ自動継承されるか、pendingへ戻るかを記録する。
- 新versionを再承認し、`allowScripts`差分、marker、runtime testを保存する。
- name-only承認は比較用copyで1回だけ試し、将来versionまで許すtrade-offを観測結果として整理する。

### 5. `npm ci`と再現性（45分）

- 承認済み`package.json`とlockfileをclean copyへ移し、npm 12で`npm ci`を実行する。
- exit code、marker、runtime testがinstall時と一致するか確認する。
- `allowScripts`だけを外したcopyでも`npm ci`を実行し、終了コードとmarkerを確認する。CIがgreenでも生成物不足になり得るかは実測結果からのみ結論する。

### 6. 証拠整理（40分）

- npm 11 / 12、未許可 / version固定承認 / name-only承認 / deny / version更新 / `npm ci`を、終了コード、warning、marker、runtime test、policy差分の表にする。
- 実在package一般へ結果を拡張せず、「自作local dependencyと使用したexact npm versionで確認した範囲」と明記する。
- Git / remote dependencyの既定拒否、native addon、global install、`npx`は公式仕様への補足に留め、未実行なら未検証と明記する。

## 完了条件

- npm 11系と12系のexact version、同一fixture、全commandのexit codeを記録できた。
- 未許可時のinstall結果とmarker有無を分けて判定できた。
- pending一覧、version固定承認後の`package.json`差分、clean reinstall後のmarkerとruntime testを記録できた。
- dependency version更新時に旧承認がどう扱われるかを確認できた。
- explicit denyまたはname-only承認の少なくとも一方をversion固定承認と比較できた。
- npm 12の`npm ci`でpolicyと生成物の再現性を確認できた。
- 全工程がローカル・無料・認証不要で完結し、run directory外への書き込みと秘密情報の記録がない。

## リスクと停止条件

- **npm 12取得失敗:** exact versionを1回だけ再確認する。registry接続またはpackage取得が再試行後も失敗したら停止し、npm 11のwarningからnpm 12の結果を推測しない。
- **CLI version不一致:** npm 11 / 12の実行commandが同じbinaryを解決している、または期待majorを返さない場合は比較を停止する。global installやrepositoryのpackage manager変更で回避しない。
- **fixtureのscope逸脱:** scriptがrun directory外へ書く、networkへ接続する、credentialや環境変数を読む場合は即時停止する。第三者packageのinstall scriptを承認して代替しない。
- **local file dependencyがpolicy対象外:** 公式は非registry依存の`prepare`も対象とするが、fixture形態によって期待したpending表示にならない場合がある（[GitHub公式予告](https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/)、閲覧: 2026-07-12）。tarball化またはworkspace外のlocal packageへ一度だけ変更し、それでも対象にならなければ停止して差異を記録する。
- **install成功とscript成功の混同:** exit code 0だけでは成功と判定せず、marker内容とruntime testを必須にする。markerがない場合は未実行または失敗として区別し、原因不明なら断定しない。
- **承認粒度を確認できない:** `package.json`に期待したversion固定entryが書かれない場合、CLI helpと公式docsを照合して一度だけ再実行する。解消しなければname-only / pinnedの比較を打ち切る。
- **lockfile差やcache混入:** 各条件で専用directoryとcacheを使い、source hashが異なる、または古いmarkerが残る場合はそのrunを無効にして一度だけclean再実行する。
- **時間超過:** 6時間を超える場合、name-only承認とexplicit denyの追加比較を打ち切る。baseline、version固定承認、version更新、`npm ci`を優先する。
- **秘密情報:** token、cookie、private registry URL、private hostname、usernameを含む絶対path、環境変数全量をartifactへ保存しない。混入した出力は保存前に伏せる。

## 結論

選定テーマは、**npm v12の`allowScripts`を自作の無害なinstall-script dependencyで検証し、未許可、version固定承認、dependency更新時の再審査、`npm ci`での再現を比較するハンズオン**とする。2026-07-08一般提供という強い時宜があり、国内外で供給網防御と移行影響への関心が確認できる。既存記事や過去選定テーマと重複せず、認証・GUI・課金なしで、終了コードだけでは見落とす生成物不足まで一次ログとして示せるため、新人向けの半日記事として最も適している。
