## 上位3候補

### 1位: Chrome 150 `focusgroup`をJS roving tabindex版と比較（29点）

Stable公開直後の時宜があり、keyboard eventと`tabindex`を手書きしたbaselineを、同一toolbarで`focusgroup`版へ置換できる。成功可否を`document.activeElement`、DOM attribute、keyboard sequence、accessibility snapshotで機械的に残せる。Open UIがfocusとselectionの責務を分けているため、「HTML属性だけでtab UIが全部完成する」という誤解も実測で防げる（[Open UI Explainer](https://open-ui.org/components/focusgroup.explainer/)、閲覧2026-07-13）。

### 2位: `background-clip: border-area`比較（27点）

Chrome 150でgradient borderを`border-image`なしに作れるため、見た目のbefore/afterを短時間で示せる（[New in Chrome 150, 2026-06-30](https://developer.chrome.com/blog/new-in-chrome-150)、閲覧2026-07-13）。ただし既存のCSS系記事が多く、選定テーマよりrepository内差別化が弱い。

### 3位: image値版`light-dark()`比較（26点）

`color-scheme`に応じて`url()`等のimage値を切り替える新挙動を、従来の`prefers-color-scheme` media queryと比較できる（[Chrome 150 beta, 2026-06-03](https://developer.chrome.com/blog/chrome-150-beta)、閲覧2026-07-13）。手軽だが、focusgroupほど新人が学べる状態管理・アクセシビリティ境界が広くない。

## 選定テーマ

**Chrome 150の`focusgroup`でtoolbarのroving tabindexを置き換え、手書きJavaScript版と同一fixtureで、Tab/矢印/Home/End、wrap、last-focused memory、RTL、disabled/hidden item、focusとselectionの責務、accessibility tree、非対応時fallbackをPlaywrightで対照検証する。**

想定タイトル:

> Chrome 150の`focusgroup`を試す: roving tabindexのJavaScriptはどこまで消せる？

選定理由:

1. Chrome 150 Stableの主要機能として2026-06-30に公開された直後である（[Chrome for Developers](https://developer.chrome.com/blog/new-in-chrome-150)、閲覧2026-07-13）。
2. 国内ではStable化と実装コスト低減が紹介され、国際コミュニティではrole exposureやedge caseの検証が呼びかけられているため、紹介記事ではなく再現可能な対照ログに余地がある（[ミツエーリンクス, 2026-07-01](https://www.mitsue.co.jp/knowledge/blog/qc/202607/01_1326.html)、[Adrian Roselli, 2026-07-05](https://adrianroselli.com/2026/07/focusgroup-tests.html)、いずれも閲覧2026-07-13）。
3. localhostの静的fixtureとPlaywrightだけで、課金・認証・外部APIなしに実行できる。run段階ではbrowser launchとfeature detectionをgateにし、未達なら結果を推測しない。
4. 既存記事にfocus management / `focusgroup`の主題がなく、CSS描画記事やtest tool記事とも差別化できる。

## 半日検証アウトライン

### 0. capability gate（20分）

1. OS、Node、Playwright、browser executable/versionを記録する。
2. Chromiumをheadlessで起動し、major versionが150以上か確認する。
3. localhostの最小pageで`'focusgroup' in HTMLElement.prototype`を評価する。HTML attributeを置いただけの`hasAttribute()`はsupport判定に使わない（feature detectionの背景: [Jeremy Keith, 2026-03-09](https://adactio.com/journal/22445)、閲覧2026-07-13）。
4. browser launch失敗、major 150未満、property未実装のいずれかなら停止し、UI結果を推測しない。

### 1. 手書きJS baseline（45分）

1. 4 buttonのtoolbarを作り、1要素だけ`tabindex="0"`、残りを`-1`にする。
2. `keydown`でArrowLeft/Right、Home/End、wrap、disabled/hidden skipを実装する。
3. キー列ごとの`document.activeElement.id`と全buttonの`tabindex`を記録する。
4. JS line数と分岐数を記録し、後の比較baselineにする。

### 2. `focusgroup`版へ置換（60分）

1. 同一markupを`focusgroup="toolbar wrap"`へ置換し、focus移動用JSを削除する。
2. Tabがgroupに1回だけ入り、ArrowLeft/Right、Home/End、端のwrapが期待どおりか記録する。
3. group外へTab移動して戻り、last-focused memoryを確認する。
4. `nomemory`と`focusgroupstart`を追加したfixtureで再入場先を比較する。これらのmodifierはChromeのAPI overviewに記載されている（[Chrome for Developers, 2026-03-05](https://developer.chrome.com/blog/focusgroup-rfc)、閲覧2026-07-13）。

### 3. 境界条件matrix（70分）

1. `dir="rtl"`でlogical inline directionに追従するか確認する。
2. disabled、`hidden`、動的追加・削除itemを含むtoolbarでfocus順を記録する。
3. `focusgroup="none"`区間のArrow移動とTab移動を分けて観測する。
4. tablist fixtureを1つ作り、focus移動だけでは`aria-selected`とpanel表示が変わらないこと、author codeがselection更新に残ることを確認する（責務境界: [Open UI Explainer](https://open-ui.org/components/focusgroup.explainer/)、閲覧2026-07-13）。

### 4. accessibilityとfallback（45分）

1. Playwrightのaccessibility snapshotまたは利用可能な同等APIでtoolbar/tablist roleとchild roleを記録する。
2. support判定をfalseにしたfallback branchで手書きroving tabindexが動くことを確認する。
3. browser APIでaccessibility treeを取得できない場合はDOM/keyboard観測と混同せず、その項目だけ未検証として記録する。

### 5. 証拠整理（40分）

1. JS版と`focusgroup`版のkeyboard matrix、DOM状態、accessibility tree、残るselection codeを表にする。
2. 成功・失敗の終了code、console、browser versionをexecution logへ残す。
3. スクリーンショットはfocus ringの補助証拠とし、focus成否の主証拠は`activeElement`とaccessibility treeにする。

## 完了条件

- browser launch、Chrome/Chromium major 150以上、`focusgroup` property supportの3 gateを通過している。
- JS baselineと`focusgroup`版を同一fixture・同一keyboard sequenceで比較できている。
- Tab、矢印、Home/End、wrap、memory、RTL、disabled/hiddenの観測値が保存されている。
- focus navigationとselection stateの責務を混同せず、tablistで残るauthor codeを示している。
- 非対応fallback、または実行不能時の停止記録がある。

## リスクと停止条件

1. **browser launch失敗**: executable不足、OS制約、Playwright errorの原文と終了codeを保存して即停止する。レンダリング、keyboard挙動、accessibility結果を推測しない。
2. **browser major不足**: 150未満なら停止する。flagで成功した結果をStable挙動として扱わない。
3. **feature gate失敗**: `'focusgroup' in HTMLElement.prototype`がfalseなら停止する。attributeがDOMに残るだけでは対応と判定しない。
4. **accessibility API不足**: keyboard/DOM検証は継続可能だが、accessibility treeは未検証と明記し、roleを推測しない。
5. **標準化途上**: Open UI explainerはnarrowed proposalとの関係や将来検討を明記しているため、Chrome 150の観測値を全browserの確定仕様と一般化しない（[Open UI Explainer](https://open-ui.org/components/focusgroup.explainer/)、閲覧2026-07-13）。
6. **focusとselectionの混同**: `focusgroup`がtab activationや`aria-selected`更新まで代行したと断定しない。Open UIの責務分離に照合する。
7. **時間超過**: 4時間時点で基本toolbar matrixが未完なら、tablistと`focusgroup="none"`は未実施として切り、基本比較の証拠整理を優先する。

## 結論

選定テーマは、**Chrome 150の`focusgroup`を手書きroving tabindex版と同一fixtureで比較し、keyboard navigation、memory、RTL、disabled/hidden、focusとselectionの境界、accessibility tree、fallbackをPlaywrightで記録するハンズオン**とする。Stable公開直後の適時性、国内外のアクセシビリティ関心、既存記事との差別化があり、無料・認証不要・localhost完結で実施できる。run段階では`danger-full-access`により実ブラウザを利用できるが、browser launch、major 150、property supportを必須gateとし、失敗時は挙動を一切推測せず停止する。
