# Zenn 記事 修正適用（レビュー指摘 → 記事修正）

> このSkillはユーザーが `/revise-article` で明示的に起動する前提。自動起動はしない。
> **非対話（headless）実行を前提とする**（例: `claude -p "/revise-article ..."`）。
> 途中でユーザーに確認・質問しない。`AskUserQuestion` は使わず、情報が不足する場合は
> 常に下記のデフォルト前提を採用してそのまま最後まで進める。
> 詳細な参照データは `reference/` に分離してある。各ステップで必要になったら該当ファイルを読む。

## パイプライン上の位置づけ

このSkillは**レビューループの修正担当**にあたる。`review-article`（判定）と対になり、
判定が「公開可」になるまで交互に回す（ループ回数の上限管理はオーケストレーター側）。

```
/draft-article  → articles/<slug>.md (published:false)
/review-article → logs/review-<slug>-*.md（判定: 公開可 / 要修正 / 公開不可）
/revise-article → articles/<slug>.md を修正 + logs/revise-<slug>-*.md（修正レポート）  ← これ
（→ /review-article で再レビュー。公開可になったら /publish-pr へ）
```

## Goal

`review-article` のレビューレポートにある **blocker / warning を、「具体的な直し方」に従って
記事ドラフトに適用し、再レビューで「公開可」になり得る状態を作る**。

修正の大原則は `draft-article` と同じ「**捏造しない**」。事実の補完（`要素材` を埋める、
数値の裏付けを足す、スクショを足す）は、出典ログ（`execution-log.md`）と
その `workspace/` / `screenshots/` にある**一次情報だけ**で行う。ログに裏付けが無い記述は、
書き足すのではなく**弱める・削る**方向で直す（節ごと削ってよい）。

最終成果物:

* 修正を適用した `articles/<slug>.md`（**published: false のまま**。slug リネーム時は新パス）
* 修正レポート `logs/revise-<slug>-<日時>.md`（指摘ごとの 適用 / スキップ＋理由 の全記録）
* セルフチェック結果（`check-article.sh` 再実行の出力）
* 標準出力の最終行に機械可読の結果行 `RESULT: ok <記事パス>` / `RESULT: abort <理由>`

## 起動方法と引数

`/revise-article` で起動。以下を引数（自然言語）で渡せる。すべて省略可。

* 対象記事: `articles/<slug>.md` のパス（無指定なら最新の `published: false` ドラフト）
* レビューレポート: `logs/review-<slug>-*.md` のパス（無指定なら対象記事に対応する最新）
* 出典ログ: `logs/run-*/execution-log.md` のパス（無指定ならレポート/記事から辿る）
* 適用範囲（例: blockerだけ / suggestionも）。**既定は blocker ＋ warning**
* slug リネームの可否（例: リネーム禁止）。**既定は指摘があればリネーム可**

### 引数が無い/不足のときのデフォルト前提

* 対象記事: `articles/` の中で `published: false` かつ最も新しく更新されたドラフト
* レビューレポート: `ls -t logs/review-<slug>-*.md | head -1`（対象記事の slug で絞る）
* 出典ログ: レビューレポートの「レビューの前提」→ 記事冒頭コメント `<!-- 前提: ... -->`
  → `ls -t logs/run-*/execution-log.md | head -1` の順で辿る。どれも無ければ「出典不明」
  として扱う（事実の**補完**はできないので、削減修正のみで対応する）
* 適用範囲: blocker ＋ warning（suggestion は安全で機械的なものだけ任意に適用してよい）

不足情報があっても止まらず、採用した前提を修正レポート冒頭に明示したうえで進める。
ユーザーへの追加質問は行わない（非対話実行のため応答を待てない）。

## Assumptions

* 入力は `review-article` のレビューレポートと、その対象記事ドラフト。
* 本Skillは**修正の適用だけ**を担当する。公開可否の再判定はしない（`/review-article` の役割）。
  新しい実行・検証もしない（`/run-practice` の役割）。記事のゼロからの執筆もしない
  （`/draft-article` の役割）。
* 事実性の正は出典ログ。補完素材はログ・`workspace/`・`screenshots/` からのみ取る。
* レビューの指摘は「箇所＋何を何に直すか＋根拠」の粒度で書かれている前提。曖昧な指摘は
  `reference/fix-policies.md` の分類に照らして安全側（削減方向）で解釈する。

## 修正の安全ルール（重要 / 必須）

* **published: false を維持する**。`published: true` には絶対にしない（公開ゲートは
  `/publish-pr` のPRマージ）。逆に true になっていたら false に戻す（これは blocker 修正）。
* **捏造しない**。ログに無い成功・数値・コード・画像を書き足さない。裏付けが無い記述の
  指摘は「ログ由来の事実に置き換える」か「弱める・削る」で直す。
* **非破壊の記録**。適用した変更・スキップした指摘とその理由を、すべて修正レポートに残す。
* **意味を保つ最小修正**。指摘に関係ない箇所は書き換えない。文体の全面リライトをしない。
* **slug リネームは一式で行う**。記事ファイル・`images/<slug>/` ディレクトリ・本文の
  `/images/<slug>/...` 参照・冒頭の前提コメントを同時に更新し、旧ファイルを残さない。
* **秘密情報は削除・マスクする**（`sk-***` 等に置換、個人パスは `~/` に一般化）。
  対象がコミット済みの場合、ファイル修正だけでは git 履歴に残る。その旨を修正レポートに
  警告として明記する（履歴の書き換えはしない）。

## Input Resolution（入力の特定）

必ず最初に「記事・レビューレポート・出典ログ」の組を確定させてから修正に入る。

1. **記事特定**: 引数にパスがあればそれを使う。無ければ `published: false` のドラフトのうち
   最も新しく更新されたものを選ぶ（`review-article` と同じ選び方）。
2. **レポート特定**: 引数にパスがあればそれを使う。無ければ
   `ls -t logs/review-<slug>-*.md 2>/dev/null | head -1`（`<slug>` は記事ファイル名）。
3. **判定確認**: レポートの「判定」を読む。**公開可**（blocker 0・warning 0）なら修正は不要。
   「修正不要（判定: 公開可）」と表示し、`RESULT: ok <記事パス>` を出して**そこで終了する**。
4. **出典ログ特定**: 上記デフォルト前提の順で辿る。
5. **フォールバック（中止条件）**: 次の場合は理由を標準出力に表示し、最終行に
   `RESULT: abort <理由>` を出して**そこで終了する**（記事を書き換えない・修正を捏造しない）。
   * 対象ドラフトが無い →「先に `/draft-article` を実行してください」
   * レビューレポートが無い →「先に `/review-article` を実行してください」

## Workflow

### Step 1: 入力を確定する
上記「Input Resolution」に従い、記事・レビューレポート・出典ログを確定する。採用した前提
（各パス／適用範囲／slugリネーム可否）を控えておく。

### Step 2: 指摘を棚卸しして修正方針を割り当てる
レポートの「指摘一覧（重大度順）」から blocker / warning（指定があれば suggestion も）を
すべて抽出し、各指摘に `reference/fix-policies.md` の修正分類を割り当てる。

* **A 機械修正**（published差し戻し・フェンス閉じ・Front Matter・壊れたパス修正 等）
* **B ログ由来の補完**（`要素材` をログの記録で埋める・実在スクショの追加参照 等）
* **C 削減修正**（裏付けの無い断定を弱める／ログに無い記述・素材の無い節を削る 等）
* **D slug リネーム**（ファイル・画像ディレクトリ・参照の一式更新）
* **E 匿名化・マスク**（個人パスの `~/` 化・トークンのマスク 等）
* **修正不能**（どの分類でも解消できない）→ Step 6 の中止判定へ

過去の修正レポート（`logs/revise-<slug>-*.md`）があれば確認し、**前回も「修正不能」だった
指摘が再び来ていないか**を見る（来ていたらループしているので Step 6 で中止に倒す）。

### Step 3: 補完素材を出典ログから収集する（分類Bがある場合）
出典ログの「タスク実行ログ」「詰まった点」「スクリーンショット一覧」「記事への写像」と
`workspace/` / `screenshots/` から、指摘箇所に対応する一次情報を探す。

* 見つかった → その記録（コマンド・エラー全文・実測値・コード抜粋・スクショ）で埋める。
  スクショは `images/<slug>/` にコピーしてから `/images/<slug>/...` で参照する。
* 見つからない → **分類Cに切り替える**（その記述・節を削るか、書ける範囲まで弱める）。
  埋められなかったことを修正レポートに記録する。出典ログが「出典不明」の場合も同様。

### Step 4: 修正を適用する
分類ごとに `reference/fix-policies.md` の手順で記事を編集する。指摘の「具体的な直し方」を
第一に従い、方針が矛盾する場合は安全側（公開安全 > 事実性 > 体裁）を優先する。
suggestion は安全で機械的なもの（誤字・alt追記など）だけ適用し、任意である旨を記録する。

### Step 5: セルフチェックを行う
`bash .claude/skills/review-article/scripts/check-article.sh <記事パス>` を再実行し、
機械検出できる指摘（published / slug / 画像 / フェンス / 要素材 / 秘密情報）が解消したことを
確認する。残っていれば Step 4 に戻って直す（このループは2回まで。解消できなければ
「修正不能」として扱う）。`published: false` のままであることを必ず最終確認する。
※ これは自己確認であり、公開可否の判定ではない（判定は `/review-article` の再実行で行う）。

### Step 6: 修正不能の扱いを確定する（中止判定）
「修正不能」が残った場合の扱い:

* 修正不能が **warning のみ** → 適用済みの修正は保存し、修正レポートに「未解消の指摘」として
  明記して続行する（再レビューの判断に委ねる）。
* 修正不能に **blocker が含まれる**、または**記事の核となる主張がログで裏付けられず、削ると
  記事が成立しない** → 適用済みの安全な修正（published差し戻し・秘密情報マスク等）だけ保存し、
  理由と推奨アクション（例:「/run-practice からやり直して素材を取り直す」）を表示して
  `RESULT: abort <理由>` で終了する。
* **前回の修正レポートでも修正不能だった同じ指摘が再来している** → ループ防止のため
  `RESULT: abort 同一指摘が解消できずループしている: <指摘>` で終了する。

### Step 7: 修正レポートを保存して結果を出力する
`reference/fix-policies.md` の Report Format に沿って修正レポートを構成し、
後述の Output に従って**必ずファイルとして保存する**。

## Output

**必ずファイルとして保存する**（非対話実行では会話ログが残らないため）。

1. `date +%Y%m%d-%H%M` で日時を取得する
2. リポジトリ直下に `logs/` が無ければ作成する
3. 修正レポートを `logs/revise-<slug>-<日時>.md` に書き込む（`<slug>` は**修正後**の
   記事ファイル名。slug リネーム時は新旧の対応をレポートに明記する）
4. 最後に、以下を標準出力にも短く表示する:
   * 修正後の記事パス（リネーム時は 旧 → 新 を明記）と `published: false` である旨
   * 適用した修正の件数（blocker / warning / suggestion 別）とスキップ・未解消の件数＋理由
   * セルフチェック結果の要約（`SUMMARY fail=.. warn=..` 行）
   * 修正レポートのパス
   * 次のアクション: 「`/review-article <記事パス>` で再レビューし、公開可になったら
     `/publish-pr` へ」
5. **標準出力の最終行**に、オーケストレーター向けの機械可読な結果行を出す:
   * 修正を適用して続行可 → `RESULT: ok <修正後の記事パス>`
   * 中止（Step 6 / フォールバック） → `RESULT: abort <理由の一文>`

   slug をリネームした場合、`RESULT: ok` のパスは**必ず新パス**にする
   （呼び出し側が以降の工程で使う）。

## Quality Rules

* 修正は指摘に基づく最小限にする。指摘の無い箇所・記事の主旨・文体は変えない。
* 事実の補完は出典ログ・workspace・screenshots の一次情報のみ。裏付けが無ければ削る（捏造しない）。
* **published: false を維持する**（true 化は /publish-pr の担当。逆方向の差し戻しはする）。
* すべての変更とスキップ理由を修正レポートに記録する（何をなぜ直した/直せなかったかを追跡可能に）。
* slug リネームは記事・画像ディレクトリ・本文参照・前提コメントを一式で行い、旧ファイルを残さない。
* セルフチェック（check-article.sh）で機械指摘の解消を確認してから保存する。
* 公開可否の判定はしない。判定・再判定は `/review-article` に委ねる。
* 採用した前提（記事・レポート・出典ログ・適用範囲）を修正レポート冒頭に明示する。
* 最終行の `RESULT:` 行を必ず出す（ok は修正後パス付き、abort は理由付き）。

## Anti-Patterns

* 指摘を解消するためにログに無い成功・数値・コード・画像を書き足す（捏造で「直す」）。
* `published: true` にして保存する／true の差し戻しを忘れる。
* 記事を全面リライトして draft-article の成果を壊す（最小修正の原則を破る）。
* slug だけリネームして images/ や本文参照を旧slugのまま残す（画像が全滅する）。
* 修正不能な blocker を黙って放置し `RESULT: ok` を返す（ループの原因になる）。
* 同じ指摘に対して何度も同じ修正を試みてループする（前回レポートを確認しない）。
* 修正レポートを残さず記事だけ書き換える（何を直したか追跡できなくなる）。
* 自分で「公開可」と判定して /publish-pr 相当の操作まで進める（役割の越境）。
