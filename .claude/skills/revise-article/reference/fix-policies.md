# 修正ポリシー（指摘の分類と直し方）

`revise-article` の Step 2〜Step 4 で使う。レビュー指摘を修正分類（A〜E / 修正不能）に
割り当て、分類ごとの手順で直す。迷ったら安全側（公開安全 > 事実性 > 体裁）に倒し、
事実は**足すより削る**方向で直す。

## 指摘カテゴリ → 修正分類の対応表

`review-article` のチェック観点（review-checklist.md の7次元）との対応。

| 指摘の典型例（レビューの次元） | 分類 | 直し方の要点 |
|---|---|---|
| `published: true` になっている（公開安全） | A | `false` に差し戻す |
| 秘密情報・APIキー・トークン（公開安全） | E | 削除またはマスク（`sk-***`） |
| 個人パス `/Users/<名前>/...`・内部ホスト名（公開安全） | E | `~/` や `example.internal` に一般化 |
| slug 文字種/長さ違反・汎用slug・ローカル重複（公開安全） | D | slug リネーム一式（下記手順） |
| Front Matter 欠落・type不正・topics超過（Front Matter） | A | ログ/記事内容から妥当な値を補う |
| title 誇大表現・60字超（Front Matter） | A/C | 経験談トーンに弱める・短くする |
| 記事の結論がログの結果サマリーと不一致（事実性） | C | ログの結論（達成/一部/未達）に合わせて書き直す |
| ログに無いコマンド・エラー・コードの創作（事実性） | B/C | ログ/workspaceの実物に差し替え。無ければ削る |
| 裏付けの無い数値・所要時間（事実性） | B/C | ログの実測値に差し替え。無ければ数値を消して定性表現に |
| ログに無い成功の断定（「速い」「簡単」等）（事実性） | C | 経験談トーンに弱める（下記の文例） |
| `要素材` マーカー残存（事実性/完成度） | B/C | ログから埋める。無ければ**節ごと削る** |
| 画像参照が実在しない（画像） | A/B | パスを直す or 実在スクショに差し替え。無ければ参照を削る |
| 詰まった点にスクショが無い（画像） | B | ログの screenshots/ からコピーして参照追加 |
| 未参照の孤立画像（画像） | A | 本文で使うか `images/<slug>/` から削除 |
| コードフェンス・`:::` 未閉じ（Markdown構造） | A | 閉じる位置を特定して補う |
| 見出し階層の破綻・空リンク・example.com（Markdown構造） | A | 階層修正・実在リンクに差し替え or リンク削除 |
| 詰まった点が書かれていない（トーン） | B/C | ログの「詰まった点と解決過程」から追記。無ければ「詰まらなかった」と正直に書く |
| 再現性（OS/バージョン）欠落（トーン） | B | ログの「再現性メモ」から転記 |
| 誤字・表記ゆれ・alt欠落（suggestion 系） | A | 機械的に直す（任意適用） |
| プレースホルダ（TODO/`<slug>`等）残存（完成度） | A/C | 正しい値に置換 or 該当行を削除 |

## 分類別の手順

### A: 機械修正
記事の意味を変えない、正解が一意に決まる修正。そのまま適用する。
値の補完（topics 等）が必要な場合は、出典ログ・記事本文から読み取れる範囲で決める
（推測で新しい事実は足さない）。

### B: ログ由来の補完
**素材の出所は 出典ログ本文 / `workspace/` / `screenshots/` の3つだけ。**

1. 指摘箇所に対応する記録を出典ログから探す（「記事への写像」→「タスク実行ログ」→
   「詰まった点」→「再現性メモ」の順で当たりをつける）。
2. 見つかったら、その**原文**（コマンド・エラー全文・実測値・コード抜粋）を使って埋める。
   要約しすぎない。コードは workspace の実物から抜粋する。
3. スクショを足す場合は `logs/run-*/screenshots/` の実物を `images/<slug>/` にコピーし、
   `/images/<slug>/NN-*.png` で参照する（存在しない画像を参照しない）。
4. **見つからなければ分類Cへ切り替える。** それらしい内容を書いて埋めるのは捏造。

### C: 削減修正（弱める・削る）
裏付けの無い記述は、次の優先順で「事実の範囲まで縮める」。

1. **差し替え**: ログにある事実で言い換えられるなら言い換える。
2. **弱める**: 断定 → 経験談の範囲に限定する。文例:
   * 「〜は高速だ」→「今回の検証（<実測値がログにある場合のみ数値>）では待たされる感じは
     なかった」／実測が無ければ「速さの検証まではできていない」
   * 「〜すれば動く」→「筆者の環境（<ログの再現性メモ>）ではこの手順で動いた」
   * 「〜が原因だ」→「ログを見る限り〜が原因のようだった（未確証）」
3. **削る**: 言い換えも弱めもできない（素材が皆無の）段落・節は、見出しごと削除する。
   削除で前後の流れが切れる場合は接続の1文だけ整える（新しい事実は足さない）。
4. 削った内容と理由は必ず修正レポートに記録する。

### D: slug リネーム一式
slug は記事ファイル名そのもの。**必ず次を1セットで行う**（漏れると画像参照が全滅する）。

1. 新 slug を決める: 技術名＋切り口で具体化。`a-z0-9_-` のみ・12〜50字。
   汎用語始まり（getting-started 等）を避け、`ls articles/` でローカル重複が無いことを確認
   （knowledge/2026-07-01-zenn-slug-already-used.md 参照）。
2. `git mv articles/<旧>.md articles/<新>.md`（git 管理外なら `mv`）。
3. `images/<旧>/` があれば `images/<新>/` にリネームする。
4. 本文の `/images/<旧>/...` 参照をすべて `/images/<新>/...` に置換する。
5. 冒頭の前提コメント `<!-- 前提: ... -->` 内の slug 記述も更新する。
6. 旧ファイル・旧ディレクトリが残っていないことを確認する。
7. 修正レポートと標準出力に 旧 → 新 の対応を明記する（`RESULT: ok` は**新パス**で出す）。

### E: 匿名化・マスク
| 対象 | 置換 |
|---|---|
| APIキー・トークン・パスワードの実値 | `sk-***`・`ghp_***`・`********` 等でマスク（形式が分かる程度に残す） |
| `/Users/<名前>/...`・`/home/<名前>/...` | `~/...` に一般化 |
| 内部ホスト名・社内URL | `example.internal` 等のダミーに置換し、ダミーである旨を一言添える |
| メールアドレス・個人名 | 記事に不要なら削除。必要なら `user@example.com` 等に |

* コマンド出力・エラー引用の中も対象（run-practice はエラー全文を転記するため混入しやすい）。
* マスクによって手順が再現不能にならないよう、「実際のキーに読み替える」旨を必要なら添える。
* **対象がすでに git コミット済みの場合**、ファイル修正だけでは履歴に残る。修正レポートに
  「⚠ git 履歴に残存。実キーであれば失効（ローテーション）を推奨」と警告を書く
  （履歴の書き換えはこのSkillではしない）。

## 修正不能の判断（Step 6 で使う）

次のいずれかに該当したら「修正不能」とする。無理に直さない。

* 記事の**核となる主張**（結論・完了条件の達成）がログで裏付けられず、削ると記事が成立しない
  → abort。推奨: `/run-practice` からやり直して素材を取り直す。
* 指摘が要求する素材（スクショ・実測値）がログに無く、削減もできない（完了条件の証拠など）
  → abort。推奨: `/run-practice` で該当タスクだけ再実行。
* slug 衝突の解消先が見つからない等、機械的に解決できない外部制約 → abort。
* レビューの指摘自体が事実誤認と思われる場合 → 記事は変えず、修正レポートに
  「指摘への反論と根拠（ログの該当箇所）」を書いて**未解消（warning扱い）**として残す。
  勝手に指摘を却下して ok にはしない（判断は再レビューに委ねる）。

## Report Format（修正レポート）

`logs/revise-<slug>-<日時>.md` に次の構成で保存する。

```md
# 修正レポート: <記事タイトル / slug>

## 修正の前提

- 対象記事: articles/<slug>.md（リネーム時: articles/<旧>.md → articles/<新>.md）
- レビューレポート: logs/review-<slug>-<日時>.md（判定: 要修正 / 公開不可）
- 出典ログ: logs/run-*/execution-log.md（または「出典不明」）
- 適用範囲: blocker + warning（+ suggestion の一部）
- 修正日時: <YYYY-MM-DD HH:MM>

## 結果サマリー

- 適用: blocker <n> 件 / warning <n> 件 / suggestion <n> 件
- 未解消: <n> 件（下記「未解消の指摘」）
- slug リネーム: なし / <旧> → <新>
- セルフチェック: SUMMARY fail=<n> warn=<n>（check-article.sh 再実行）

## 適用した修正（指摘ごと）

| # | 元の指摘（重大度 / 箇所） | 分類 | 適用した修正（before → after の要点） | 素材の出典 |
|---|---|---|---|---|
| 1 |  | A〜E |  | ログ該当箇所 / 機械修正 |

## 削除した記述（分類C で削ったもの）

- <節/段落>: <削った内容の要点> — 理由: <ログに裏付けが無い 等>

## 未解消の指摘

| # | 指摘（重大度） | 解消できない理由 | 推奨アクション |
|---|---|---|---|
| 1 |  |  | /run-practice で素材再取得 等 |

## 警告

- <あれば。例: ⚠ 秘密情報が git 履歴に残存。キーの失効を推奨>

## セルフチェック出力（check-article.sh）

```
<check-article.sh の出力を貼る>
```

## 次のアクション

- [ ] `/review-article <記事パス>` で再レビューする
- [ ] 判定が「公開可」になったら `/publish-pr` で公開準備する
```

## 書き方の注意

* 「適用した修正」は before → after が追える粒度で書く（再レビュー時の照合に使われる）。
* スキップ・未解消は理由を必ず書く。黙って放置しない（ループ検出の材料になる）。
* レポートのファイル名の `<slug>` は**修正後**の記事ファイル名を使う。
