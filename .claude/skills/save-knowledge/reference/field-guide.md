# フィールド記入ガイド

各テンプレートフィールドの記入方針。**会話履歴・git・ファイルから導出できるものは自動で埋め、
不足分だけユーザに質問する。**
空欄は禁止 — `N/A`（該当なし）または `未確認（要確認）` を使う。推測を事実として書かない。

> **記録言語**: 報告書の本文（各フィールドの中身・見出し）は **日本語** で記述する。
> ただし以下は英語のまま:
> - エラーメッセージ・ログ・コマンド・コードは **原文のまま**（改変しない）
> - 検索タグ `tech` / `error_type` / `library` と `cause_category` の値（後述の管理語彙）

---

## タイトル（自動生成）
- 1行・検索しやすく。「症状 + 技術/対象」で構成する。
- 例: `Vite開発サーバがポート競合で起動しない` / `pnpm installがpeer依存衝突で失敗する`
- ファイル名 slug はこのタイトルから導出する（英語 kebab-case を推奨）。

## 概要（自動生成）
- 問題 + 原因 + 修正を1〜3文で要約。一覧表示で内容が掴める粒度にする。

---

## 背景（できる限り自動収集）
| フィールド | 取得元 / 埋め方 |
|-------|----------------------|
| プロジェクト | repo / ディレクトリ名（`git remote` / `basename $(pwd)`） |
| 機能 / 作業内容 | 発生時に作っていたもの（会話から） |
| 技術スタック | `package.json` / `requirements.txt` / `go.mod` / Dockerfile から推定 |
| 環境 | OS、ランタイムversion、コンテナ/ローカル、CI など（環境情報・会話） |
| 発生タイミング | いつ起きたか（`起動時` / `ビルド時` / `デプロイ時` / `テスト中`） |
| 関連ファイル | 関与したファイルパス（編集差分・スタックトレースから） |
| 関連コマンド | 実行していたコマンド（会話中のコマンド履歴から） |

## 問題
- **期待した挙動**: 本来どうなるべきだったか。
- **実際の挙動**: 実際に起きたこと（症状）。
- **エラーメッセージ**: エラー / スタックトレースを **原文のまま** コードブロックに貼る（言い換えない）。
  長い場合は要点の行 + 省略記号。
- **再現手順**: 再現ステップをリストで。「このコマンドを実行 → こうなる」を推奨。

## 原因
- **推測した原因**: 最初に疑った原因（誤りでも記録する — トラブルシュートの軌跡）。
- **確定した原因**: 最終的に確認できた根本原因。未確定なら `未確認` と書く。
- **原因カテゴリ**: 下記リストから1つ選ぶ。
- **根拠**: 根本原因と判断した根拠（ログ行・実験結果・ドキュメント等）。

### 原因カテゴリ一覧（1つ選ぶ。値は英語のまま記録）
- `Configuration` — 設定ファイルや環境変数の誤り/不足
- `Dependency/Version` — 依存のversion不整合・パッケージ欠如
- `Environment` — OS/ランタイム/コンテナ差異、環境依存
- `Network/Connectivity` — 到達不可、DNS、プロキシ、ファイアウォール
- `Permission/Auth` — 権限不足、認証/トークン、証明書
- `Code/Logic` — 実装バグ、ロジック誤り、型/構文
- `Build/Compile` — ビルド/コンパイル/バンドル失敗
- `Resource` — メモリ/ディスク/ポート競合などのリソース枯渇
- `External Service` — 外部API/サービスの障害・仕様変更
- `Data/State` — データ不整合、キャッシュ、マイグレーション、状態破損
- `Concurrency/Timing` — 競合、順序、タイミング、非同期
- `Operation/Human` — オペミス、手順誤り
- `Unknown` — 根本原因が特定できていない

## 解決策
- **試したこと**: 試した順に（失敗した試行も含める — 価値が高い）。
- **最終的な修正**: 実際に効いた解決策。
- **変更ファイル**: 変更したファイル（`git diff --name-only` など）。
- **Before / After**: 主要な変更を before/after のコードブロックで。

## 検証
- **検証方法**: どう確認したか（コマンド再実行、アクセス確認等）。
- **テスト結果**: テスト結果（`pass` / `fail` / 件数 / `N/A`）。
- **ビルド結果**: ビルド結果（`success` / `failed` / `N/A`）。
- **残課題**: 残った作業 / 別途フォロー。なければ `なし`。

## 再発防止
- **防止策**: 再発防止のために行った/行うべきこと。
- **次回チェック手順**: 同じ症状が出たとき最初に確認すべき手順。
- **チェックリスト項目**: チェックリスト化できる項目（`[ ]` 形式）。

## 検索タグ（将来の検索性を最大化）
> タグ値（Tech / Error Type / Library）は **英語のまま** にする。grep検索性のため。
> Keywords のみ日本語の検索語を併記してよい。
- **Tech**: 言語/フレームワーク/ツール（例: `node`, `vite`, `docker`, `python`）
- **Error Type**: エラーの種類（例: `EADDRINUSE`, `ModuleNotFoundError`, `timeout`, `403`）
- **Library**: 関連ライブラリ名（例: `react`, `fastapi`, `prisma`）
- **Keywords**: 自由語。症状/原因の検索語（例: `port conflict`, `ポート競合`, `peer dependency`）

> 同じタグ値を YAML frontmatter と本文 `## 検索タグ` の両方に入れる。
> frontmatter は配列（`tech: [node, vite]`）、本文はカンマ区切りで可読性を優先。
