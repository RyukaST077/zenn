# Zenn 実践タスク ランナー（実装・検証・ログ記録）

> このSkillはユーザーが `/run-practice` で明示的に起動する前提。自動起動はしない。
> **非対話（headless）実行を前提とする**（例: `claude -p "/run-practice ..."`）。
> 途中でユーザーに確認・質問しない。`AskUserQuestion` は使わず、情報が不足する場合は
> 常に下記のデフォルト前提を採用してそのまま最後まで進める。
> 詳細な参照データは `reference/` に分離してある。各ステップで必要になったら該当ファイルを読む。

## パイプライン上の位置づけ

このSkillは3段パイプラインの**3段目**にあたる。

```
/search-topic  → research/search-topic-*.md   （テーマ候補レポート）
/plan-practice → practice/practice-*.md        （実践タスクのチェックリスト）
/run-practice  → logs/run-*/execution-log.md   （実装・検証の結果＝記事の素材）  ← これ
```

## Goal

`plan-practice` が出力した**実践タスク（practice/practice-*.md）を、AIエージェントが単独で
実際に手を動かして実行・検証し、その過程を記事の一次素材として記録する**。目的は、
Zennの経験談（試してみた / 詰まった点まとめ / 検証ログ）を書くための素材が
「実行するだけで揃っている」状態を作ること。

タスクのチェックボックスを上から順に実行し、各タスクで **実行コマンド・出たエラー（全文）・
効いた対処・所要時間（見積もり→実測）・気づき・スクリーンショット** を残す。うまくいった点
だけでなく、**詰まった点と解決過程**を記録の中心に置く。

**このSkillの担当範囲は「素材収集まで」**。実行して得た記録を `plan-practice` の
「記事への写像」に沿って整理し、`logs/run-*/execution-log.md` に保存するところで終わる。
**Zenn記事の本文（articles/*.md）は書かない**（記事化は別工程 / 人間 or 別スキルに委ねる）。

最終成果物:

* 実行した実践タスクの結果（達成 / 一部達成 / 未達 を完了条件で判定）
* 各タスクの実行ログ（コマンド・エラー全文・対処・所要時間）
* スクリーンショット（ブラウザ表示は Playwright、CLIは出力ログ）
* 詰まった点と解決過程の棚卸し
* 「記事への写像」を実績で埋めた対応表（どの記録がどの見出しの素材か）
* 未解決・撤退した項目とその理由

## 起動方法と引数

`/run-practice` で起動。以下を引数（自然言語）で渡せる。すべて省略可。

* 対象タスクファイル: `practice/practice-*.md` のパス（無指定なら最新を自動選択）
* 実行に使える時間（例: 1〜2時間, 半日, 1日 / タイムボックス超過時は撤退ライン適用）
* 途中まで実行済みの場合の再開指示（例: フェーズ3から / 未達タスクだけ）
* 撤退ラインの上書き（例: 1タスク30分詰まったら記録して次へ）
* 成果物コードの置き場の希望（無指定なら `logs/run-<slug>-<日時>/workspace/`）

### 引数が無い/不足のときのデフォルト前提

* 対象タスク: `practice/` 内で最も新しい `practice-*.md`
* 実行時間: 対象タスクの「検証に使える想定時間」。無ければ半日〜1日
* 実行者: AIエージェント単独（人手の介在なし・非対話）
* 撤退ライン: 対象タスクの「想定リスク・注意点」の撤退ラインを採用。無ければ
  「1タスクで30分以上詰まったら、状況を記録してスキップ or 代替に切替」
* 成果物コードの置き場: `logs/run-<slug>-<日時>/workspace/`

引数の値は対象タスクファイル内の前提より優先する。不足情報があっても止まらず、
採用した前提を出力（execution-log.md）の冒頭に明示したうえで進める。
ユーザーへの追加質問は行わない（非対話実行のため応答を待てない）。

## Assumptions

* 実行者はAIエージェント単独。人手のサインアップ・認証・手動操作は一切できない。
* 対象タスクは `plan-practice` の Executability Gate を通過済み（無料・認証不要・ローカル完結）
  である前提。ただし実行時に人手必須が判明したら撤退ライン（後述）で扱う。
* 「詰まった点」も記事の価値。うまくいく前提で進めず、エラーと解決過程を漏らさず記録する。
* 記録はすべて**一次情報**（実際に出た出力・エラー・スクショ）で残す。捏造しない。

## 実行環境の制約（重要 / 必須）

すべての作業は「**AIエージェントが単独で、人手を介さず**最後まで実行・検証できる」範囲で行う。

必須（すべての実行が満たすこと）:

* CLI・コード実行・ファイル操作だけで完結する
* 完全無料・**認証不要**・ローカルで完結する（またはトークン不要の公開API）
* 対話プロンプトを避ける（`--yes` / `-y` / `--non-interactive` などの非対話フラグを使う）
* ブラウザ表示の確認は **Playwright で自動操作・スクリーンショット**する
  （完了条件の判定も Playwright のスクショで行う）

禁止（実行しない。遭遇したら撤退ラインで扱う）:

* **課金APIキー**やクレジットカード登録が必要な操作
* 人手のサインアップ／メール・電話認証／手動OAuthログイン／ダッシュボードでの手動トークン発行
* 手動デプロイ・手動UI操作・実機・GUI専用アプリ・有料クラウドの本番リソース

実行中にタスクが上記の禁止事項を要求してきた場合の扱い:

* まず **ローカル・無料・認証不要で完結する等価手段に置き換えられないか**を試す
  （例: デプロイ→`wrangler dev` 等のローカル起動、外部SaaS→ローカルCLI/Docker）。
  置き換えた場合は、その旨と理由を execution-log.md に記録する。
* 置き換えても人手が必須なら、そのタスクは**未達（撤退）として記録**し、次のタスクへ進む。
  完了したふり・手順の捏造はしない。

判定基準の OK/NG 例は `reference/execution-guide.md` の Executability を参照。

## トラブル対応（knowledge ループ / 必須）

このSkillはコマンドを大量に実行するため、トラブルに当たる前提で動く。CLAUDE.md の
ナレッジループに従う。

* コマンド失敗・エラーに当たったら、**ゼロから調べる前に `consult-knowledge` で
  `knowledge/` を検索**し、過去の解決策があれば再利用する。
* 自力で解決した新しいトラブルは、`save-knowledge` で `knowledge/` に記録する。
* いずれの場合も、そのエラー全文・試したこと・効いた対処を execution-log.md にも残す
  （記事の「詰まった点」素材になる）。

## Input Resolution（入力の特定）

必ず最初に対象タスクを1つに確定させてから実行に入る。

1. **タスク特定**: 引数にパスがあればそれを使う。無ければ Bash で最新を選ぶ。
   `ls -t practice/practice-*.md 2>/dev/null | head -1`
2. **タスク読込**: 特定したファイルを Read し、成果物・完了条件・完了確認の方法・
   フェーズ別タスク・詰まりポイント表・記録テンプレ・記事への写像・想定リスク（撤退ライン）
   を把握する。
3. **フォールバック**: `practice-*.md` が1件も無い場合、
   「対象の実践タスクが見つからない。先に `/plan-practice` を実行して practice/practice-*.md
   を作ってください」と標準出力に表示して**そこで終了する**（タスクを捏造しない）。

## Workflow

### Step 1: 入力を確定する
上記「Input Resolution」に従い、対象タスクファイルを1つに確定する。採用した前提
（タスクパス／実行時間／撤退ライン／成果物コードの置き場）を控えておく。

### Step 2: 実践タスクを解析してタスク化する
対象ファイルから「完成イメージ（成果物・完了条件・完了確認の方法）」「事前準備チェックリスト」
「フェーズ別タスク」「詰まりポイント表」「記録テンプレ」「記事への写像」「想定リスク」を抽出する。
フェーズ別タスクのチェックボックスを **TaskCreate** でそのままタスク登録し、実行順に並べる
（実行中は **TaskUpdate** で in_progress / completed を更新する）。

### Step 3: 作業環境を準備する
`reference/execution-guide.md` の Workspace Setup に従い、実行成果物の置き場を作る。

* `date +%Y%m%d-%H%M` で日時を取得し、対象ファイル名からスラッグを引き継ぐ
  （`practice-<slug>-*.md` の `<slug>`。無ければ `task`）。
* 出力ディレクトリ `logs/run-<slug>-<日時>/` を作り、その中に `workspace/`（成果物コード）と
  `screenshots/`（スクショ）を用意する。
* 事前準備チェックリストに沿ってランタイム・バージョンを確認し、必要な依存をインストールする
  （非対話フラグを使う）。ブラウザ表示を伴う成果物なら Playwright を用意する
  （`npm i -D playwright` は宣言済み。実行時に `npx playwright install chromium` を行う）。

### Step 4: フェーズ順にタスクを実行・検証する【本編】
チェックボックスを上から順に実行する。各タスクで:

* 対応する Task を in_progress にし、`date +%s` などで開始時刻を控える。
* コマンド実行・コード記述で作業する（`reference/execution-guide.md` の Run & Capture）。
* **標準出力・標準エラーを要約せず全文で捕捉**し、`logs/run-*/commands.log` と
  該当タスクの記録に残す。
* 完了条件を客観的に判定する。CLI/API なら実際にコマンド/リクエストを叩いて期待出力を確認。
  ブラウザ表示なら Playwright で操作してスクショを `screenshots/NN-<label>.png` に保存する。
* エラーに当たったら「トラブル対応（knowledge ループ）」に従い、対象タスクの詰まりポイント表の
  対処も試す。効いた対処・所要時間・つまずいた理由を記録する。
* 撤退ライン（既定: 1タスク30分）を超えて詰まったら、状況・エラー全文・試したことを記録して
  スキップ or 等価手段に切替し、次のタスクへ進む。**完了のふりはしない**。
* 対応する Task を completed（または未達なら in_progress のまま + 未達メモ）にする。

### Step 5: 完了条件を判定する
成果物の「できたと言える完了条件」を1つずつ客観的に検証する（CLI出力 / レスポンス /
Playwrightスクショ）。達成 / 一部達成 / 未達 を根拠（ログ・スクショのファイル名）付きで確定する。
未達は理由とともに正直に記録する。

### Step 6: 記録を素材として整える
`reference/output-template.md` に従い、各タスクの記録テンプレを埋める（実行コマンド・
エラー全文・効いた対処・所要時間の見積もり→実測・つまずいた理由・既存技術との違い・
スクショの参照・記事に書きたい気づき）。再現性のため OS・ランタイム・バージョンを明記する。

### Step 7: 記事への写像を実績で埋める
対象ファイルの「記事への写像」表を引き継ぎ、右列に**実際の記録（どのログ・どのスクショ・
何を書くか）**を紐づける。「実行すればその見出しの素材が埋まる」状態にする。
※ ここで**記事本文は書かない**。見出しごとに使う素材を指し示すところまで（境界）。

### Step 8: 素材ファイルを組み立てて保存する
`reference/output-template.md` のテンプレートに沿って `execution-log.md` を構成し、
後述の Output に従って**必ずファイルとして保存する**。

## Output

最終成果物は `reference/output-template.md` のMarkdownテンプレートに沿って構成し、
**必ずファイルとして保存する**（非対話実行では会話ログが残らないため）。

1. `date +%Y%m%d-%H%M` で日時を取得する
2. リポジトリ直下に `logs/` が無ければ作成する
3. 対象ファイルのスラッグを引き継ぎ、`logs/run-<slug>-<日時>/` を作成する
4. 検証ログ本体を `logs/run-<slug>-<日時>/execution-log.md` に書き込む。
   スクショは同ディレクトリの `screenshots/` に、成果物コードは `workspace/` に置く
5. 最後に、以下を標準出力にも短く表示する:
   * 保存した execution-log.md のパス
   * 対象タスクファイルのパス
   * 完了条件の判定結果（達成 / 一部達成 / 未達）
   * 所要時間の合計（見積もり → 実測）
   * 取得したスクショ枚数 / 未解決・撤退した項目の数
   * 次の工程（記事化）の案内: 「記事本文は未作成。execution-log.md の『記事への写像』を
     もとに articles/*.md を書く」

## Quality Rules

* 実行してログを残す。解説だけ書かない。すべての記録は実際に出た一次情報にする（捏造禁止）。
* エラーは**全文**を残す（要約しない）。効いた対処と所要時間（見積もり→実測）を必ず添える。
* 完了条件は客観的に判定する（CLI出力 / レスポンス / Playwrightスクショ）。未達は正直に書く。
* トラブルは先に `consult-knowledge`、新規解決は `save-knowledge`（CLAUDE.md ループ）。
* ブラウザ確認は Playwright でスクショ。CLI/APIはコマンド出力ログで残す。
* 非対話で完結させる（`--yes` 等）。人手サインアップ・課金・手動デプロイはしない（撤退で記録）。
* 「記事への写像」を実績で埋め、実行するだけで素材が揃う状態にする。
* **記事本文は書かない**（素材収集が担当範囲。articles/ は触らない）。
* 採用した前提（対象タスク・時間・撤退ライン）を execution-log.md の冒頭に明示する。

## Anti-Patterns

* 実行せずに手順を清書するだけ／エラーを要約して全文を残さない。
* 完了条件を確認せず「できた」と書く／未達を隠す・成功を捏造する。
* スクショを撮らずにブラウザ表示を「確認した」と書く。
* トラブル時に `knowledge/` を確認せず毎回ゼロから調べる／解決を記録しない。
* 課金・人手サインアップ・手動デプロイに手を出す（AIエージェント単独で完結しない）。
* 担当範囲を越えて記事本文（articles/*.md）まで書いてしまう。
* 対象タスクが無いのにタスクを捏造して実行する。
