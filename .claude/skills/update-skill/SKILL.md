---
name: update-skill
description: 既存 Skill（.claude/skills/<name>/）を安全に更新するための Skill。引数で対象 Skill のパスと更新要望（自然言語）を受け取り、現状読み取り → 影響範囲の分析 → 差分ドラフト提示 → 承認 → 書き出し、までを一連でガイドする。ユーザが「Skill を更新して」「この Skill に○○を追加」「<skill名>のdescriptionを直して」などと言ったら起動する。既存 Skill の編集専用（新規作成は skillify を使う）。
argument-hint: <skill-path> "<update-description>"
---

# 目的

**既存の Skill** を、意図を崩さず・壊さずに更新する。

対象となる更新例:
- SKILL.md の frontmatter（`name` / `description`）修正
- 手順の追加・削除・順序変更
- 起動条件（トリガーフレーズ）の拡張・絞り込み
- 付随ファイル（`scripts/` / `templates/` / `reference/`）の追加・編集・削除
- Skill 全体の構造リファクタ（長くなりすぎた SKILL.md を `reference/` に分割する等）

**このSkillが扱わないもの**:
- 新規 Skill の作成 → `skillify` を使う
- 複数 Skill の一括更新（1 回の呼び出しでは 1 Skill のみを対象）
- Skill の削除（ユーザが直接 `rm -rf` を実行すべき、もしくは明示的に許可を求める）

---

# 引数

本 Skill は Claude Code の位置引数として次の 2 つを受け取る:

| 位置 | 意味 | 例 |
|------|------|-----|
| `$1` | 対象 Skill のディレクトリ or SKILL.md への絶対/相対パス | `.claude/skills/skillify` / `/home/.../SKILL.md` |
| `$2` | どのように更新したいかの自然言語記述（空白を含む場合はクォートで囲む） | `"description にトリガー例を 2 つ追加"` |

呼び出し例:

    /update-skill .claude/skills/skillify "Phase 7 の手順を簡潔にまとめ直して"

`$1` または `$2` が空の場合は、Phase 1 冒頭でユーザに質問して補完する。

---

# 起動条件

ユーザが次のような発話をしたら本 Skill を呼ぶ:

- 「<skill名> Skill を更新して」
- 「この Skill の description を変えて」
- 「<skill名>に手順を追加」
- 「SKILL.md をリファクタして」
- 「既存の Skill を直したい」

**前提**:
- 対象 Skill がすでに `.claude/skills/<name>/SKILL.md` として存在する
- サンドボックス環境なら `.claude/settings.local.json` の `sandbox.filesystem.allowWrite` に `./.claude/skills/**` が含まれる

---

# 全体フロー

```
Phase 1: 引数確認      → $1 / $2 の検証
Phase 2: 現状読み取り  → SKILL.md + 付随ファイル一式を読み、構造を把握
Phase 3: 影響分析      → 更新要望がどのファイル/どのセクションに及ぶか特定
Phase 4: ドラフト生成  → 変更案（diff または全文）を画面に提示
Phase 5: 承認          → OK / 修正依頼 / キャンセル
Phase 6: 書き出し      → Edit / Write で反映、結果を提示
```

Phase 5 の承認前に **ディスクへ一切書き込まない**。

---

# Phase 1: 引数確認

1. `$1`（対象パス）が与えられたか確認
   - ディレクトリなら `<dir>/SKILL.md` を対象にする
   - 単体ファイルならそのまま対象
   - 相対パスは現在の作業ディレクトリ基準で絶対パスに解決
2. `$2`(更新要望)が与えられたか確認
3. どちらか不足していれば、その場でユーザに聞く:

   ```
   以下を教えてください:
     - $1: 対象 Skill のパス（例: .claude/skills/skillify）
     - $2: どのように更新したいか（例: description にトリガー例を追加）
   ```

4. パス存在確認: `ls` / `Glob` で `$1` が指すディレクトリに `SKILL.md` があるか確認。無ければ「新規作成なら `skillify` を使ってください」と案内して終了。

---

# Phase 2: 現状読み取り

対象 Skill のすべてのファイルを読む:

1. `SKILL.md` 全文（`Read`）
2. 付随ファイル一覧（`Glob` で `<dir>/**/*`）
3. 必要に応じて `scripts/` / `templates/` / `reference/` の中身も読む（update_request の範囲に応じて）

読み終えたら、内部で次を把握する（ユーザには要点だけ見せる）:
- 現在の `name` / `description`
- セクション構成（見出し一覧）
- 参照している付随ファイルと、実ファイルの有無の突合
- SKILL.md の行数（200行超なら分割も視野に）

要約をユーザに提示:

```
現状を把握しました:
  - name: <name>
  - description: <先頭80文字>…
  - SKILL.md: <行数> 行
  - 付随ファイル: scripts/xxx.sh, reference/yyy.md, …
  - 参照整合性: OK / 下記に不整合あり
```

---

# Phase 3: 影響分析

`$2`(更新要望)を読み解き、**どのファイルの・どのセクションが・どう変わるか** を列挙する。

提示例:

```
更新計画:

  [1] SKILL.md
      - frontmatter.description: トリガーフレーズを 2 つ追加
      - 「# 起動条件」節: 例示リストに 2 行追加

  [2] reference/interview-questions.md
      - 変更なし

  [3] 新規追加: なし
      削除: なし

この方針でドラフトを作成してよいですか？
他に直したい箇所があれば先に教えてください。
```

`$2` が曖昧で判断できない場合は **必ず質問** する。勝手に解釈して範囲を広げない。

---

# Phase 4: ドラフト生成

変更量に応じて提示形式を使い分ける:

| 変更量 | 提示形式 |
|--------|----------|
| 数行程度 | **差分（before/after 並記）** |
| 1 セクション以上 | **変更後の該当セクション全文** |
| 構造変更・ファイル追加/削除 | **変更後のファイルツリー + 主要ファイル全文** |

frontmatter を変更する場合は必ず **変更後の frontmatter ブロックを全文** 提示する（部分変更は壊しやすい）。

ドラフトに加え、以下の **セルフチェック結果** も併記:

- [ ] `name` フィールドと実ディレクトリ名は一致しているか
- [ ] `description` は具体的なトリガーフレーズを含むか
- [ ] 追加したリンク（`reference/xxx.md` など）は実ファイルが存在するか / 生成予定か
- [ ] 200 行を超えるなら分割を提案したか
- [ ] 既存の見出し階層と整合しているか

---

# Phase 5: 承認

| ユーザ応答 | 次アクション |
|-----------|-------------|
| 「OK」「反映して」「書いて」 | Phase 6 へ |
| 「ここを直して」「～は外して」 | 指摘を反映し Phase 4 を再提示 |
| 「キャンセル」「やめる」 | 何も書かずに終了 |

---

# Phase 6: 書き出し & 完了案内

1. 変更の適用
   - **部分編集** → `Edit`(old_string/new_string で最小差分)
   - **全面書き換え** → `Write`（事前に `Read` 必須）
   - **新規ファイル** → `Write`
   - **ファイル削除** → Bash の `rm`（事前にユーザへ明示確認）
2. 変更後の `ls -la <dir>/` 結果を提示
3. 主要ファイルの冒頭数行を `Read` で再確認し、frontmatter が壊れていないことを示す
4. コミット案内:

```
✅ Skill「<name>」を更新しました。

📁 変更ファイル:
  - SKILL.md (修正)
  - reference/xxx.md (新規)
  - ...

🔄 次のステップ:
  - 動作確認: 別会話でトリガーフレーズを試す
  - 必要なら SKILL.md を直接再編集
  - コミット: `git add .claude/skills/<name>/ && git commit -m "update skill: <name>"`
```

# 失敗・エッジケース

## `$1` に SKILL.md が無い
→ 「新規作成なら `skillify` を使ってください」と案内して終了。勝手に新規作成しない。

## `$2` が広すぎる / 曖昧
→ Phase 3 で必ず質問し範囲を確定させる。推測で進めない。

## サンドボックス書き込み拒否
→ `.claude/settings.local.json` の `sandbox.filesystem.allowWrite` を確認・追加依頼。
→ 不可なら `$TMPDIR` に生成 → 手動移動を案内。

## frontmatter が壊れそうな編集
→ frontmatter は **必ず全文置換** で更新する（部分 Edit は `---` の数や YAML 整合性を壊しやすい）。

## 参照ファイルの不整合
→ SKILL.md 内で参照している `reference/xxx.md` が存在しない等、既存の不整合を見つけたら Phase 2 で報告。ユーザ依頼に含まれなければ勝手に直さず、「ついでに直しますか？」と確認。

## 想定外の巨大化
→ 更新後 SKILL.md が 250 行を超える見込みなら、分割（`reference/` 切り出し）を Phase 4 で提案する。ユーザ承認を取ってから分割。

---

# 実行時の心構え

- **最小差分を守る**: 依頼された範囲だけを変える。ついでのリファクタ・スタイル統一は勝手にやらない。
- **意図を壊さない**: 既存の書きぶり（丁寧さ・絵文字の有無・日本語/英語）は踏襲する。
- **frontmatter は神経質に**: `name` / `description` は Skill 発見の唯一の手がかり。変更時は特に慎重に。
- **承認前に書かない**: Phase 5 を通過するまで `Edit` / `Write` / `rm` を呼ばない。
- **description 更新時は name と整合**: `name` 変更はディレクトリ名の rename を伴うため、原則避ける（必要ならユーザに明示確認）。
