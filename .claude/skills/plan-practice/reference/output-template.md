# 出力テンプレート & 記事写像

`plan-practice` の Step 7（記事への写像）と最終出力（Step 8）で使う。

## Output Format

最終出力は次のMarkdownテンプレートに沿って構成する。プレースホルダは対象テーマに合わせて埋める。
チェックボックス・目安時間・記録項目は必ず残す。

## 記録テンプレ（実行中に埋める）

各タスクを実行しながら、以下を都度メモする。これがそのまま経験談の一次情報になる。

- 実行したコマンド:
- 出たエラー（全文）:
- 効いた解決方法 / 試したこと:
- 所要時間（見積もり → 実測）:
- つまずいた理由・分かっていなかった前提:
- 既存技術と比べて感じた違い:
- スクショを撮った箇所:
- 記事に書きたい気づき:

## 記事への写像（タスク → 見出し）

出典レポートの「記事構成案」に対応させる。実行後、右列の記録を左の見出しに流し込む。

| 記事の見出し | 対応するタスク / 記録 | 書くこと |
|---|---|---|
| はじめに | 前提・動機 | なぜこの技術を試したか |
| 環境構築 | フェーズ2の記録 | 手順・詰まった点 |
| 実際に試したこと | フェーズ3の記録 | 作った成果物・コード |
| 詰まった点 | 詰まりポイント表・記録テンプレ | エラーと解決 |
| 分かったこと / 比較 | フェーズ4の記録 | 既存技術との違い |
| まとめ | フェーズ5の棚卸し | 向いている人・次にやること |

## 経験談として書くときのコツ

- 「新人が試した範囲」を明示し、専門家として断定しすぎない
- うまくいった点だけでなく、詰まった点と解決過程を書く
- 実行ログ・スクリーンショット・コードを残して貼る
- 公式ドキュメントへのリンクを入れる
- 手順の再現性（バージョン・OS・前提）を明記する

## Slug Rules（保存ファイル名のスラッグ）

`practice/practice-<スラッグ>-<日時>.md` のスラッグは対象技術から作る。

* 英語技術名を小文字にし、空白・記号をハイフンにする（例: `Cloudflare Workers` → `cloudflare-workers`）。
* 複数技術なら中心の1つを使う。日本語しか手がかりが無い場合はローマ字か英語キーワードにする。
* どうしても作れない場合は `task` を使う。

## Article Mapping の作り方

* 出典レポートの「記事構成案」があればそれを見出しの左列に採用する。無ければ上の既定表を使う。
* 各見出しに、対応するフェーズ番号または記録項目を必ず紐づける。
* 「実行すればその見出しの素材が埋まる」状態になっているか点検する（素材が出ないタスクは削るか記録項目を足す）。
