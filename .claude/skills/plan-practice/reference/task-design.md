# 実践タスク設計ガイド

`plan-practice` の Step 4（成果物定義）/ Step 5（タスク分解）/ Step 6（詰まりポイント予測）で使う。

## Deliverable（成果物の決め方）

実践タスクは「動く小さな成果物」をゴールに置く。抽象的な理解ではなく、
**完了条件が客観的に判定できるもの**にする。

良い成果物の条件:

* 半日〜1日（目安3〜7時間）で完成する
* 無料枠・ローカル環境・個人開発レベルで動く
* 「動いた／動かない」が画面・ログ・レスポンスで判定できる
* 記事に載せられる（スクショ・コード・実行ログが残る）
* 対象技術1つに集中している（複数技術を混ぜない）
* AIエージェント単独（CLI・コード・Playwright）で作成から検証まで完結できる
* 完全無料・認証不要・ローカルで完結する（課金キー・人手サインアップが不要）

成果物の型（テーマに合わせて選ぶ）:

* ミニアプリ（Todo / メモ / 簡易ダッシュボード）
* 小さなAPI（1〜3エンドポイント）
* CLIツール（1コマンド）
* 既存アプリ/チュートリアルへの組み込み（認証・DB・UIライブラリ等）
* 公式チュートリアルの完走＋独自アレンジ1つ
* 比較検証（既存ツール vs 新ツールで同じ小タスクを作る）

完了条件の書き方（例）:

* `http://localhost:3000` でTodoの追加・削除ができる（Playwrightで操作しスクショで確認）
* `curl` で3つのエンドポイントが期待どおり応答する
* `npm run build` が通り、`npm run preview` などローカル起動で表示される（手動デプロイは含めない）

## Executability Gate（AIエージェント単独実行の必須条件）

すべてのタスクは、人手を介さずAIエージェントだけで実行・検証できること。
成果物・各タスクを次で判定し、NGは作らない（等価タスクに置き換える）。

### OK

* ローカルで動くライブラリ・CLI・フレームワーク（npm / pip / cargo 等）
* トークン不要の公開API、ローカルのモデル・DB（Docker / SQLite / ローカルLLM）
* `wrangler dev` / `supabase start` などローカルで完結する範囲のクラウド系ツール
* Webアプリの表示・操作は Playwright で自動操作し、スクリーンショットで完了確認

### NG（→ 代替に置き換える）

* 課金APIキー・クレジットカード登録が必要 → ローカルモデル／トークン不要APIに置換
* 手動サインアップ・メール/電話認証・手動OAuth → 認証不要のツール/ローカル完結に置換
* 手動デプロイ・ダッシュボード操作 → `... dev` 等のローカル起動に置換
* CAPTCHA・ログイン壁でブラウザ自動化が不可 → 対象サイト/機能を変更

置き換えても人手が必須なら、そのテーマは実践タスク化しない（SKILL.md の Step 4/5 で次候補へ）。

## Phase Design（フェーズ分解）

タスクは次の5フェーズに分ける。各フェーズにチェックボックス粒度のタスクを並べる。

1. **事前調査**: 公式ドキュメント確認、前提の把握、作る成果物の確定
2. **環境構築**: インストール、初期化、Hello World レベルで起動確認
3. **実装・検証（本編）**: 成果物を作る中心タスク。ここを厚く
4. **深掘り・比較**: 既存技術との違い、追加で試したこと、限界を確認
5. **振り返り・記事化準備**: 記録の整理、詰まった点の棚卸し、記事構成へのマッピング

## Task Granularity（タスクの粒度）

* 1タスク = 目安 **15〜60分**。60分を超えそうなら分割する。
* チェックボックスで「やった／やってない」が判定できる動詞で書く
  （例: 「〜をインストールする」「〜を起動して表示を確認する」）。
* 「〜を理解する」「〜に慣れる」など完了判定できない書き方は避ける。

### Good

* [ ] `npm create hono@latest` で雛形を作り、`npm run dev` で起動する（目安10分）
* [ ] `/users` GET を実装し、`curl localhost:8787/users` でJSONが返るのを確認する（目安30分）

### Bad

* [ ] Honoを理解する
* [ ] APIを作る（← 大きすぎ・完了条件なし）

## Recording Items（記録すること = 経験談の素材）

各タスクに「記録すること」を必ず添える。実行中にこれを埋めれば経験談が書ける。

標準の記録項目:

* 実行したコマンド（コピペできる形で）
* 出たエラーの**全文**（要約せず貼る）
* 解決までに試したこと・効いた対処
* 所要時間（見積もりとの差）
* つまずいた理由・分かっていなかった前提
* 既存技術と比べて感じた違い
* スクリーンショットを撮るべき箇所（画面・ログ）
* 「これは記事に書きたい」と思った気づき

## Time Budget（時間配分の目安）

指定時間に合わせてフェーズへ配分する。

* **1〜2時間**: 事前調査15分 / 環境構築30分 / 本編60分 / 振り返り15分（深掘りは省略可）
* **半日（約3〜4時間）**: 事前調査30分 / 環境構築45分 / 本編120分 / 深掘り30分 / 振り返り30分
* **1日（約6〜7時間）**: 事前調査45分 / 環境構築60分 / 本編180分 / 深掘り90分 / 振り返り45分

各タスクの目安時間の合計が指定時間に収まるか確認する。超える場合は本編以外を削るか、
成果物を小さくする。

## Pitfalls（詰まりポイント予測の観点）

新人が詰まりそうな点を、次の観点から最低3件は予測する。

* バージョン不一致・ランタイム要件（Node/Bun/Python のバージョン、Nightly依存 等）
* 環境変数・APIキー・認証設定の不足
* 認証・課金の壁（APIキーが有料、鍵取得に人手のサインアップが必要）→ ローカル/無料/認証不要な代替に置き換える
* 無料枠の制限・レート制限・リージョン制約
* 破壊的変更・古い記事のコマンドが動かない
* 初期化コマンドの対話プロンプト（headless/CIで詰まる。`--yes` 等の非対話フラグを探す）
* ブラウザ確認が手動前提になっている → Playwright で自動操作・スクリーンショットに置き換える
* CORS・ポート衝突・localhost まわり
* 型エラー・ビルド設定（tsconfig / bundler）
* OS差（macOS / Windows / WSL）

各ポイントは「なぜ起きるか / 最初に試すこと / 記事での活かし方」の3点で整理する。

## Quality Rules（設計時の品質基準）

* 成果物と完了条件を最初に固定してからタスクを分解する。
* すべてのタスクに完了判定できる書き方と記録項目を付ける。
* 手順は裏取りした一次情報に基づく。未確認は「要確認」と明記する。
* 目安時間の合計を指定時間内に収める。
* 詰まりポイントを事前に予測し、記事価値に変換する導線を用意する。

## Anti-Patterns

* 完了条件のない抽象タスク／記録項目のないタスク。
* 成果物が定義されないまま手順だけ並べる。
* 1タスクに複数技術・複数目的を詰め込む。
* 課金前提・重い環境構築前提・数日かかる規模。
* うまくいく前提だけで詰まりどころを想定しない。
