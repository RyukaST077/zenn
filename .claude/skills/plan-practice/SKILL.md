# Zenn 実践タスク プランナー

> このSkillはユーザーが `/plan-practice` で明示的に起動する前提。自動起動はしない。
> **非対話（headless）実行を前提とする**（例: `claude -p "/plan-practice ..."`）。
> 途中でユーザーに確認・質問しない。`AskUserQuestion` は使わず、情報が不足する場合は
> 常に下記のデフォルト前提を採用してそのまま最後まで進める。
> 詳細な参照データは `reference/` に分離してある。各ステップで必要になったら該当ファイルを読む。

## Goal

`search-topic` が選んだZenn記事テーマ候補を入力に、新人エンジニアが**実際に手を動かして
検証できる「実践タスク」**へ落とし込む。目的は、そのタスクを順番に実行するだけで、そのまま
Zennの**経験談（試してみた / 詰まった点まとめ / 検証ログ）**の素材が溜まる状態を作ること。

「解説記事の目次」ではなく「作業チェックリスト＋記録項目」を作る。各タスクには
「何をやるか」「完了条件」「経験談として何を記録するか」を必ず付ける。

最終成果物:

* 対象テーマと、そこから作る成果物（完成イメージ）の定義
* 事前準備チェックリスト
* フェーズ別の実践タスク（チェックボックス／所要時間／記録すべきこと付き）
* 新人が詰まりそうなポイントと対処の指針
* 実行中に埋める記録テンプレ
* 「タスク → 記事の見出し」への写像（経験談への展開表）
* 参考リンク・想定リスク・次のアクション

## 起動方法と引数

`/plan-practice` で起動。以下を引数（自然言語）で渡せる。すべて省略可。

* 対象レポート: `research/search-topic-*.md` のパス（無指定なら最新のレポートを自動選択）
* 対象テーマ: テーマ番号（例: `テーマ2`）またはテーマ名・キーワード
  （無指定ならレポートの「最初に試すべき1本」を採用）
* 検証に使える時間（例: 1〜2時間, 半日, 1日）
* スキルレベル（例: 新人, 実務1年目, 独学中）
* 成果物の希望（例: Todoアプリ, 小さなAPI, CLIツール, 既存アプリへの組み込み）
* 避けたいこと（例: 課金, 重い環境構築, 数学重め）
* レポートを使わず直接テーマを渡す（フォールバック。例: `テーマ: Honoで小さなAPIを作ってみる`）

### 引数が無い/不足のときのデフォルト前提

* 対象レポート: `research/` 内で最も新しい `search-topic-*.md`
* 対象テーマ: そのレポートの「最初に試すべき1本」。無ければ候補表の優先度1位
* 想定筆者: Web系の新人エンジニア / 想定読者: 新人〜実務2年目
* 検証時間: 半日〜1日
* 記事タイプ: 「試してみた」「初めて触ってみた」「詰まった点をまとめた」
* 成果物: 半日〜1日で作れる最小の動く成果物（ミニアプリ／小さなAPI／既存への組み込み）

引数の値はレポート内の前提より優先する。不足情報があっても止まらず、採用した前提を
出力の冒頭に明示したうえで進める。ユーザーへの追加質問は行わない（非対話実行のため応答を待てない）。

## Assumptions

* 筆者は新人〜実務経験が浅いエンジニア。対象技術の深い経験はまだない。
* これから実際に手を動かして検証し、その過程を記事にする。
* 半日〜1日で完了でき、無料枠・ローカル環境・個人開発レベルで実行できるタスクにする。
* 「詰まった点」も記事の価値。うまくいく前提で書かず、詰まりどころを事前に洗い出す。

## 実行環境の制約（重要 / 必須）

実践タスクは「**AIエージェント（Claude Code等）が単独で、人手を介さず**最後まで実行・
検証できる」ものだけにする。人手が必要なタスクは組み込まない。

必須（すべてのタスクが満たすこと）:

* CLI・コード実行・ファイル操作だけで完結する
* 完全無料・**認証不要**・ローカルで完結する（またはトークン不要の公開API）
* ブラウザ操作や画面確認は **Playwright で自動操作・スクリーンショット**する
  （完了条件の確認も Playwright のスクショで行う）

禁止（タスクに含めない）:

* **課金が必要なAPIキー**やクレジットカード登録（用意する人がいない）
* 人手のサインアップ／メール・電話認証／手動OAuthログイン／ダッシュボードでの手動トークン発行
* 手動デプロイ・手動UI操作・実機・GUI専用アプリ・有料クラウドの本番リソース

対象テーマがこの制約に反する場合の扱い（Step 4 / Step 5 で判断）:

* まず **ローカル・無料・認証不要で完結する等価タスクに置き換える**
  （例: デプロイ→`wrangler dev` 等のローカル起動、外部SaaS→ローカルCLI/Docker、
  有料LLM→ローカルモデルやトークン不要API）。
* 置き換えても人手が必須になる場合は、出典レポートから**次に実行可能な候補**へ切り替え、
  切り替えた旨と理由を出力冒頭に明記する。
* いずれも不可なら、その旨を標準出力に表示して**終了する**（実行不能なタスクは作らない）。

判定基準の OK/NG 例は `reference/task-design.md` の Executability Gate を参照。

## Input Resolution（入力の特定）

必ず最初に対象を1つに確定させてから設計に入る。

1. **レポート特定**: 引数にパスがあればそれを使う。無ければ Bash で最新を選ぶ。
   `ls -t research/search-topic-*.md 2>/dev/null | head -1`
2. **レポート読込**: 特定したファイルを Read し、候補表・上位候補の詳細・
   「最初に試すべき1本」を把握する。
3. **テーマ選択**: 引数の番号/キーワードに一致するテーマ → 無ければ「最初に試すべき1本」
   → それも無ければ候補表の優先度1位を採用する。
4. **フォールバック**: レポートが1件も無い場合、引数で直接テーマが渡されていればそれで進める。
   テーマも無い場合は「対象レポート/テーマが見つからない。先に `/search-topic` を実行するか、
   `テーマ: ...` を引数で渡してください」と標準出力に表示して**そこで終了する**（推測で捏造しない）。

## Research Requirement

`search-topic` と違い深いトレンド調査は不要。ただし**タスクを実行可能な粒度にするための
裏取り**は必ず行う。学習済み知識だけで手順を書かず、直近の一次情報で具体化する。

* レポート内の参考リンク、および対象技術の**公式ドキュメント/チュートリアル**を
  `WebFetch` で確認する（目安: 1〜2件）。
* 確認すること: インストール手順・**現行バージョン**・前提ランタイム・**無料枠/コスト条件**・
  クイックスタートの有無。
* 既知の**詰まりポイント**を1件以上調べる（公式のTroubleshooting、GitHub Issues、
  「試してみた」系記事、`reference/task-design.md` の観点を使う）。
* 手順やコマンドは、確認できた一次情報に基づいて書く。確認できない箇所は
  「要確認」と明記し、断定しない。

## Workflow

### Step 1: 入力を確定する
上記「Input Resolution」に従い、対象レポート・対象テーマを1つに確定する。
採用した前提（レポートパス／テーマ／時間／スキルレベル）を控えておく。

### Step 2: テーマを分解する
レポートから対象テーマの 概要 / タイトル案 / 検証でやること / 検証手順 /
記事構成案 / 参考リンク / 注意点 を抽出する。ここが実践タスクの骨子になる。

### Step 3: 技術を裏取りする
「Research Requirement」に従い、公式ドキュメント等を `WebFetch` で確認し、
インストール手順・バージョン・無料枠・既知の詰まりポイントを最新化・具体化する。

### Step 4: 成果物（ゴール）を定義する
このタスクで作る/確かめる**最小の成果物**と「できたと言える完了条件」を明確にする。
成果物は半日〜1日で作れる粒度にする。ここで「実行環境の制約」を満たすか点検し、
反する場合はローカル・無料・認証不要な等価タスクへ置き換える（不可なら次候補へ）。
完了条件がブラウザ表示なら **Playwright のスクリーンショットで判定できる形**にする。
設計指針は `reference/task-design.md` の Deliverable / Executability Gate。

### Step 5: 実践タスクに分解する
準備 → 環境構築 → 実装・検証 → 深掘り・比較 → 振り返り の各フェーズに、
チェックボックスで完了判定できる粒度のタスクを並べる。各タスクに
**目安時間**と**記録すること（経験談の素材）**を付ける。
各タスクは AIエージェント単独で実行できること（CLI / コード / Playwright）を満たし、
人手が必要な手順は入れない。粒度・時間配分・記録項目の基準は `reference/task-design.md`。

### Step 6: 詰まりポイントを予測する
新人が詰まりそうな点を洗い出し、「なぜ起きるか / 最初に試すこと / 記事での活かし方」を
表にする。予測の観点は `reference/task-design.md` の Pitfalls。

### Step 7: 記事への写像を作る
レポートの記事構成案に沿って、「どのタスクの記録がどの見出しになるか」を対応表にする。
経験談として書くときのコツも添える。写像の作り方は `reference/output-template.md`。

### Step 8: 出力を組み立てて保存する
`reference/output-template.md` のテンプレートに沿ってレポートを構成し、後述の Output に従って
**必ずファイルとして保存する**。

## Output

最終成果物は `reference/output-template.md` のMarkdownテンプレートに沿って構成し、
**必ずファイルとして保存する**（非対話実行では会話ログが残らないため）。

1. `date +%Y%m%d-%H%M` で日時を取得する
2. リポジトリ直下に `practice/` ディレクトリが無ければ作成する
3. 対象技術から短いASCIIスラッグを作る（英語技術名を小文字ハイフン化。
   作れない場合は `task`）
4. `practice/practice-<スラッグ>-<日時>.md` としてレポート全文を書き込む
5. 最後に、保存したファイルパス・対象テーマ名・出典レポートパス・想定所要時間の合計を
   標準出力にも短く表示する

## Quality Rules

* 解説の目次ではなく「作業チェックリスト＋記録項目」を作る。各タスクは完了判定できる粒度にする。
* 各タスクに必ず「記録すること（経験談の素材）」を付ける。ログ・エラー・所要時間・気づきを残させる。
* 手順は裏取りした一次情報に基づく。未確認は「要確認」と明記する。
* 半日〜1日・無料/低コスト・ローカル/無料枠で完了できる範囲に収める。
* すべてのタスクをAIエージェント単独（CLI / コード / Playwright）で実行できる形にする。課金キー・人手サインアップ・手動デプロイは入れない。
* 詰まりポイントを事前に予測し、記事での活かし方まで示す。
* タスクと記事の見出しを対応づけ、実行すればそのまま経験談の素材が揃う状態にする。
* 採用した前提（レポート・テーマ・時間）を出力冒頭に明示する。

## Anti-Patterns

* 実行できない抽象的なタスク（「〇〇を理解する」だけ等、完了条件が無い）。
* 記録項目が無く、実行しても経験談の素材が残らない。
* 手順を捏造する／未確認のコマンドやバージョンを断定で書く。
* 半日〜1日で終わらない、課金や重い環境構築が前提のタスク。
* 1タスクに複数技術を詰め込む、成果物が定義されていない。
* うまくいく前提だけで、詰まりどころを一切想定しない。
* 課金APIキー・人手のサインアップ/ログイン・手動デプロイなど、AIエージェント単独では完結できない手順を含める。
