# 候補選定・変換・評価基準

`search-topic` の Step 5（変換）/ Step 6（評価）で使う。

## Candidate Selection Rules

候補にする技術は、以下を満たすものを優先する。

* 最近話題になっている
* 公式ドキュメントやチュートリアルがある
* 無料または低コストで試せる
* ローカル環境または無料枠で検証できる
* 半日〜1日程度で最小検証できる
* 新人エンジニア目線で学びが出せる
* Zenn読者が興味を持ちやすい
* 既存記事と差別化できる
* 小さな成果物を作れる
* 詰まった点も記事の価値にできる
* AIエージェント（Claude Code等）が単独でCLI・コード実行だけで最後まで検証できる
* 完全無料・認証不要・ローカルで完結する（またはトークン不要の公開API）
* ブラウザ操作が要る場合も Playwright で自動化・スクリーンショットできる

避けるもの。

* 大規模な本番環境が必要
* 高額なクラウド費用が発生しやすい
* 専門知識が重すぎる
* 検証に数日以上かかる
* 公式情報が少なすぎる
* セキュリティリスクが高い
* 規約やライセンス面が不明確
* 単なるニュース紹介で終わりそう
* 新人が実践するには難しすぎる
* 課金が必要なAPIキー／クレジットカード登録が前提（用意する人がいない）
* 人手のサインアップ・メール/電話認証・手動OAuthログインが必要
* 外部SaaSダッシュボードでの手動トークン発行や手動デプロイが必要
* GUI専用アプリ・実機・ハードウェア・有料クラウドの本番リソースが必要

## AI Agent Executability（AIエージェント単独実行の足切り）

検証・実践するのはAIエージェント単独で、人手を介さない。人手が要るテーマは**採点前に除外**する。
「ローカルCLI・コード・Playwright だけで最後まで完結できるか？」で判定する。

### OK（採用してよい）

* ローカルで動くライブラリ・CLI・フレームワーク（npm / pip / cargo 等でインストール）
* トークン不要の公開API、ローカルで動くモデル・DB（Docker / SQLite / ローカルLLM 等）
* `wrangler dev` / `supabase start` など**ローカルで完結する範囲**のクラウド系ツール
* Webアプリの表示・操作確認（localhost を Playwright で操作しスクリーンショットで確認）

### NG（除外する）

* 利用に課金APIキー／クレジットカードが必要（OpenAI等の有料キー前提 など）
* 鍵やアカウントの取得に人手のサインアップ・メール/電話認証・手動OAuthが必要
* SaaSダッシュボードでの手動設定・手動デプロイ・手動トークン発行が必要
* CAPTCHAやログイン壁でブラウザ自動化がブロックされる
* 実機・ハードウェア・GUI専用アプリ・有料クラウドの本番環境が必要

### 判定のコツ

* クラウド/BaaS/デプロイ系は「ローカルCLI・Dockerで完結する部分」に限定すれば採用可。
  デプロイやダッシュボード操作を含む検証は除外、またはローカル完結の記事に絞る。
* 「無料枠」でも登録に人手が要るなら不可。**登録・認証が不要か**で判断する。
* NGに1つでも該当したら候補から外す。迷ったら「人が一度でも手を動かす必要があるか」で切る。

## Past Article De-duplication（過去記事との重複除外）

Step 6 の足切り②で使う。`articles/` にある自分の過去記事と**実質同じ内容**の候補は除外する。
「同じ技術 × 同じ検証内容 × 同じ条件」なら焼き直しとみなす。判定は Step 1 で読んだ
既出テーマ一覧（title / topics / 検証内容）との照合で行う。

### 重複とみなす（除外）

* 対象技術が同じで、検証する内容・切り口もほぼ同じ
* バージョンも検証条件（対象アプリ・比較対象・OS/ランタイム等）も過去記事とほぼ同じ

### 重複とみなさない（採用可）

以下のいずれかで過去記事と明確に差がつくなら採用してよい。記事側で違いを明示すること。

* **バージョン違い**: 対象ツールのメジャー/マイナー更新後に再検証する（例: Vite 8 → Vite 9）
* **検証条件違い**: 比較対象・対象アプリ・データ規模・OS/ランタイムを変える
* **切り口違い**: 同じ技術でも別テーマ（例: ビルド速度 → プラグイン移行/設定移行/ハマり所）

### 例（現状の過去記事基準）

| 候補 | 判定 | 理由 |
|------|------|------|
| Vite 8/Rolldown のビルド時間を測ってみた | ❌ 除外 | `vite8-rolldown-build-benchmark-log` と実質同一 |
| Vite 9 が出たので同じアプリで再計測してみた | ⭕ 採用可 | バージョン違い（差分を明示） |
| Rolldown と Turbopack のビルド速度を比較してみた | ⭕ 採用可 | 比較対象＝検証条件が違う |
| Vite 8 移行で壊れた設定・プラグインを直した記録 | ⭕ 採用可 | 切り口が違う（速度ではなく移行ハマり所） |
| ZennとGitHubを連携して自動公開する | ❌ 除外 | `zenn-github-integration-auto-publish` と実質同一 |

## Topic Conversion

調査した技術は、そのまま出さず、Zenn記事として書ける形に変換する。

### Bad

* Bun
* Cloudflare Workers
* AIエージェント
* Drizzle ORM
* Playwright
* Supabase

### Good

* Bunを初めて触って、Node.jsとの違いを小さなAPIで試してみた
* Cloudflare Workersで新人が初めてサーバーレスAPIを作ってみた
* AIコーディングツールを使ってTodoアプリの実装をどこまで任せられるか試してみた
* Prismaしか知らない新人がDrizzle ORMを触ってみた
* Playwrightで初めてE2Eテストを書いてみた
* Supabase AuthをNext.jsアプリに組み込んでみた

## Evaluation Criteria

各テーマ候補を以下の観点で5段階評価する。

### Trend Strength

現在の話題性。

* 5: 直近で非常に話題になっている
* 4: 複数の技術コミュニティで注目されている
* 3: 一定の注目がある
* 2: ややニッチ
* 1: 話題性が弱い

### Beginner Feasibility

新人エンジニアが検証できるか。

* 5: 公式チュートリアルだけで試せる
* 4: 少し調べれば試せる
* 3: 基礎知識があれば試せる
* 2: ある程度の実務経験が必要
* 1: 新人には重すぎる

### Article Potential

Zenn記事として成立しやすいか。

* 5: 検証ログとして非常に書きやすい
* 4: タイトルと構成が作りやすい
* 3: 工夫すれば記事にできる
* 2: 内容が薄くなりやすい
* 1: 記事化しづらい

### Practical Value

読者にとって実用性があるか。

* 5: 実務や個人開発にすぐ役立つ
* 4: キャッチアップ価値が高い
* 3: 学習価値がある
* 2: 興味本位に近い
* 1: 実用性が弱い

### Verification Cost

検証コストが適切か。

* 5: 1〜2時間で試せる
* 4: 半日で試せる
* 3: 1日あれば試せる
* 2: 数日かかる
* 1: 検証が重すぎる

### Differentiation

既存記事との差別化ができるか。

* 5: 新人目線・比較・失敗ログで差別化しやすい
* 4: 検証内容を工夫すれば差別化できる
* 3: 多少の差別化は可能
* 2: 既存記事と似やすい
* 1: 焼き直しになりやすい
