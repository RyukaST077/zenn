# Zenn Trend Topic Researcher

> このSkillはユーザーが `/search-topic` で明示的に起動する前提。自動起動はしない。
> **非対話（headless）実行を前提とする**（例: `claude -p "/search-topic ..."`）。
> 途中でユーザーに確認・質問しない。`AskUserQuestion` は使わず、情報が不足する場合は
> 常に下記のデフォルト前提を採用してそのまま最後まで進める。
> 詳細な参照データは `reference/` に分離してある。各ステップで必要になったら該当ファイルを読む。

## Goal

国内外の技術サイトを調査し、トレンド技術・最新技術の中から、新人エンジニアが実際に
検証・実践してZenn記事にできるテーマ候補を探す。筆者はまだ対象技術を深く実践していない
状態を想定し、専門家の解説ではなく「試してみた / 入門 / 検証ログ / 詰まった点まとめ」に
つながるテーマを優先する。

最終成果物:

* 調査した国内外の情報源
* 見つけたトレンド技術・最新技術
* Zenn記事テーマ候補（評価付き）
* 上位候補の検証プランと記事構成案
* 最初に取り組むべき1本

## 起動方法と引数

`/search-topic` で起動。以下を引数（自然言語）で渡せる。省略可。

* 興味のある技術領域（例: フロントエンド, AI, インフラ, 個人開発）
* 現在使える技術（例: TypeScript, React, Python, Docker）
* スキルレベル（例: 新人, 実務1年目, 独学中）
* 記事にしたい方向性（例: 試してみた, 比較, 入門, 失敗談）
* 検証に使える時間（例: 1〜2時間, 半日, 1日）
* 避けたいテーマ（例: 高額なクラウド費用, 数学重め, 大規模インフラ）

### 引数が無い/不足のときのデフォルト前提

* 想定筆者: Web系の新人エンジニア
* 想定読者: 新人〜実務2年目
* 対象領域: Web開発, AI開発支援, 開発ツール, 個人開発向け技術
* 検証時間: 半日〜1日
* 記事タイプ: 「試してみた」「初めて触ってみた」「詰まった点をまとめた」

不足情報があっても止まらず、上記デフォルトを明示したうえで調査を進める。
ユーザーへの追加質問は行わない（非対話実行のため応答を待てない）。

## Assumptions

* 筆者は新人〜実務経験が浅いエンジニア。対象技術の深い経験はまだない。
* これから調査・検証して記事を書く。断定しすぎず学習過程と検証結果を中心に書く。
* 半日〜1日で検証でき、無料枠・ローカル環境・個人開発レベルで試せるテーマを優先する。

## 実行環境の制約（重要 / 足切り条件）

このリサーチでは「検証・実践するのは**AIエージェント（Claude Code等）単独**で、人手を介さない」
ことを前提とする。人がいないと進められないテーマは**候補から除外する（足切り）**。
「AIエージェントが CLI・コード実行・Playwright だけで最後まで完結できるか？」で判定する。

必須条件（すべて満たすもののみ候補にする）:

* AIエージェントが CLI・コード実行・ファイル操作だけで最後まで検証できる
* 完全無料・**認証不要**・ローカルで完結する（またはトークン不要の公開API）
* ブラウザ操作や画面確認が必要でも、**Playwright で自動操作・スクリーンショット**できる範囲

除外するもの（1つでも該当したら候補にしない）:

* **課金が必要なAPIキー**やクレジットカード登録が前提（用意する人がいない）
* 人手のサインアップ／メール・電話認証／手動OAuthログイン／ダッシュボードでの手動トークン発行
* 手動デプロイ・手動UI操作・実機・GUI専用アプリ・有料クラウドの本番リソースが必要
* CAPTCHA等でブラウザ自動化がブロックされる

クラウド/BaaS/デプロイ系は `wrangler dev` や `supabase start` のように
**ローカルCLI・Dockerで完結する範囲**に限れば採用可。判定基準の OK/NG 例は
`reference/evaluation.md` の AI Agent Executability を参照。

## Research Requirement

必ず `WebSearch` ツールで実際に検索し、`WebFetch` で記事本文を確認したうえで調査する。
学習済みの知識だけで済ませず、直近の一次情報にあたること。
国内・海外の両方の情報源を調査する。目安:

* 国内サイト: 3件以上 / 海外サイト: 5件以上
* 公式情報: 2件以上 / コミュニティ・記事系: 3件以上
* GitHubやリリースノートなどの一次情報: 1件以上

直近1〜3か月の情報を優先。古い情報を使う場合は現在も有効か確認する。

→ 具体的な情報源リストと検索クエリ例: `reference/sources.md`
→ 調査カテゴリ（AI開発ツール/フロント/バックエンド 他7分類）: `reference/categories.md`

## Workflow

### Step 1: 調査範囲を決める
まず `articles/*.md` の Front Matter（title / topics）と冒頭を読み、**すでに書いた記事のテーマ一覧**を
把握しておく（Step 6 の重複除外で使う）。記事が無ければスキップ。
ユーザー指定領域を優先。無指定なら AI開発ツール → Webフロント → TS/JS周辺 →
個人開発向けBaaS → 開発効率化 → テスト・CI → サーバーレス・デプロイ の順。
カテゴリ定義は `reference/categories.md`。

### Step 2: 国内サイトを調査する
`reference/sources.md` の Domestic Search Examples を元に `WebSearch` を複数回実行し、
気になる記事は `WebFetch` で本文を確認する。最近見かける技術・記事化テーマ・注目記事・
キャッチアップ需要を洗い出す。情報源は `reference/sources.md`。

### Step 3: 海外サイトを調査する
`reference/sources.md` の Overseas Search Examples を元に `WebSearch` を複数回実行し、
公式リリース・GitHubの伸び・Hacker News/Reddit・Product Hunt・海外ブログの反復テーマ・
直近の新機能/新ツールを `WebFetch` で確認する。情報源は `reference/sources.md`。

### Step 4: トレンド技術を抽出する
候補技術を10〜20件抽出し、各技術について 技術名 / カテゴリ / 概要 / 話題性の理由 /
公式情報 / 国内言及 / 海外言及 / 新人が試せる範囲 / 記事化しやすさ /
**AIエージェント単独で実行可能か** を整理する。

### Step 5: Zenn記事テーマに変換する
技術名のまま出さず、記事として書ける形に変換する（試してみた/初めて触った/
チュートリアル/小さなアプリ/比較/詰まった点/新人目線）。
変換の良い例・悪い例は `reference/evaluation.md` の Topic Conversion。

### Step 6: 候補を評価する
2段の足切りを適用してから採点する。
**足切り①**: 「実行環境の制約」より、**AIエージェント単独で完結できない候補を除外**
（`reference/evaluation.md` の AI Agent Executability）。
**足切り②**: Step 1 の既出テーマ一覧と照合し、**過去記事と実質同じ内容の候補を除外**する。
ただしバージョン違い・検証条件違い・切り口違いは重複とみなさず採用可
（`reference/evaluation.md` の Past Article De-duplication）。
残った候補を6軸
（トレンド性 / 新人が検証できるか / 記事化しやすさ / 実用性 / 検証コスト / 差別化）で
5段階採点し、合計点で優先度順に並べる。採点基準の詳細と選定ルールは `reference/evaluation.md`。

### Step 7: 上位候補を深掘りする
上位3〜5件について タイトル案 / なぜ今よいか / 想定読者 / 検証でやること /
検証手順 / 記事構成案 / 参考リンク / 差別化ポイント / 注意点 を作る。
タイトル案のパターンは `reference/output-template.md` の Title Patterns。

### Step 8: 最初に試す1本を選ぶ
最初に取り組む1本を選び、トレンド性 / 新人でも検証できる理由 / Zenn読者に刺さる理由 /
半日〜1日で検証できるか / 記事の完成イメージ を明示する。

## Output

最終レポートは `reference/output-template.md` のMarkdownテンプレートに沿って構成し、
**必ずファイルとして保存する**（非対話実行では会話ログが残らないため）。

1. `date +%Y%m%d-%H%M` で日時を取得する
2. リポジトリ直下に `research/` ディレクトリが無ければ作成する
3. `research/search-topic-<日時>.md` としてレポート全文を書き込む
4. 最後に、保存したファイルパスと候補一覧の要約（上位3件のテーマ名のみ）を
   標準出力にも短く表示する

## Quality Rules

* 国内外の両方の情報源を確認し、公式・一次情報と直近情報を優先する。
* 話題性の根拠を示す。技術名でなく記事テーマに変換する。
* 新人が半日〜1日・無料/低コストで検証できる粒度にする。
* 断定しすぎず「試してみた/詰まった点/学んだこと」を中心にする。
* 上位候補には必ず検証プランを付け、最後に最初の1本を提案する。
* 既存記事が多いテーマは新人目線・比較・失敗ログで差別化する。
* 自分の過去記事（`articles/`）と実質同じ内容は候補にしない。再訪する場合はバージョン・条件・切り口の違いを明示する。
* AIエージェント単独で完結できるテーマに限定する（課金APIキー・人手サインアップが必要なものは除外。ブラウザ確認はPlaywright前提）。

## Anti-Patterns

* 実践せずニュース/公式要約をまとめるだけ、環境構築だけで終わる。
* 新人には重すぎる/高額/検証に数日/話題性の根拠がない。
* 「完全理解」「徹底解説」など経験値に対し強すぎるタイトル。
* 1記事に複数技術を詰め込む、海外で話題だが日本語読者価値が薄い。
* 課金APIキーや人手のサインアップ/ログイン/デプロイが必要で、AIエージェント単独では検証できない。
* 過去に書いた記事の焼き直し（同じ技術・同じ検証を、バージョンや条件を変えずに再度書く）。
