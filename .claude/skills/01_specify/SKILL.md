# 01_specify

ユーザー提供の資料と対話から、次の2つのテンプレートを **矛盾なく・漏れなく・テスト可能な形で** 埋める対話型オーケストレーション。

| 成果物                     | テンプレート             | 出力先                                          |
| -------------------------- | ------------------------ | ----------------------------------------------- |
| 要件定義書                 | `01_Requirements.md`     | `docs/01_Project_Design/01_Requirements.md`     |
| 業務フロー／業務ルール定義 | `02_Business_Process.md` | `docs/01_Project_Design/02_Business_Process.md` |

## テンプレート解決ルール

1. テンプレートは **常に本スキル同梱の `references/templates/` を使用する**。リポジトリ側に `templates/` がある前提は取らない。
   - 要件定義書: `references/templates/01_Requirements.md`
   - 業務フロー／業務ルール: `references/templates/02_Business_Process.md`
2. 同梱テンプレート（`references/templates/` 本体）は **絶対に編集しない**（読み取り専用）。
3. 書き込み先は常に `docs/01_Project_Design/` 配下（ディレクトリが無ければ作成する）。

## 絶対ルール

1. **不明点は必ず質問する。** 重要な不明点（スコープ/意思決定者/例外/権限/受入基準/外部連携など）が残る場合、推測で確定しない
2. 重要度が低く前に進められるものは **仮置き（ASSUMPTION）として明示** し、ユーザー承認待ちにする。すべての仮置きは 01_Requirements の ASM に記録する
3. **要件は「何を(What)」** を書く。実装方式/画面レイアウト/技術設計(How)は書かない（必要なら「検討事項」または「制約/前提」に分類）
4. **テスト可能性を最優先**：FRは必ずAC（受入条件）と紐付ける。曖昧語（適切に、迅速に、柔軟に 等）を放置しない
5. **トレーサビリティ**：BP/BRL/EX → BREQ → FR → AC →（必要ならNFR/CON）を辿れるID設計で管理する

## ID体系（フェーズ0で確定・固定）

```
BP-001（業務プロセス） / ACT-001（アクティビティ） / BRL-001（業務ルール） / EX-001（業務例外）
BREQ-001（ビジネス要求） / FR-001（機能要件） / NFR-001（非機能要件） / CON-001（制約） / AC-001（受入条件）
ASM-001（前提） / DEP-001（依存） / RSK-001（リスク） / SC-001（データスコープ）
```

## ワークフロー（フェーズ制・ゲート方式）

**作業開始前に必ず `references/phases.md` を読む**こと。各フェーズの詳細手順・質問例・ゲートのチェック項目が記載されている。

| フェーズ | 内容                                         | 主対象              | ゲート                               |
| -------- | -------------------------------------------- | ------------------- | ------------------------------------ |
| 0        | セットアップ（ID体系・用語集・スコープ確認） | 両方                | スコープ/意思決定者/KPI の合意       |
| 1        | To-Be業務プロセス（BP一覧・標準フロー）      | 02_Business_Process | BP一覧と主要フローの合意             |
| 2        | 例外・分岐・取消/差戻し・通知                | 02_Business_Process | 口頭運用の例外が残っていないか       |
| 3        | 権限（ロール×操作×データ範囲）               | 02_Business_Process | ロール/権限の抜け・矛盾なし          |
| 4        | 業務ルール（BRL）を決定表で固める            | 02_Business_Process | 判断基準がテスト可能な粒度か         |
| 5        | BREQ → FR → AC                               | 01_Requirements     | BREQ/FR/ACの対応が取れているか       |
| 6        | NFR・CON・ASM/DEP/RSK                        | 01_Requirements     | NFR/CONが測定可能か                  |
| 7        | トレーサビリティ表・最終レビュー             | 両方                | 重要論点が未確定のまま残っていないか |

各フェーズ末尾の **確認ゲート** では、対話セッションなら AskUserQuestion で選択肢付きの確認を行い、合意を得てから次フェーズへ進む。

## 初回応答の構成（必須）

最初の回答は必ず次を含める：

1. **(a) 理解したことの要約**（提供資料の文書分析結果）
2. **(b) 不足情報の質問**（優先度付き。初回質問テンプレは `references/phases.md` 冒頭を使用）
3. **(c) 次に作る成果物（範囲）**

## 毎回の回答フォーマット（必須）

1. **今回更新した成果物**（どの章/どのIDを追加・更新したか）
2. **内容（ドラフト）**（テンプレ章立てに沿って）
3. **未確定事項**（ASSUMPTION / DEP / RISK の一覧）
4. **質問**（最大10件、優先度付き）
5. **次回の作業予定**（次に埋める章/作る表）

質問の出し方・曖昧語の変換・要件文の書き方は `references/writing-guide.md` を参照（フェーズ5以降に着手する前に必ず読む）。

## トレーサビリティ検証（フェーズ5・7で実行）

`scripts/check_traceability.py` で機械チェックを行う：

```bash
python3 .claude/skills/01_specify/scripts/check_traceability.py \
  docs/01_Project_Design/01_Requirements.md \
  docs/01_Project_Design/02_Business_Process.md
```

> Windows / PowerShell では `python3` が無い場合があります。`python` または `py` に読み替えてください（例: `py .claude/skills/01_specify/scripts/check_traceability.py ...`）。

検出内容：ID種別ごとの件数、**ACが1つも紐付かないFR**、曖昧語の残存箇所（ファイル:行番号）。指摘があれば修正案を提示してからゲートに進む。

## やってはいけないこと

- 推測で要件を「確定」させる（質問するか ASSUMPTION にする）
- How（実装方式・画面レイアウト・技術選定）を要件として書く
- ACの無いFRを確定扱いにする
- 曖昧語を含む要件文・NFRをそのまま残す
- 同梱テンプレート（`references/templates/`）本体の編集
- フェーズを飛ばして一気に全文書を生成する（ゲートを必ず通す）
- コミット・PR・プッシュ等のリモート操作（ユーザ依頼が無い限り）

## 最終目的

「ユーザーと合意できる要件定義書」を作ること。推測で確定させず、質問と仮置き管理で確実に前進し、トレーサビリティと受入条件で“合意可能な形”に落とし込む。
