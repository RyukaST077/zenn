# 質問・記述ガイド

質問の出し方、不明点の扱い、曖昧語の変換、要件文・AC・NFRの書き方。

---

## 質問の出し方（重要）

- 質問は **一度に最大10件**。各質問に以下を付ける：
  - **目的**（なぜ必要か）
  - **決めないと影響する成果物**（BP/BRL/FR/AC/NFR等のID・章）
  - 選択肢が出せる場合は **候補を提示**（A/B/C）
- 重要度ラベル：**Must / Should / Could**
- 対話セッションでは AskUserQuestion を活用し、選択肢付き質問は1ターンにまとめる

### 質問の記述例

> **Q1 [Must]** 承認が却下された注文は、申請者が修正して再申請できますか？それとも新規起票が必要ですか？
> - 目的：差戻しフローの有無を確定するため
> - 影響：BP-003（承認プロセス）、EX-002、FR-012
> - 候補：A) 修正して再申請可 / B) 新規起票のみ / C) 承認者が修正依頼コメントを付けて差戻し

---

## 不明点の扱い

| 不明点の重要度 | 扱い |
|---|---|
| 重要（スコープ/意思決定者/例外/権限/受入基準/外部連携） | **質問して止める（確定待ち）** |
| 重要度が低く前に進められる | **仮置き（ASSUMPTION）** として明示し、後で承認を取る |

- すべての仮置きは 01_Requirements の **ASM** に記録する（ID採番・承認状態を管理）
- 仮置きには「この仮置きが外れた場合に影響するID」を必ず添える

## 曖昧語の検出と変換

以下の語が要件文・AC・NFRに現れたら、**測れる表現** に変換するために質問する（放置禁止）：

> 適切に / 迅速に / 柔軟に / ユーザーフレンドリー / 使いやすい / わかりやすい / 高速に / 簡単に / 十分な / 必要に応じて / できるだけ / なるべく / 速やかに / 効率的に / 安定して / 直感的に

### 変換例

| 曖昧な表現 | 測れる表現への変換例 |
|---|---|
| 検索結果を迅速に表示する | 検索結果を **P95で2秒以内** に表示する（対象データ100万件、同時接続50） |
| 適切な権限を持つユーザーのみ承認できる | **承認者ロール（ROLE-APPROVER）かつ申請者と異なるユーザー** のみ承認できる |
| エラーをわかりやすく通知する | 入力エラー時、**対象項目の直下にエラー理由と修正方法** を表示する |
| 必要に応じて再送する | 送信失敗時、**最大3回・指数バックオフ（1/2/4分）** で自動再送し、3回失敗で運用者へ通知する |

## 要件文（FR）の書き方

- 要件文は「**主語（システム/ユーザー/外部）＋動詞＋目的語＋条件**」で書く
- **1つの要件に複数の振る舞いを詰め込まない**（1要件=1振る舞い。「および」「また」で繋がったら分割を検討）
- 正常系だけでなく **例外時の振る舞い** も含める（該当する EX へリンク）

### 良い例 / 悪い例

| | 例 |
|---|---|
| ❌ | システムは注文を適切に処理し、必要に応じて承認者へ通知する |
| ✅ | FR-010：システムは、注文金額が10万円以上の場合（BRL-003）、注文を「承認待ち」状態にし、承認者ロールの全ユーザーへメール通知する |
| ✅ | FR-011：システムは、承認者が不在設定中の場合（EX-004）、代理承認者へ通知を転送する |

## AC（受入条件）の書き方

- ACはテストで検証できる形（**入力/操作/期待結果**）にする
- 推奨：**Given/When/Then 形式** または箇条書きの明確な合否条件
- 各FRは最低1つ以上のACを持つ。境界値・例外系のACも検討する

### 記述例

```
AC-021（FR-010 に対応）
Given: 承認者ロールのユーザーが2名登録されている
When:  金額100,000円の注文が確定される
Then:  注文ステータスが「承認待ち」になり、承認者2名へメールが送信される

AC-022（FR-010 の境界値）
Given: 同上
When:  金額99,999円の注文が確定される
Then:  注文ステータスが「処理中」になり、承認通知は送信されない
```

## NFR（非機能要件）の書き方

NFRは必ず4点セットで記述する：

1. **品質特性**（性能/可用性/セキュリティ/保守性 等）
2. **測定指標**（応答時間P95、稼働率、RTO/RPO 等）
3. **測定条件**（データ量、同時接続数、対象画面/API、測定環境）
4. **合否基準**（閾値）

### 記述例

> NFR-003（性能）：注文検索APIは、データ100万件・同時接続50の条件下で、応答時間 **P95 ≤ 2秒** を満たすこと。測定は本番相当環境で実施する。

## What / How の切り分け

| 書く（What） | 書かない（How → 設計書へ委譲） |
|---|---|
| 業務上の振る舞い・ルール・例外 | 実装方式・フレームワーク選定 |
| 受入に直結するUI制約（法令・ブランドガイド等は CON へ） | 画面レイアウト詳細 |
| 外部連携の相手先・データ・タイミング（業務要件として） | API仕様の詳細（→ IF設計書） |
| データに関する業務ルール（保持期間・権威マスタ等） | テーブル物理定義（→ データ設計書） |

How に踏み込んだ情報をユーザから受け取った場合は、捨てずに「検討事項」または「制約/前提（CON/ASM）」に分類して記録する。
