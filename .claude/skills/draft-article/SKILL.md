# Zenn 記事ドラフト ジェネレーター（実践ログ → 記事）

> このSkillはユーザーが `/draft-article` で明示的に起動する前提。自動起動はしない。
> **非対話（headless）実行を前提とする**（例: `claude -p "/draft-article ..."`）。
> 途中でユーザーに確認・質問しない。`AskUserQuestion` は使わず、情報が不足する場合は
> 常に下記のデフォルト前提を採用してそのまま最後まで進める。
> 詳細な参照データは `reference/` に分離してある。各ステップで必要になったら該当ファイルを読む。

## パイプライン上の位置づけ

このSkillは3〜4段パイプラインの**最終段（記事化）**にあたる。

```
/search-topic  → research/search-topic-*.md   （テーマ候補レポート）
/plan-practice → practice/practice-*.md        （実践タスクのチェックリスト）
/run-practice  → logs/run-*/execution-log.md   （実装・検証の結果＝記事の素材）
/draft-article → articles/<slug>.md            （Zenn記事ドラフト）  ← これ
```

## Goal

`run-practice` が出力した**実践ログ（execution-log.md）をもとに、Zennの記事ドラフトを組み立てる**。
ログの「記事への写像」に沿って見出しを立て、そこに記録された一次情報（実行コマンド・エラー全文・
所要時間・スクリーンショット・比較・気づき）を流し込む。目的は、**実行済みの素材を、Zennに
そのまま preview / 公開できる記事ドラフトへ変換する**こと。

このSkillは**新しい実行や検証はしない**（`run-practice` の役割）。ログに記録済みの事実だけを
使って書く。ログに無い結果・数値・コードは**創作しない**。素材が足りない見出しは
`<!-- 要素材: ... -->` を残し、空欄のまま人間の追記に委ねる。

記事は「新人が試した経験談」のトーンで書く。うまくいった点だけでなく、詰まった点と解決過程を
中心に据え、断定しすぎない。

最終成果物:

* `articles/<slug>.md`（Zenn Front Matter ＋ 本文、**published: false** のドラフト）
* `images/<slug>/` に配置したスクリーンショットと、本文からの参照
* 素材が不足した箇所を示す `<!-- 要素材 -->` マーカー（捏造の代わり）

## 起動方法と引数

`/draft-article` で起動。以下を引数（自然言語）で渡せる。すべて省略可。

* 対象ログ: `logs/run-*/execution-log.md` のパス（無指定なら最新を自動選択）
* 記事タイプ / トーン（例: 試してみた, 詰まった点まとめ, 検証ログ, 比較）
* slug（無指定ならログのスラッグを引き継ぎ、具体化する）
* title / emoji / topics（無指定ならログの「記事タイトル案」と対象技術から決める）
* published の希望（**既定は false 固定**。true 指定でも警告し、ドラフト維持を促す。後述）

### 引数が無い/不足のときのデフォルト前提

* 対象ログ: `logs/` 内で最も新しい `run-*/execution-log.md`
* 記事タイプ: ログの「実行の前提」にある記事の方向性。無ければ「試してみた / 検証ログ」
* slug: ログのスラッグ（`run-<slug>-*`）を引き継ぎ、汎用的すぎる場合は技術名＋切り口で具体化
* emoji: 対象技術に合う絵文字1つ
* topics: 対象技術・カテゴリから最大5個（英小文字）
* published: **false**（ドラフト）

引数の値はログ内の情報より優先する。不足情報があっても止まらず、採用した前提を本文冒頭の
コメント（`<!-- 前提: ... -->`）に残したうえで進める。ユーザーへの追加質問は行わない
（非対話実行のため応答を待てない）。

## Assumptions

* 入力は `run-practice` の `execution-log.md`（記録済みの一次情報）。本Skillは実行・検証をしない。
* 記事は新人〜実務経験の浅いエンジニアの経験談。専門家として断定しすぎない。
* 本文はログにある事実のみで構成する。ログに無い成功・数値・コードは書かない。
* スクリーンショットはログの `screenshots/` にある実物を使う（存在しない画像を参照しない）。

## 公開安全とZenn制約（重要 / 必須）

このリポジトリは **GitHub連携で `main` に push すると自動公開**される。事故公開を避けるため
次を必ず守る。

* **`published: false` のドラフトで保存する**。公開判断は人間が行い、準備ができてから
  `published: true` に変えて push する（本Skillは true にしない。true 指定時も false のまま
  保存し、標準出力で公開手順を案内する）。
* **slug は Zennサイト全体でグローバルに一意**である必要がある。汎用的すぎる slug
  （`getting-started-*`, `hello-world`, `introduction` 等）は他ユーザーの記事と衝突しやすい。
  技術名＋切り口で具体化し、ローカルの `articles/` に同名が無いか確認する
  （参考: `knowledge/2026-07-01-zenn-slug-already-used.md`）。
* **画像は `images/<slug>/` に置き、本文から `/images/<slug>/...` で参照**する（Zenn仕様）。

Front Matter・slug 規約・Markdown拡張・公開フローの詳細は `reference/zenn-format.md` を参照。

## Input Resolution（入力の特定）

必ず最初に対象ログを1つに確定させてから記事化に入る。

1. **ログ特定**: 引数にパスがあればそれを使う。無ければ Bash で最新を選ぶ。
   `ls -t logs/run-*/execution-log.md 2>/dev/null | head -1`
2. **ログ読込**: 特定したファイルを Read し、「実行の前提」「結果サマリー」「完了条件の検証」
   「タスク実行ログ」「詰まった点と解決過程」「スクリーンショット一覧」「記事への写像」
   「再現性メモ」を把握する。出典 `practice/` や `research/` を辿れれば、タイトル案・
   記事構成案の補強に使う。
3. **フォールバック**: `execution-log.md` が1件も無い場合、
   「対象の実践ログが見つからない。先に `/run-practice` を実行して logs/run-*/execution-log.md
   を作ってください」と標準出力に表示して**そこで終了する**（記事を捏造しない）。

## Workflow

### Step 1: 入力を確定する
上記「Input Resolution」に従い、対象ログを1つに確定する。採用した前提
（ログパス／記事タイプ／slug／published=false）を控えておく。

### Step 2: ログを解析して素材を棚卸しする
ログの「記事への写像」を軸に、各見出しに対応する素材（どのタスクログ・どのスクショ・
どのコード抜粋・どの比較）を対応づける。`workspace/` のコードから記事に貼る抜粋を選ぶ。
素材が無い見出しは「要素材」として印を付けておく。

### Step 3: 記事メタデータを決める
`reference/zenn-format.md` に従い title / slug / emoji / type / topics / published を決める。

* slug: ログのスラッグを引き継ぎ、汎用的なら技術名＋切り口で具体化する。
  `ls articles/` でローカル重複が無いか確認する（あれば別名にする）。
* published: **false 固定**（引数で true が来ても false にする）。
* title はログの「記事タイトル案」から、経験談トーンで1つ選ぶ／整える。
  emoji は1つ、topics は英小文字で最大5個。

### Step 4: 画像を配置する
`images/<slug>/` を作成し、ログの `screenshots/` から本文で使うスクショをコピーする。
本文参照用に「元ファイル → `/images/<slug>/NN-*.png`」の対応を作る。存在しない画像は参照しない。

### Step 5: 本文を書く
`reference/article-template.md` の骨子に沿って、ログの「記事への写像」の見出し順に本文を書く。

* 各見出しに、対応するログの記録（実行コマンド・**エラー全文**・所要時間・比較・気づき）を流す。
* コードは `workspace/` の抜粋を、実行できた形で貼る（言語・ファイル名付きコードブロック）。
* スクショは `/images/<slug>/...` で参照し、何を示す画像か一言添える。
* 「詰まった点」は解決過程まで具体的に書く（記事の核）。再現性のため OS・ランタイム・
  バージョンを明記し、公式ドキュメントへのリンクを入れる。
* トーンは経験談（新人が試した範囲・断定しすぎない）。素材が無い見出しは
  `<!-- 要素材: ... -->` を残し、埋めない／創作しない。

### Step 6: Front Matter を付けて保存する
Front Matter（title / emoji / type / topics / **published: false**）を先頭に付け、
後述の Output に従って `articles/<slug>.md` に保存する。preview と公開手順を案内する。

## Output

**必ずファイルとして保存する**（非対話実行では会話ログが残らないため）。

1. `images/<slug>/` を作成し、使用するスクショをコピーする
2. `articles/<slug>.md` に Front Matter ＋ 本文を書き込む（**published: false**）
3. 最後に、以下を標準出力にも短く表示する:
   * 保存した記事ファイルのパスと slug
   * `published: false`（ドラフト）である旨
   * 使用したスクショ枚数 / 残した `要素材` マーカーの数
   * プレビュー方法: `npx zenn preview`
   * 公開手順: 「内容を確認し、準備ができたら Front Matter を `published: true` に変えて
     `git push`。『サイト内で既に使用されています』が出たら slug を具体的な名前に変更
     （`knowledge/2026-07-01-zenn-slug-already-used.md` 参照）」

## Quality Rules

* ログにある一次情報だけで書く。ログに無い成功・数値・コードは創作しない。不足は `要素材` で残す。
* **パイプライン内部メタデータは本文に転記しない。** ログにある次の情報は「AIが単独で
  実行するための足切り条件」であり読者価値ではないので、記事には書かない／持ち込まない:
  「実行者: AIエージェント単独」「完全無料・認証不要・ローカル完結」「課金/クレカ不要」等。
  - これらを記事の売り（「無料で試せます！」）として書かない。新人が実際に手を動かした
    体験として書く。
  - 所要時間は「AI単独の自動実行で0.5h」のように書かない。書くなら人間が試す前提の
    自然な粒度（「ひととおり試すのに数時間」程度）にするか、不自然なら省く。
* **文体のAIっぽさを避ける**（太字乱打・定型テンプレ・伏線予告・対句・全部解決で締める、を禁止）。
  詳細は `reference/article-template.md` の「AIっぽさを避ける」節に従う。
* **published: false 固定**。公開判断は人間に委ねる（このリポジトリは push で自動公開されるため）。
* slug は具体的で一意性が高いものにする。汎用 slug を避け、ローカル重複を確認する。
* エラーは全文を転記する（要約しすぎない）。コードは実行できた形で貼る。
* 経験談トーン（新人が試した範囲・断定しすぎない・詰まった点も書く）を守る。
* スクショは `images/<slug>/` に置いて `/images/<slug>/...` で参照する。存在しない画像を貼らない。
* 再現性（OS・ランタイム・バージョン）を明記し、公式ドキュメントへのリンクを入れる。
* 採用した前提（ログ・slug・published）を本文冒頭コメントか標準出力に明示する。

## Anti-Patterns

* ログに無い結果・数値・コードを創作して「動いた」と書く。
* `published: true` で保存して事故公開する。
* 汎用 slug（`getting-started-*` 等）で衝突リスクを作る。ローカル重複を確認しない。
* スクショを本文に貼らずにブラウザ表示を「確認済み」と書く／存在しない画像を参照する。
* エラーを要約しすぎて再現性を失わせる。公式リンク・バージョンを書かない。
* 対象の実践ログが無いのに記事をでっち上げる。
* ログの内部メタデータ（実行者=AI単独 / 無料・認証不要・課金回避）を記事の価値であるかの
  ように本文へ転記し、新人ペルソナを壊す。
* 太字で要点を強調して回る／「詰まった点」を症状・原因・対処・学びの4点セットに整形する／
  「これがあとで効いてきます」等の伏線予告を張る。
