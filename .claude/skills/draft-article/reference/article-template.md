# 記事ドラフト テンプレート＆執筆ガイド

`draft-article` の Step 5（本文執筆）で使う。実践ログ（execution-log.md）の「記事への写像」を
そのまま見出しに落とし込む骨子と、経験談トーンの書き方。

> **原則**: 本文はログにある一次情報だけで書く。ログに無い成功・数値・コードは創作しない。
> 素材が足りない見出しは `<!-- 要素材: 何が足りないか -->` を残し、空欄のまま人間に委ねる。

## ドラフトの骨子

Front Matter は `reference/zenn-format.md` 参照。本文は次の構成を基本に、ログの
「記事への写像」の見出し列に合わせて調整する。

```md
---
title: "<タイトル案から1つ>"
emoji: "<絵文字1つ>"
type: "tech"
topics: ["<tech1>", "<tech2>"]
published: false
---

<!-- 前提: 出典ログ logs/run-<slug>-<日時>/execution-log.md / 記事タイプ <...> -->

## はじめに

- なぜこの技術を試したか（動機）。出典: ログの「実行の前提」。
- 何を作った / 確かめたか（成果物の一言説明）と結論の予告（達成 / 一部達成）。
- 想定読者（新人〜実務2年目）と、この記事で分かること。

:::message
筆者は<新人 / 実務N年目>で、この技術を試すのは初めてです。実行環境: <OS> / <ランタイム vX>。
:::

## 使ったもの・環境

- 対象技術とバージョン、前提ランタイム（ログの「再現性メモ」から）。
- 完成イメージ（成果物）と「できたと言える完了条件」。

## 環境構築

- 実際に叩いたコマンド（ログのフェーズ2）。ファイル名付きコードブロックで。
- 詰まった点があればここで軽く触れる（詳細は後段）。
- 起動確認のスクショ: `![](/images/<slug>/01-*.png)`

## 実際に作ってみる（本編）

- 成果物を作る中心部分（ログのフェーズ3）。コードは workspace の抜粋を貼る。
- 完了条件の確認: CLI出力 / レスポンス / 画面スクショ `![](/images/<slug>/02-*.png)`。

## 詰まった点と解決（この記事の核）

- ログの「詰まった点と解決過程」表を、詰まった→原因→効いた対処の順で物語る。
- エラーは全文を（長ければ `:::details エラー全文` で折りたたむ）。
- 「最初こう思ったが実は…」の気づきを書くと経験談として刺さる。

## 分かったこと・既存技術との比較

- ログのフェーズ4（深掘り・比較）。既存技術との違い、限界、向き/不向き。

## まとめ

- 結果（達成 / 一部達成 / 未達）と所要時間（見積もり→実測）。
- どんな人に向いているか、次に試したいこと。

## 参考リンク

- 公式ドキュメント / チュートリアル（ログ・出典レポートの参考リンク）。
```

## 見出し ← 素材の対応（写像の使い方）

ログの「記事への写像（実績で埋める）」表の左列＝見出し、中列＝使う記録/スクショ、
右列＝書くこと。この対応どおりに素材を各見出しへ流す。

| 記事の見出し | 主に使うログの記録 |
|---|---|
| はじめに | 実行の前提 / 結果サマリー |
| 使ったもの・環境 | 再現性メモ / 完成イメージ |
| 環境構築 | フェーズ2のタスク実行ログ / スクショ01 |
| 実際に作ってみる | フェーズ3のログ / workspace のコード抜粋 / スクショ02〜 |
| 詰まった点と解決 | 「詰まった点と解決過程」表 / エラー全文 |
| 分かったこと・比較 | フェーズ4のログ |
| まとめ | 結果サマリー（達成・所要時間） |
| 参考リンク | 参考リンク（公式ドキュメント） |

素材が無い見出しは削るか、`<!-- 要素材 -->` を残す。ログに無い見出しは足さない。

## 経験談トーンの書き方

`plan-practice` の「経験談として書くときのコツ」に合わせる。

* 「新人が試した範囲」を明示し、専門家として断定しすぎない。
* うまくいった点だけでなく、詰まった点と解決過程を書く。
* 実行ログ・スクリーンショット・コードを実物で貼る（一次情報）。
* 公式ドキュメントへのリンクを入れる。
* 手順の再現性（バージョン・OS・前提）を明記する。
* タイトルは強すぎる語（完全理解 / 徹底解説）を避け、「試してみた / 詰まった点まとめ /
  検証ログ」の温度感にする。

## やってはいけないこと

* ログに無い結果・数値・コードを創作する（「たぶん動く」を「動いた」と書く等）。
* 完了条件を確認したように書くが、根拠のスクショ/ログを貼らない。
* エラーをきれいに要約しすぎて再現できなくする。
* `published: true` にする（公開は人間の判断。`reference/zenn-format.md` の公開フロー）。
* 素材が無い見出しを埋めるために内容をでっち上げる（`<!-- 要素材 -->` を残す）。
