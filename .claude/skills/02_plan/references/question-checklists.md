# 確認ゲート・質問チェックリスト

各フェーズの確認ゲートで聞くべき質問と、曖昧語の具体化パターン。対話セッションでは AskUserQuestion を使い、選択肢付きで **1ターンにまとめて** 聞く（最大4問/ターン。それ以上は優先度順に分割）。

---

## ゲート0（セットアップ）

| 質問 | 選択肢の例 |
|---|---|
| 技術選定の優先順位は？ | コスト最優先 / 開発スピード最優先 / スケーラビリティ最優先 / バランス型 |
| チームが経験のある技術スタックは？ | （自由記述で確認。言語/FW/クラウドの実務経験年数まで） |
| インフラ方針は？ | AWS / GCP / Azure / オンプレ / 既存インフラに従う |
| 既存インフラ・社内標準の制約は？ | あり（詳細を確認）/ なし / 不明（TBD化） |

## フェーズ1（アーキテクチャ設計）で不足があれば聞く質問

- 想定ユーザー数 / 同時接続数 / ピーク負荷は？（例：通常100req/s、ピーク時1,000req/s）
- 可用性目標（SLA/SLO）は？（例：99.9% = 月間ダウンタイム約43分）
- データ保持期間 / バックアップ要件（RPO/RTO）は？
- 外部連携先の API 仕様 / 認証方式（OAuth2 / APIキー / mTLS等）は？
- データ量の見込みは？（初期件数、年間増加率）
- 多言語・マルチテナント・タイムゾーン対応の要否は？

**ゲート1の確認事項**
- [ ] アーキテクチャスタイル（モノリス/マイクロサービス等）に合意できたか
- [ ] 技術スタック（クライアント/サーバー/DB/インフラ）に合意できたか
- [ ] 各選定が NFR（性能/可用性/拡張性）と整合しているか
- [ ] 重要な選定すべてに ADR があるか

## フェーズ2（基本設計）で不足があれば聞く質問

- In/Out Scope の境界で迷う機能はどちらに倒すか？
- 認証基盤は新規構築か既存（IdP/SSO）連携か？
- 監視・アラートの通知先と運用体制（24/365か営業時間内か）は？
- 障害時の縮退運転の許容範囲は？（参照のみ可 / 全停止許容 等）

**ゲート2の確認事項**
- [ ] 機能構成図がすべての FR をカバーしているか
- [ ] NFR ごとに対応する設計方針（章・節）が存在するか
- [ ] 外部 IF の一覧に漏れがないか（02_Business_Process の EX/IF 言及と突合）

## 曖昧語の具体化パターン

| 曖昧語 | 具体化の問い方 | 落とし所の例 |
|---|---|---|
| 高速に | 何の操作が何秒以内？パーセンタイルは？ | 「検索結果表示は p95 で 2 秒以内」 |
| 安全に | 何から何を守る？準拠基準は？ | 「保存時 AES-256 暗号化、ISMS 準拠」 |
| 使いやすい | 誰がどの操作を何ステップで？ | 「新規登録は 3 画面以内で完了」 |
| 大量の | 件数・サイズ・期間は？ | 「月間 10 万件、1 件最大 5MB」 |
| 柔軟に | 誰が・何を・いつ変更できる？ | 「管理者が画面から承認ルートを変更可能」 |
| 安定稼働 | 稼働率・復旧目標は？ | 「SLO 99.9%、RTO 4 時間、RPO 1 時間」 |

## ADR の書式（03_Architecture に記録）

```markdown
### ADR-001: <決定のタイトル>

- **ステータス**: Proposed / Accepted / Superseded
- **コンテキスト**: なぜこの決定が必要か（要件ID: NFR-xxx, CON-xxx に紐付け）
- **検討した選択肢**: 案A / 案B / 案C（評価軸ごとの比較）
- **決定**: 採用した案と理由
- **結果（トレードオフ）**: この決定で得るもの・失うもの・再検討のトリガー
```

## TBD / ASSUMPTION の書式

```markdown
- **TBD-001**: <未決事項>（決定者: ○○、決定目安: フェーズ△△まで / YYYY-MM）
- **ASM-001**: <仮置き内容>（根拠: ○○、外れた場合の影響: △△）
```
