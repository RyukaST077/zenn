# 目的

実装計画書を入力として、**タスク分解 → 実装 → システム起動 → Playwright による動作確認** までを
一連のフローとして必ず実施する。

本 Skill の絶対条件（スキップ不可）:

1. **タスクリストツールを必ず使う** — TaskCreate / TaskUpdate / TaskList で作業を管理する
2. **変更後は必ずシステムを起動する** — 起動しないまま「完了」と報告しない
3. **動作確認は Playwright で行う** — スクリーンショットを撮り、画像を自分の目で確認する

このSkillが扱わないもの:
- 実装計画書そのものの作成（計画が無ければユーザに作成を促す）
- 本番環境へのデプロイ・リリース作業

# 引数

| 位置 | 意味 | 例 |
|------|------|-----|
| `$1` | 実装計画書への絶対/相対パス | `docs/plan/feature-login.md` |
| `$2` | 計画書のどの範囲が今回の作業対象かの文章（クォートで囲む） | `"フェーズ2のログインAPIとログイン画面まで"` |

呼び出し例:

    /implement docs/plan/feature-login.md "フェーズ2のログインAPIとログイン画面まで"

`$1` または `$2` が空の場合は、Phase 1 冒頭でユーザに質問して補完する。

---

# 全体フロー

```
Phase 1: 引数確認・計画書読込  → $1 / $2 の検証、計画書全文と対象範囲の特定
Phase 2: タスク分解            → TaskCreate でタスク登録（必須）、ユーザに一覧提示
Phase 3: 実装                  → タスクごとに in_progress → 実装 → completed
Phase 4: システム起動          → バックグラウンドで起動、起動成功を確認
Phase 5: Playwright 動作確認   → 画面操作 + スクリーンショット撮影 + 画像目視確認
Phase 6: 完了報告              → タスク最終状態・変更ファイル・確認結果を報告
```

Phase 4〜5 を通過するまで、いかなるタスクも「完了」と報告してはならない。

---

# Phase 1: 引数確認・計画書読込

1. `$1`（計画書パス）の存在を確認し、`Read` で **全文** を読む。
   - 存在しなければ「実装計画書のパスを教えてください」と質問して終了しない限り進まない。
2. `$2`（対象範囲）を計画書の構成（章・フェーズ・タスク番号など）と突き合わせ、
   **今回の作業対象セクションを特定** する。
3. 対象範囲が曖昧（複数解釈できる）場合は **必ず質問** する。推測で範囲を広げない。
4. 対象範囲外のセクションも軽く読み、依存関係（前フェーズの成果物が前提か等）を把握する。
5. プロジェクトの起動方法をこの時点で調べておく:
   - 計画書内の記載 → `CLAUDE.md` → `README.md` → `package.json` / `pom.xml` /
     `build.gradle` / `compose.yaml` / `Makefile` の順で確認
6. ユーザに提示:

```
対象範囲を特定しました:
  📋 計画書: <パス>
  🎯 今回の範囲: <セクション名・タスク一覧の要約>
  🔗 依存: <前提条件があれば> / なし
  🚀 起動方法: <例: docker compose up -d → http://localhost:8080>
```

---

# Phase 2: タスク分解（タスクリストツール必須）

1. タスクリストツールが未ロードなら `ToolSearch` で読み込む:
   `select:TaskCreate,TaskUpdate,TaskList,TaskGet`
2. 対象範囲を実装単位に分解し、**TaskCreate で全タスクを登録** する。
   - 1 タスク = 1 つの検証可能な変更（例:「ログインAPIの実装」「ログイン画面の作成」）
   - 最後に必ず次の 2 タスクを含める（固定）:
     - 「システムを起動して起動確認」
     - 「Playwright で動作確認（スクリーンショット撮影・目視確認）」
   - 依存関係があれば TaskCreate の依存指定（addBlockedBy 等）で表現する
3. `TaskList` の結果をユーザに提示し、過不足がないか一言確認する
   （明確な指示が既にある場合は提示のみで進行してよい）。

> **TodoWrite や頭の中の管理で代替しない。** 必ず TaskCreate / TaskUpdate を使う。

---

# Phase 3: 実装

タスクごとに次のサイクルを回す:

1. `TaskUpdate` で対象タスクを `in_progress` にする（同時に in_progress は原則 1 つ）
2. 関連コードを読み、**周辺コードの流儀（命名・構成・コメント密度）に合わせて** 実装する
3. ビルド/コンパイル/リント/ユニットテストがあれば随時実行し、壊していないことを確認
4. 完了したら `TaskUpdate` で `completed` にする
5. 次のタスクへ

注意:
- 計画書に書かれていない「ついで実装」はしない。気づいた改善点は完了報告で提案として伝える。
- 途中で計画書と実装の矛盾を見つけたら、勝手に計画を変えずユーザに確認する。

---

# Phase 4: システム起動

実装タスクがすべて completed になったら、起動確認タスクを `in_progress` にして:

1. Phase 1 で調べた方法でシステムを **バックグラウンド起動**（`Bash` の `run_in_background: true`）
   - 既に起動中のプロセスがあれば、再起動して最新の変更を反映させる
2. 起動ログを確認し、エラーなく起動したことを確認する
   - DB 等の依存サービスが必要なら先に起動する（compose 等）
3. `curl` などで疎通確認（例: `curl -s -o /dev/null -w "%{http_code}" http://localhost:<port>/`）
4. **起動に失敗したら**: ログから原因を特定して修正 → 再起動。解決するまで Phase 5 に進まない。
   修正内容は該当タスクの作業として記録する。
5. 成功したらタスクを `completed` にする

---

# Phase 5: Playwright 動作確認（スクリーンショット必須）

動作確認タスクを `in_progress` にして:

1. **Playwright の実行手段を決める**:
   - `ToolSearch` で `+playwright` を検索し、Playwright MCP ツールがあればそれを使う（推奨）
   - 無ければ `npx playwright` が使えるか確認し、一時的な確認スクリプト
     （Node スクリプト）を書いて実行する
   - どちらも無ければユーザにセットアップ方法を確認する
2. **確認シナリオを実行する**。計画書の受け入れ条件・今回実装した機能に沿って:
   - 対象画面へ遷移 → 実装した操作を実行 → 期待結果を検証
   - 正常系だけでなく、計画書に明記されたバリデーション・エラー系も確認
3. **スクリーンショットを活用する**:
   - 保存先: `<project-root>/verification/<YYYYMMDD-HHmm>/`（無ければ作成。gitignore 推奨）
   - 撮影ポイント: ①操作前の初期表示 ②操作直後 ③結果表示（最低 3 枚を目安）
   - 撮影した画像を **必ず `Read` で開いて目視確認** する。
     「スクリーンショットを撮った」だけでは確認したことにならない。
     レイアウト崩れ・エラーメッセージ・文字化けがないかを画像から判断する。
4. **NG だった場合**: 該当する実装タスクを再オープン（`TaskUpdate`）して修正し、
   Phase 4（再起動）→ Phase 5 をやり直す。OK になるまで繰り返す。
5. すべて OK ならタスクを `completed` にし、起動したプロセスを停止するか
   起動したままにするかユーザの文脈に合わせる（迷ったら起動したままにして報告）。

---

# Phase 6: 完了報告

```
✅ 実装完了・動作確認済み

📋 計画書: <パス>（範囲: <今回の範囲>）
🗂 タスク: <N>/<N> completed（TaskList 最終状態）

📁 変更ファイル:
  - src/... (新規/修正)
  - ...

🚀 起動: <起動コマンド> → <URL>（現在 起動中/停止済み）

🔍 Playwright 確認結果:
  - <シナリオ1>: OK
  - <シナリオ2>: OK
  📸 スクリーンショット: verification/<YYYYMMDD-HHmm>/ （<枚数>枚）

💡 気づき・提案（あれば）:
  - ...
```

テストが失敗した・確認できなかった項目があれば、**取り繕わずそのまま報告** する。

# 失敗・エッジケース

## 計画書が存在しない / 範囲が特定できない
→ Phase 1 で必ず質問。推測で実装を始めない。

## システムが起動しない
→ ログから原因究明 → 修正 → 再起動を繰り返す。解決不能なら状況・ログ・試したことを報告して指示を仰ぐ。「起動確認は省略しました」は禁止。

## Playwright が使えない（MCP も npx も無い）
→ `npm i -D playwright && npx playwright install chromium` 等のセットアップをユーザに提案。承認を得てからインストールする。代替（curl のみ）で済ませない。

## UI を持たないシステム（API のみ等）
→ Playwright の `request` コンテキストで API を叩いて検証する。レスポンスを確認し、Swagger UI 等の画面があればそのスクリーンショットも撮る。

## 確認の途中で別のバグを発見
→ 今回の範囲内なら修正してタスク追加（TaskCreate）。範囲外なら報告のみ（勝手に直さない）。

# 実行時の心構え

- **タスクリストが正**: 進捗は常に TaskUpdate に反映。ユーザはタスクリストで進捗を追えること。
- **動かして初めて完了**: コードが書けても、起動 + Playwright 確認を通るまで完了ではない。
- **スクリーンショットは証拠**: 撮影 → Read で目視 → 報告に添える、を 1 セットで行う。
- **計画書に忠実**: 範囲外の実装・リファクタは提案に留める。
- **正直に報告**: 失敗・スキップ・未確認は、その旨を明記する。
