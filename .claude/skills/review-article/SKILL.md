# Zenn 記事 公開前レビュー（品質チェック）

> このSkillはユーザーが `/review-article` で明示的に起動する前提。自動起動はしない。
> **非対話（headless）実行を前提とする**（例: `claude -p "/review-article ..."`）。
> 途中でユーザーに確認・質問しない。`AskUserQuestion` は使わず、情報が不足する場合は
> 常に下記のデフォルト前提を採用してそのまま最後まで進める。
> 詳細な参照データは `reference/` に分離してある。各ステップで必要になったら該当ファイルを読む。

## パイプライン上の位置づけ

このSkillはパイプラインの**公開前ゲート（最終チェック）**にあたる。

```
/search-topic  → research/search-topic-*.md
/plan-practice → practice/practice-*.md
/run-practice  → logs/run-*/execution-log.md      （実践ログ＝素材）
/draft-article → articles/<slug>.md (published:false)（記事ドラフト）
/review-article→ logs/review-<slug>-<日時>.md      （公開前レビュー）  ← これ
```

## Goal

`draft-article` が作った記事ドラフトを、Zennに公開する前にレビュー・品質チェックする。
目的は「**安心して `published: true` にして push できるか**」を客観的に判定し、直すべき点を
重大度付きで具体的に示すこと。特に**公開安全**（事故公開・秘密情報の漏れ・slug衝突）と
**事実整合**（実践ログを超えた主張の混入）を最優先で見る。

デフォルトは**レポートのみ（記事本文は書き換えない）**。レビューは非破壊で行う。安全で
機械的な修正は引数で明示されたときだけ適用し、変更内容を必ずレポートに記録する。
**`published: true` には絶対にしない**（公開判断は人間）。

最終成果物:

* レビューレポート `logs/review-<slug>-<日時>.md`
* 公開可否の判定（**公開可 / 要修正 / 公開不可(blocker)**）
* 重大度付きの指摘一覧（blocker / warning / suggestion）と、それぞれの具体的な直し方
* 機械チェック結果（`scripts/check-article.sh` の出力）

## 起動方法と引数

`/review-article` で起動。以下を引数（自然言語）で渡せる。すべて省略可。

* 対象記事: `articles/<slug>.md` のパス（無指定なら最新の `published: false` ドラフト）
* 出典ログ: `logs/run-*/execution-log.md` のパス（無指定なら記事から辿る／最新。事実整合に使う）
* 修正の適用: 「安全な修正も適用して」等（無指定なら**レポートのみ・非破壊**）
* 公開基準の強弱（例: 厳しめ / ゆるめ）

### 引数が無い/不足のときのデフォルト前提

* 対象記事: `articles/` の中で `published: false` かつ最も新しく更新されたドラフト
* 出典ログ: 記事冒頭コメント `<!-- 前提: 出典ログ ... -->` から辿る。無ければ最新の
  `execution-log.md`。それも無ければ「出典不明」として事実整合は警告扱いにする
* 修正: 適用しない（レポートのみ）
* 公開基準: 標準（下記の判定ルール）

不足情報があっても止まらず、採用した前提をレポート冒頭に明示したうえで進める。
ユーザーへの追加質問は行わない（非対話実行のため応答を待てない）。

## Assumptions

* 入力は `draft-article` の記事ドラフト。本Skillはレビュー担当であり、記事の新規執筆や
  コードの実行検証はしない（それぞれ `draft-article` / `run-practice` の役割）。
* 事実性の正は出典ログ（`execution-log.md`）。記事がログを超えて主張していないかを見る。
* 公開判断は人間が行う。本Skillは「公開してよいかの材料」を出すだけ。

## 公開安全（最優先 / blocker 判定）

次のいずれかに該当したら **blocker（公開不可）** として最上位で報告する。

* **`published: true` になっている**（公開前レビューの対象は未公開ドラフトであるべき。
  このリポジトリは push で自動公開されるため事故公開リスク）。
* **秘密情報の漏れ**: APIキー・トークン・パスワード・秘密鍵・接続文字列・内部ホスト名・
  個人を特定するローカルパスなどが本文やログ引用に含まれている。
* **slug 不正**: 文字種・長さ違反、または汎用的すぎて衝突しやすい
  （`getting-started-*` 等）／ローカル `articles/` に重複がある
  （参考: `knowledge/2026-07-01-zenn-slug-already-used.md`）。

これらは機械チェック（`scripts/check-article.sh`）で一次検出し、観点チェックで最終判断する。

## Input Resolution（入力の特定）

必ず最初に対象記事を1つに確定させてからレビューに入る。

1. **記事特定**: 引数にパスがあればそれを使う。無ければ `published: false` のドラフトのうち
   最も新しく更新されたものを選ぶ。
   例: `for f in articles/*.md; do grep -q '^published: *false' "$f" && ls -t "$f"; done | head -1`
   （簡便には `grep -L '^published: *true' articles/*.md` の中から最新更新を選ぶ）
2. **ログ特定**: 記事冒頭の `<!-- 前提: 出典ログ ... -->` から辿る。無ければ
   `ls -t logs/run-*/execution-log.md 2>/dev/null | head -1`。それも無ければ「出典不明」。
3. **フォールバック**: `articles/` にレビュー対象のドラフトが無い場合、
   「レビュー対象のドラフトが見つからない。先に `/draft-article` で articles/<slug>.md を
   作ってください」と標準出力に表示して**そこで終了する**（レビュー結果を捏造しない）。

## Workflow

### Step 1: 入力を確定する
上記「Input Resolution」に従い、対象記事と出典ログを確定する。採用した前提
（記事パス／出典ログ／修正適用の有無）を控えておく。

### Step 2: 機械チェックを実行する
`bash .claude/skills/review-article/scripts/check-article.sh <記事パス>` を実行し、
Front Matter・slug・画像参照の解決・コードフェンス/`:::`の対応・`要素材`マーカー・
秘密情報パターン・誇大表現などの機械的な結果を得る。出力をレポートに引用する。

### Step 3: 事実整合をチェックする（ログ照合）
出典ログ（`execution-log.md`）と記事を突き合わせ、記事の主張・実行コマンド・エラー・結果・
コードが**ログに裏付けられているか**を確認する。ログを超えた成功の断定・数値・存在しない
スクショ参照・創作コードを検出する。`要素材` マーカーの残存も未完成のサインとして拾う。
観点は `reference/review-checklist.md` の Fidelity。

### Step 4: 観点チェックを行う
`reference/review-checklist.md` の各次元（公開安全 / Front Matter / 事実性 / 画像 /
Markdown構造 / 文章品質・トーン / 完成度）を評価する。各指摘に**重大度（blocker /
warning / suggestion）と具体的な直し方（どこを何にするか）**を必ずセットで付ける。

### Step 5: 公開可否を判定する
`reference/output-template.md` の判定ルールに従う。
blocker が1件でもあれば **公開不可**、blocker が無く warning があれば **要修正**、
どちらも無ければ **公開可**。判定の根拠（該当指摘）を明示する。

### Step 6:（任意）安全な修正を適用する
引数で修正適用が指定されたときのみ実施する。対象は**安全で機械的な範囲**に限る
（`published: true` → `false` への差し戻し、末尾空白の除去、明らかに壊れた画像パスの修正、
未使用の前提コメント除去など）。本文の意味を変える書き換えはしない。適用した差分は
レポートの「適用した修正」に列挙する。無指定なら**何も書き換えない**。

### Step 7: レポートを保存する
`reference/output-template.md` のテンプレートに沿って構成し、後述の Output に従って
**必ずファイルとして保存する**。

## Output

**必ずファイルとして保存する**（非対話実行では会話ログが残らないため）。

1. `date +%Y%m%d-%H%M` で日時を取得する
2. リポジトリ直下に `logs/` が無ければ作成する
3. レビュー結果を `logs/review-<slug>-<日時>.md` に書き込む（`<slug>` は対象記事のファイル名から）
4. 最後に、以下を標準出力にも短く表示する:
   * 対象記事のパスと判定（公開可 / 要修正 / 公開不可）
   * blocker / warning / suggestion の件数
   * 最優先で直すべき指摘 上位3件
   * 次のアクション: 「指摘を直して再レビュー → 問題なければ Front Matter を
     `published: true` に変えて `git push` で公開」

## Quality Rules

* 公開安全（`published: false` / slug / 秘密情報の漏れ）を最優先で判定する。
* 事実性は出典ログを正として照合する。ログに無い主張・数値・コード・画像参照は指摘する。
* 指摘は**重大度＋具体的な直し方**をセットで出す。抽象的な「改善しよう」で終わらせない。
* 機械チェック（script）＋観点チェック（判断）の両輪で、再現性と網羅性を両立する。
* 既定は記事を書き換えない（レビュー＝非破壊）。修正適用は明示時のみ・変更をレポートに記録。
* **`published: true` には絶対にしない**（公開は人間の判断）。
* 採用した前提（対象記事・出典ログ・修正適用の有無）をレポート冒頭に明示する。

## Anti-Patterns

* `published: true` を見逃す／自分で公開する。
* 秘密情報の混入を見逃す。個人パス・内部ホスト名を公開可としてしまう。
* ログ照合をせず記事の主張を鵜呑みにする（捏造の混入を見逃す）。
* 「なんとなく良い / 悪い」だけで、どこを何に直すかを書かない。
* レビューと称して記事を勝手に大幅に書き換える（非破壊の原則を破る）。
* 対象ドラフトが無いのにレビュー結果をでっち上げる。
