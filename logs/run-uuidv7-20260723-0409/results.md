# 検証結果まとめ: randomUUIDv7() vs randomUUID()(v4)

- 環境: macOS (Darwin 25.5.0, arm64) / Node.js v26.5.0 / 標準モジュールのみ（`node:crypto` / `node:sqlite`）
- 生成件数: 特記なき限り各1000件

## 1. 辞書順ソートで生成順に戻る割合（gen-compare.mjs）

配列を辞書順 `sort()` した後、各要素が「生成順の位置」にぴたり一致した件数。

| 種別 | 一致 | 割合 |
|---|---|---|
| v7 | 3/1000 | 0.30% |
| v4 | 0/1000 | 0.00% |

- 予想（v7は高い一致率）は**外れた**。理由 → 3節・4節。1000件がわずか5ms程度で生成され、同一ms内に約200件詰まったため、ms内の順序が崩れた。

## 2. 先頭48bitからのtimestamp復元（extract-ts.mjs）

v7の先頭12hex（`parseInt(hex,16)`）を unix_ts_ms として復元し、生成時の `Date.now()` レンジと比較。

| 項目 | 結果 |
|---|---|
| サンプル | 5件 |
| 復元値が生成msレンジ外へはみ出した最大誤差 | **0ms** |

- RFC 9562 の「先頭48bit = unix_ts_ms」どおり、復元値は生成ミリ秒にぴたり一致。v4にはこの情報は無い。

## 3. SQLite `ORDER BY id`(TEXT PK) が挿入順と一致する割合（sqlite-order.mjs）

`DatabaseSync` でファイル(.sqlite)にv4/v7をTEXT PRIMARY KEYとして1000件ずつINSERT。

| テーブル | `ORDER BY id`==挿入順 | 割合 | 先頭5件のseq |
|---|---|---|---|
| v7t | 1000/1000 | **100.00%** | [0,1,2,3,4] |
| v4t | 2/1000 | 0.20% | [840,926,599,258,804] |

- ここでは v7 が**完全にソート可能**。1節(0.3%)との差は「生成ペース」の違い（4節）。
- ファイル: `workspace/uuid-order.sqlite`（217KB）

## 4. なぜ1節(0.3%)と3節(100%)で逆の結果に？（why-sqlite-100.mjs / verify-monotonic.mjs）

### 生成ペースと distinct ms（1000件）

| 作り方 | 所要 | distinct ms |
|---|---|---|
| (a) 純生成（tight loop） | 5.76ms | 5個（≒200件/ms） |
| (b) メモリDBへINSERT付き | 16.79ms | 13個（≒77件/ms） |
| ファイルDBへINSERT（3節の実測） | — | 先頭が2msおき（≒1件/ms未満） |

→ **1件/ms を下回るペースなら各IDが別msに分かれ、辞書順=生成順になる**。同一msに詰め込むほど崩れる。

### 同一ms内の順序（連続20万件、先頭48bit一致ペアのみ）

| モード | 同一ms隣接ペア | 逆転(cur<prev) | 逆転率 |
|---|---|---|---|
| 既定 `randomUUIDv7()` | 199,809 | 99,983 | **50.04%** |
| `{disableMonotonicity:true}` | 199,859 | 99,875 | 49.97% |

→ Node v26.5.0 の観測では、**既定でも同一ms内の順序は保証されない（約50%が逆転）**。`disableMonotonicity` を付けても実測差はほぼ無かった。

## 結論（観測範囲での言い方）

- v7は「**ミリ秒粒度で**」ソート可能。実務の通常INSERT（≒1件/ms以下）ではDBの`ORDER BY id`が挿入順と100%一致した。
- ただし「v7は完全にソート可能」は**言い過ぎ**。同一ミリ秒に大量生成すると、その中の順序は保証されない（本環境で約50%逆転）。
- v4は生成順とは無関係にしか並ばない（本検証で0〜0.2%）。timestamp復元も不可。
- 断定を避けるべき点: 上記はNode v26.5.0・macOS・1件/ms前後という自分の観測範囲の話。実装やバージョンで変わりうる。
