# 実践タスク: Bun 1.3 の Bun.SQL で SQLite を初めて触ってみた

## このタスクの前提

- 出典レポート: `research/search-topic-20260714-0400.md`
- 元テーマ: テーマ1「Bun 1.3 の Bun.SQL で SQLite を触ってみた（Node/betterのDB接続と比較）」（レポートの「最初に試すべき1本」）
- 対象技術: Bun 1.3 系の内蔵DBクライアント `Bun.SQL`（SQLite アダプタ）
- 記事の方向性（記事タイプ）: 「試してみた」「初めて触ってみた」「詰まった点をまとめた」
- 想定筆者 / 想定読者: Web系の新人エンジニア / 新人〜実務2年目（Bunを名前だけ知っている、Nodeでのdb経験が浅い層）
- 検証に使える想定時間: 半日（約3〜4時間）
- 判断方針: 引数は対象レポートのパスのみ。テーマ・時間・スキルレベルは無指定のためデフォルト前提を採用（テーマ=最初に試すべき1本、時間=半日、筆者=新人）。手順・API は Bun 公式ドキュメント（`bun.com/docs/runtime/sql`）と公式ブログ（`bun.com/blog/bun-v1.3`）で裏取りした一次情報に基づく。
- 実行環境の担保: SQLite はサーバ不要・追加依存なし・認証不要で、`bun run` と CLI・ファイル操作だけで完結する。DB操作の結果は CLI 出力（stdout）で判定でき、AIエージェント単独で完結できる。ブラウザUIを持たないため Playwright は本編では使わず、確認は「CRUD結果の標準出力ログ」で行う（任意で結果をHTML表として出力しPlaywrightスクショ化する深掘りタスクを1つ用意）。テーマの置き換えは不要（MySQL/PostgreSQL はサーバが要るため対象外とし、レポートの注意点どおり **SQLite単体に絞る**）。

## 完成イメージ（成果物）

- 作るもの: Bun 単体で動く最小の SQLite CRUD スクリプト群（`db.ts` 等）。`users` テーブルを作成し、INSERT / SELECT / パラメータバインド / トランザクション / エラーケースを一通り実行する。あわせて旧 `bun:sqlite` で同じ操作を書いた比較スクリプトを用意する。
- 「できた」と言える完了条件:
  - `bun run src/crud.ts` が **エラーなく終了**し、INSERT した行が SELECT で取得できて stdout に表示される
  - パラメータバインドで `${値}` が SQL に安全に渡り、期待した1件が返る
  - `begin()` トランザクション内の複数書き込みがコミットされ、途中で throw した場合はロールバックされる（両方をログで確認）
  - 旧 `bun:sqlite` 版が同じ結果を返し、両者の書き味の差分表が埋まる
- 完了確認の方法: CLI出力（stdout に流した SELECT 結果・件数・エラー全文）。深掘りタスクのみ Playwright スクショ（結果をHTML表に出力して撮る）。
- 記事タイトル案（そのまま使える形）:
  1. Bun 1.3 の Bun.SQL で SQLite を初めて触ってみた
  2. 「1つのAPIで3つのDB」Bun.SQL を新人が SQLite で試してみた
  3. Bun.SQL の SQLite を触って詰まったところと分かったこと

- [ ] インストールするもの: Bun 本体のみ（`curl -fsSL https://bun.sh/install | bash` または `npm install -g bun@1.3.14`）。DBドライバの追加インストールは**不要**（`Bun.SQL` は内蔵、`bun:sqlite` も内蔵）

## 実践タスク（フェーズ別）

### フェーズ1: 事前調査（目安: 30分）

- [ ] Bun 公式 SQL ドキュメント（https://bun.com/docs/runtime/sql）で SQLite の接続構文を確認する（目安: 15分）
  - 記録すること: 接続の書き方が複数ある点（`new SQL(":memory:")` / `new SQL("sqlite://myapp.db")` / `new SQL({ adapter: "sqlite", filename: "./app.db" })`）。**プロトコルなしのファイル名だけを渡す場合は `{ adapter: "sqlite" }` の明示が必須**という注意書きをメモ。「読む前に想像していた書き方」と実際の差。
- [ ] 旧 `bun:sqlite`（https://bun.com/docs/api/sqlite）のAPIをざっと確認し、比較の観点を決める（目安: 15分）
  - 記録すること: 旧APIは `import { Database } from "bun:sqlite"` → `db.query(...).all()` 系の**同期API**、新 `Bun.SQL` は**タグ付きテンプレート＋Promise**という差。比較で見たい項目（戻り値の形・パラメータバインドの書き方・同期/非同期）をリスト化。

### フェーズ2: 環境構築（目安: 45分）

- [ ] Bun をインストールし `bun --version` を記録する（目安: 15分）
  - 記録すること: 実行コマンド全文、出た版数（1.3.x）、既に別バージョンが入っていた場合の更新手順（`bun upgrade`）。想定と実測の差。
  - 補足: バージョンを固定したい場合は `npm install -g bun@1.3.14` で明示指定できる（**要確認**: 実行時点の最新安定版に合わせる。v1.3.14 は公式ブログ記載の最新安定版）。
- [ ] 作業ディレクトリを作り `bun init -y` で最小プロジェクトを初期化する（目安: 10分）
  - 記録すること: `bun init` の非対話フラグ（`-y` / `--yes`）で対話プロンプトを回避できたか。生成された `package.json` / `tsconfig.json` の中身。
- [ ] Hello World として `new SQL(":memory:")` で接続し `SELECT 1` を実行できることを確認する（目安: 20分）
  - 記録すること: `import { SQL } from "bun"` で読み込めたか、`await sql\`SELECT 1 as n\`` の戻り値の形（配列か、要素の形）を stdout に出して確認。ここで型エラー・import エラーが出たら全文記録。

### フェーズ3: 実装・検証【本編】（目安: 120分）

- [ ] `src/crud.ts` を作り、`new SQL("sqlite://app.db")` で接続して `users` テーブルを作成する（目安: 25分）
  - 記録すること: `await sql\`CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, name TEXT, email TEXT)\`` の実行ログ。ファイル `app.db` が生成されたか（`ls -la`）。
- [ ] INSERT を実装し、`RETURNING *` で挿入行を受け取って stdout に表示する（目安: 25分）
  - 記録すること: `const [row] = await sql\`INSERT INTO users (name,email) VALUES (${name},${email}) RETURNING *\`` の戻り値。`RETURNING` が SQLite でも効いたか。挿入件数・id の採番。
- [ ] SELECT ＋ パラメータバインドを実装し、`WHERE id = ${id}` で1件取得する（目安: 25分）
  - 記録すること: タグ付きテンプレートに変数を埋めた実際のコード。SQLインジェクションにならず**プレースホルダとして**渡っている実感（`${"1 OR 1=1"}` を入れても全件返らないことを試す）。戻り値が配列で返る点。
- [ ] トランザクション `sql.begin(async tx => { ... })` で複数INSERTをコミットし、次に途中で throw してロールバックされることを確認する（目安: 30分）
  - 記録すること: 正常時に全件コミットされたlog、throw 時に**1件も入っていない**ことを SELECT 件数で確認したlog。`begin` のコールバック内で `tx` を使う必要がある点（外側の `sql` を使うとどうなるかも軽く試す）。
- [ ] わざとエラーを起こす（存在しない列へのINSERT等）→ エラー全文を記録する（目安: 15分）
  - 記録すること: 出たエラーメッセージの**全文**（要約せず）。エラーオブジェクトの形（`.code` などがあるか）。新人が読んで原因に辿り着けたか。

### フェーズ4: 深掘り・比較（目安: 30分）

- [ ] 旧 `bun:sqlite` で同じ「テーブル作成→INSERT→SELECT」を書き、`Bun.SQL` 版と結果・書き味を比較する（目安: 20分）
  - 記録すること: 両版のコードを並べた差分。同期（`bun:sqlite` の `.all()`）vs Promise（`Bun.SQL` の `await`）、戻り値の形の違い、行数・パフォーマンス体感。「新人としてどっちが読みやすいか」の所感。
- [ ] （任意）SELECT結果を最小HTMLの表に出力し、Playwright でスクショを撮って記事用画像にする（目安: 10分）
  - 記録すること: 生成したHTML、Playwright スクショのパス。CLIだけで完結する検証を「記事映えする画像」にできたか。

### フェーズ5: 振り返り・記事化準備（目安: 30分）

- [ ] 記録テンプレを見返して詰まった点を棚卸しする（目安: 15分）
  - 記録すること: 詰まった箇所トップ3、解決までの試行、想定と実測の時間差。
- [ ] 「記事への写像」に沿って本文ドラフトの見出しを埋める（目安: 15分）

## 詰まりそうなポイントと対処の指針

| # | 詰まりそうな点 | なぜ起きるか | 最初に試すこと | 記事での活かし方 |
|---|---|---|---|---|
| 1 | バージョンが古く `Bun.SQL` の SQLite が動かない | SQLite対応は 1.3 系で統一APIになった新機能。古い Bun では PostgreSQL 専用のまま | `bun --version` で 1.3.x を確認。古ければ `bun upgrade` または `npm i -g bun@1.3.14` で固定 | 「まず版数を疑う」定番の教訓として冒頭に。`bun --version` の出力を必ず貼る |
| 2 | プロトコルなしのファイル名を渡して接続できない | `new SQL("app.db")` だけだと SQLite と判定されない。**`{ adapter: "sqlite" }` の明示が必要**（公式ドキュメント記載） | `new SQL("sqlite://app.db")` にするか `new SQL("app.db", { adapter: "sqlite" })` にする | 公式の落とし穴として「接続文字列3パターン」を表で紹介 |
| 3 | 同期のつもりで書いて結果が Promise のまま出る | SQLiteは内部同期実行だが `Bun.SQL` は**一貫してPromiseを返す**（他アダプタと同一API）。旧 `bun:sqlite` の同期感覚で `await` を忘れる | すべてのクエリに `await` を付ける。戻り値を `console.log` して Promise か配列か確認 | 旧APIとの一番の違いとして「同期 vs Promise」を比較の目玉に |
| 4 | トランザクションで `tx` でなく外側の `sql` を使ってしまう | `begin(async tx => ...)` のコールバック引数 `tx` を使わないとトランザクション境界に乗らない | コールバック内は必ず `tx\`...\`` を使う。throw 時にロールバックされるか件数で検証 | 「ロールバックを実際に見た」一次ログとして詰まった点セクションに |
| 5 | `bun init` の対話プロンプトで止まる | 非対話実行だと選択待ちでハングする | `bun init -y`（`--yes`）で既定値採用。生成物を確認 | headless/AI実行での定番Tipsとして環境構築の注釈に |
