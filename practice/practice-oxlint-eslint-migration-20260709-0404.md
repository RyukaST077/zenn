## このタスクの前提

- 出典レポート: `research/search-topic-20260709-0400.md`
- 元テーマ: テーマ#1「ESLintしか知らない新人がoxlintに移行し、JSプラグインalphaまで試してみた」（合計28点 / 優先度: 高 / レポートの「最初に試すべき1本」）
- 対象技術: oxlint（Rust製リンタ / oxc プロジェクト）＋ `@oxlint/migrate` ＋ JS Plugins（2026-03 alpha）
- 記事の方向性（記事タイプ）: 移行ログ＋試してみた＋詰まった点まとめ
- 想定筆者 / 想定読者: Web系の新人エンジニア（ESLintは使えるがoxlintは未経験） / 新人〜実務2年目
- 検証に使える想定時間: 半日（約3〜4時間）※最小構成なら1〜2時間で移行と実測まで到達可
- 判断方針: 引数で対象レポートのみ指定。テーマ・時間・スキルレベルはデフォルト前提を採用。手順は oxc.rs 公式ドキュメント（Linter Usage / JS Plugins Alpha ブログ 2026-03-11 / Migrate from ESLint）で裏取り済み
- 実行環境の担保: `pnpm add -D oxlint` と `npx @oxlint/migrate`、CLI 実行のみで完結。認証・課金・クラウド不要でローカル完結し、AIエージェント単独（CLI / コード / 必要なら Playwright スクショ）で最後まで実行できる。テーマ置き換えは不要

## 完成イメージ（成果物）

- 作るもの: ESLint flat config を持つ小さな React + TypeScript サンプルプロジェクト。これを oxlint に移行し、(1) 移行結果 (2) ESLint との実行時間・検出差の比較ログ (3) JS プラグイン（alpha）を1つ有効化した挙動、を記録した検証ログ一式
- 「できた」と言える完了条件:
  1. `npx @oxlint/migrate` で `.oxlintrc.json` が生成され、`pnpm lint`（= `oxlint`）がエラーなく完走して結果が表示される
  2. 同一対象に対する ESLint と oxlint の実行時間を計測し、数値の比較表が残っている
  3. `.oxlintrc.json` の `jsPlugins` に ESLint プラグインを1つ指定し、そのルールが「動いた／動かなかった」を出力ログで確認できている
- 完了確認の方法: 主に CLI 出力（lint 結果・`time` 計測値・バージョン）をログ保存。lint 結果の一覧表示や比較を視覚的に残したい場合は端末出力のスクショ、または（サンプルが動く UI を持つなら）Playwright でのスクショで補足
- 記事タイトル案（そのまま使える形）:
  1. ESLintしか知らない新人がoxlintに移行してみた — JSプラグインalphaはどこまで動くか
  2. oxlintへの移行で詰まったところと、速度を実測した話
  3. 話題のoxlintを半日試して、ESLintプラグインが本当に動くか確かめた

## 実践タスク（フェーズ別）

### フェーズ1: 事前調査（目安: 30分）

- [ ] oxc.rs 公式の Linter Usage と「JS Plugins Alpha（2026-03-11）」ブログ、「Migrate from ESLint」を開き、install / config / migrate / jsPlugins の要点を1〜2件確認する（目安: 20分）
  - 記録すること: 参照した URL と日付、`.oxlintrc.json` / `oxlint.config.ts` が既定の設定ファイルであること、migrate が **ESLint v9/v10 flat config 専用**（v8 legacy 不可）である点、alpha の既知の制約（フロントFW独自ファイル形式は限定サポート / Windows で OOM 既知 / JS プラグインは custom type-aware ルール非対応）。「事前に調べたことが後で効いた／外した」を書けるようメモ
- [ ] 作る成果物と完了条件をこのファイルの「完成イメージ」で確定し、実行手順の全体像をメモする（目安: 10分）
  - 記録すること: なぜこのテーマを試すのか（動機）、新人として不安に思っている点（例: 「本当に既存プラグインが動くのか」）

### フェーズ2: 環境構築（目安: 45分）

- [ ] React + TypeScript の最小サンプルを用意し、ESLint v9/v10 の **flat config**（`eslint.config.js`）と最低1つのプラグインを設定して `npx eslint .` が動く状態にする（目安: 25分）
  - 記録すること: 使った雛形コマンド、`eslint`・プラグインのバージョン、flat config の中身、`npx eslint .` の出力（警告/エラー件数）。※migrate は v8 の `.eslintrc` では動かないので flat config である点を明記
- [ ] `pnpm add -D oxlint` を実行し、`package.json` に `"lint": "oxlint"` / `"lint:fix": "oxlint --fix"` を追加、`oxlint --version` と素の `pnpm lint` が動くことを確認する（目安: 20分）
  - 記録すること: `oxlint --version`、インストール所要時間、初回 `pnpm lint`（設定移行前）の出力とデフォルト挙動。見積もり時間との差

### フェーズ3: 実装・検証【本編】（目安: 120分）

- [ ] `npx @oxlint/migrate ./eslint.config.js` を実行して `.oxlintrc.json` を生成し、中身（rules / env / globals / overrides / ignore）を ESLint 設定と見比べて差分を記録する（目安: 40分）
  - 記録すること: 実行コマンドと標準出力の全文、生成された `.oxlintrc.json` の全文、「移行されたルール／されなかったルール（unsupported）」、custom plugin が自動移行されないという注意が実際に当てはまったか
- [ ] 移行後の `.oxlintrc.json` で `pnpm lint` を実行し、ESLint の検出結果と突き合わせて「検出差」（oxlint で減った/増えた/消えたルール）を一覧化する（目安: 30分）
  - 記録すること: oxlint の lint 出力全文、ESLint との検出件数・ルールの差分表、差が出た理由の推測（未実装ルール等）
- [ ] 同一対象に対し ESLint と oxlint の実行時間を `time`（例: `time npx eslint .` / `time pnpm lint`）で3回ずつ計測し、平均を比較表にまとめる（目安: 30分）
  - 記録すること: 各回の計測値（real/user/sys）と平均、対象ファイル数・行数、マシン/OS/Node バージョン。速度比較は「この規模・この環境での簡易計測」である旨も併記
- [ ] `.oxlintrc.json` の `jsPlugins` に ESLint プラグインを1つ（例: `"eslint-plugin-testing-library"` など手元にある/入れやすいもの）指定し、対応する `rules` を有効化して `pnpm lint` で発火するか確認する（目安: 20分）
  - 記録すること: 追加した `jsPlugins` / `rules` の記述、プラグインが読み込まれたか、ルールが発火したか（発火例のログ全文）、`{ "name": ..., "specifier": ... }` のエイリアス記法が必要だったか。動かなかった場合はエラー全文

### フェーズ4: 深掘り・比較（目安: 30分）

- [ ] alpha の既知の制約を1つ実地で踏みに行く（例: フロントFW独自ファイル=`.vue`/`.svelte` を混ぜる、または custom type-aware ルールを JS プラグインで試す）か、`oxlint --fix` の自動修正範囲を ESLint と比べる（目安: 30分）

## 詰まりそうなポイントと対処の指針

| # | 詰まりそうな点 | なぜ起きるか | 最初に試すこと | 記事での活かし方 |
|---|---|---|---|---|
| 1 | `@oxlint/migrate` が既存の ESLint 設定を変換してくれない | migrate は **ESLint v9/v10 flat config 専用**で、v8 の `.eslintrc.*` legacy 形式には対応しない | 移行元を flat config（`eslint.config.js`）にする。v8 なら先に flat config へ移行 or 手書きで `.oxlintrc.json` を用意 | 「移行の前提条件」を新人がハマる罠として最初に提示できる |
| 2 | 移行後にルールが減っている／検出が変わる | oxlint は全 ESLint ルールを実装済みではなく、未対応ルールは移行時にスキップされる。custom plugin も自動移行されない | 生成された `.oxlintrc.json` を ESLint 設定と diff。unsupported ルール一覧と互換マトリクスを確認 | 「移行で何が失われるか」を実測差分で示す=記事の核 |
| 3 | `jsPlugins`（alpha）でプラグインが読み込めない／ルールが発火しない | JS Plugins は alpha。プラグイン名の解決にエイリアス（`{ name, specifier }`）が要る場合や、custom type-aware ルールは非対応 | 拡張記法 `{ "name": "...", "specifier": "eslint-plugin-..." }` を試す。type-aware 依存ルールは避ける | 「alphaはどこまで動くか」の一次記録。動かない事例そのものが価値 |
| 4 | Node/pnpm やランタイムのバージョン差で挙動が違う | 公式に最小 Node バージョンの明示が見当たらず（要確認）、環境依存で差が出うる | `node -v` / `pnpm -v` / `oxlint --version` を記録し、動いた組み合わせを明示 | 再現性の担保。読者が同じ環境で追試できる |
| 5 | 速度比較が「盛りすぎ」に見える | 対象規模・キャッシュ・並列度で数値が大きく変わる。1回計測はブレる | 同一対象で3回計測して平均、対象ファイル数と環境を併記 | 「50〜100倍」の触れ込みを新人が実測で検証した誠実なログにできる |
| 6 | Windows で Out of memory になる | JS Plugins alpha の既知の問題として Windows での OOM が公式に挙がっている | まずは macOS/Linux で検証。Windows なら該当を記録し回避策を探す | OS 差の詰まりどころとして共有価値が高い |

## 経験談として書くときのコツ

- 「新人が試した範囲」を明示し、専門家として断定しすぎない（特に速度と alpha の評価）
- うまくいった点だけでなく、詰まった点と解決過程を書く（migrate の前提・未対応ルール・jsPlugins の制約）
- 実行ログ・計測値・`.oxlintrc.json` 全文・コードを残して貼る
- 公式ドキュメント（oxc.rs のブログ/ドキュメント）へのリンクを入れる
- 手順の再現性（oxlint/eslint/Node/pnpm のバージョン、OS、対象ファイル数）を明記する

## 想定リスク・注意点

- コスト（無料枠の範囲・課金トリガー）: なし。すべて OSS・ローカル完結。ネットワークは npm 取得のみ
