# 実践タスク: ESLint/oxlintしか知らない新人がBiomeの「型認識lint」を試してみた

## このタスクの前提

- 出典レポート: `research/search-topic-20260711-0402.md`
- 元テーマ: テーマ1「ESLint/oxlintしか知らない新人がBiomeの『型認識lint』を試してみた」（レポートの「最初に試すべき1本」）
- 対象技術: Biome（型認識リンティング / type-aware linting、`types` ドメイン）
- 記事の方向性（記事タイプ）: 「試してみた」「検証ログ」「詰まった点まとめ」
- 想定筆者 / 想定読者: Web系の新人エンジニア / 新人〜実務2年目（ESLint・Prettier または oxlint 経験あり）
- 検証に使える想定時間: 半日（約3〜4時間）
- 判断方針: 引数で渡されたのは対象レポートのみ。テーマ・時間・スキルレベルは未指定のため、レポートの「最初に試すべき1本」＝テーマ1、時間＝半日、レベル＝新人のデフォルト前提を採用した。
- 実行環境の担保: `npm i -D -E @biomejs/biome` と CLI（`biome init` / `biome check`）だけで完結。完全ローカル・認証不要・課金不要・ブラウザ依存なしで、AIエージェント単独（CLI + コード + 必要ならログのファイル保存）で作成から検証まで実行できる。テーマの置き換えは不要。

## 完成イメージ（成果物）

- 作るもの: 型依存バグ（Promiseの未await、`+` の誤用、switchの網羅漏れ等）を仕込んだ**最小TypeScriptプロジェクト**と、それに対する Biome の型認識lint（`types` ドメイン）の検出結果ログ。加えて oxlint / ESLint で同じコードを検査した比較ログ。
- 「できた」と言える完了条件:
  1. `biome.json` で `types` ドメインを有効化した状態で `npx @biomejs/biome check` を実行し、**型依存バグが検出される**こと（例: `noFloatingPromises` が発火）。
  2. `types` ドメインを無効にした（=型認識なしの）状態では**同じバグが検出されない**ことを対比で確認できること。
  3. oxlint（と可能なら ESLint）で同じコードを検査し、**型依存バグを拾えない／拾える差**をログとして残せること。
- 完了確認の方法: すべて CLI 出力（`biome check` の stdout/stderr）をテキストとして保存。ブラウザ確認は不要のため Playwright は使わない。検出あり/なしの出力差分がそのまま証跡になる。
- 記事タイトル案（そのまま使える形）:
  1. ESLint/oxlintしか知らない新人がBiomeの「型認識lint」を試してみた
  2. tscなしで型を見るlint？ Biomeの type-aware linting を触ってみた
  3. oxlintに無くてBiomeにある「型認識lint」を新人が検証してみた

- [ ] ローカル環境（言語・ランタイム・バージョン）: Node.js（LTS推奨。**実際に使ったバージョンを `node -v` で記録**。Biome v2 系は比較的新しい Node を前提とするため要確認）。TypeScript を devDependency として導入。
- [ ] インストールするもの: `@biomejs/biome`（型認識lintの主役）、`typescript`、比較用に `oxlint`、（任意）`eslint` + `typescript-eslint`。
- [ ] 無料枠 / コストの確認: すべて OSS・無料・オフライン実行可。ネットワークは npm install 時のみ。

## 実践タスク（フェーズ別）

### フェーズ1: 事前調査（目安: 30分）

- [ ] Biome 公式の型認識lint関連ドキュメント（Linter Domains / Roadmap 2026）を読み、`types` ドメインで有効になるルール名を控える（目安: 15分）
  - 記録すること: 型認識ルールの正式名（例: `noFloatingPromises` / `noMisusedPromises` / `useAwaitThenable` / `noUnsafePlusOperands` / `useExhaustiveSwitchCases`）と、各ルールが nursery（実験）扱いか安定扱いか。公式に「tscに依存しない世界初の型認識lint」と書かれた根拠URL。
- [ ] oxlint が型認識ルールを持たない（=ESLint依存が残る）ことを一次情報で確認する（目安: 15分）
  - 記録すること: 「oxlintで型情報が要るルールはどう扱われるか」の出典。Biomeとの差別化ポイントを1〜2文でメモ。

### フェーズ2: 環境構築（目安: 45分）

- [ ] 検証用ディレクトリを作り `npm init -y` → `npm i -D -E @biomejs/biome typescript` を実行する（目安: 15分）
  - 記録すること: 実行コマンド全文、インストールされた**Biomeの実バージョン**（`npx @biomejs/biome --version`）、`node -v` / `npm -v`。
- [ ] `npx @biomejs/biome init` で `biome.json` を生成し、`tsconfig.json` を用意する（目安: 15分）
  - 記録すること: `init` が対話プロンプトを出すか（headless/CIでの挙動）。生成された `biome.json` の初期内容。詰まったら試したフラグ。
- [ ] 何も直していないサンプル1ファイルに対し `npx @biomejs/biome check` を実行し、Biomeが動く（Hello World）ことを確認する（目安: 15分）
  - 記録すること: 初回 `check` の出力全文。想定どおり動いたか、エラーが出たか。

### フェーズ3: 実装・検証【本編】（目安: 120分）

- [ ] 型依存バグを仕込んだ最小TSファイルを作る（未awaitのPromise、`number + string`の誤用、網羅漏れswitch など3種）（目安: 30分）
  - 記録すること: 作成したコード全文。各バグが「型情報がないと検出できない」理由を1行ずつメモ。
- [ ] `biome.json` の `linter.domains.types` を `"recommended"`（または `"all"`）に設定し、`npx @biomejs/biome check` を実行して検出を確認する（目安: 30分）
  - 記録すること: 設定変更後の `biome.json`。`check` の出力全文（どのルールが発火し、行番号・メッセージがどう出たか）。所要時間。
- [ ] `types` ドメインを無効化（`"off"` または該当行を削除）して再度 `check` し、**同じバグが検出されなくなる**ことを対比で確認する（目安: 30分）
  - 記録すること: 有効時/無効時の出力差分。検出件数の変化。「型認識の有無で何が変わったか」の気づき。
- [ ] `types` ドメインの scan が重い/遅いなど挙動を観察し、`--verbose` などで追加情報を取る（目安: 30分）
  - 記録すること: 実行時間の体感、`types` 有効時に project scan が走る旨のログ、詰まった設定（tsconfig未設定で警告が出た等）。

### フェーズ4: 深掘り・比較（目安: 30分）

- [ ] `npm i -D oxlint`（任意で `eslint` + `typescript-eslint`）を入れ、同じTSファイルを oxlint で検査して結果を比較する（目安: 30分）
  - 記録すること: oxlint の実行コマンドと出力全文。**型依存バグを検出できたか否か**。Biome（型認識あり）との検出差を表形式でメモ。ESLibまで試せた場合は typescript-eslint の型認識ルール（`@typescript-eslint/no-floating-promises`）との比較も残す。

### フェーズ5: 振り返り・記事化準備（目安: 30分）

- [ ] 「記事への写像」に沿って本文ドラフトの見出しを埋める（目安: 15分）

## 詰まりそうなポイントと対処の指針

| # | 詰まりそうな点 | なぜ起きるか | 最初に試すこと | 記事での活かし方 |
|---|---|---|---|---|
| 1 | 型認識ルールが検出されない | `types` ドメイン未有効、または対象ルールが nursery で recommended に含まれていない | `biome.json` に `linter.domains.types: "recommended"`（不足なら `"all"`）を設定。ルールを個別に `error` 指定して確実に発火させる | 「有効化でハマった手順」を再現可能な形で書く。設定の前後で出力がどう変わるかを見せる |
| 2 | 型情報が解決されず期待通り動かない | `types` ドメインはプロジェクト全体を scan して型を推論する。`tsconfig.json` が無い/参照解決できないと検出が不安定 | `tsconfig.json` を用意し、ファイルが `include` されているか確認。単一ファイルではなくプロジェクトとして `check` する | 「tscに依存しない」と言っても前提（プロジェクトscan）は要る、という新人目線の発見として書く |
| 3 | Node/Biomeのバージョン不一致で init/check が動かない | Biome v2 系は新しめの Node を前提。古い記事のコマンド（v1系の書式）が混在 | `node -v` と `npx @biomejs/biome --version` を先に記録し、公式の現行ドキュメント（v2.x）を参照。v1系の記事のコマンドを鵜呑みにしない | 「どのバージョンで動いたか」を明記し再現性を担保。バージョン差でハマった点を共有 |
| 4 | `biome init` が対話プロンプトで止まる（headless想定） | 一部の初期化コマンドは対話を求める場合がある | 生成後の `biome.json` を直接編集する方針に切り替える。非対話フラグの有無を確認 | 自動実行（CI/エージェント）で使う際の注意として書くと実用性が上がる |
| 5 | oxlint/ESLintとの比較が噛み合わない | ツールごとにルール名・デフォルト有効範囲が違い、単純比較できない | 「同一コードに対する各ツールの素の出力」を並べ、検出の有無だけを表で比較する | 「新人が同じコードを3ツールに通した」比較表そのものが記事の核になる |

## 経験談として書くときのコツ

- 「新人が試した範囲」を明示し、専門家として断定しすぎない。
- うまくいった点だけでなく、型認識が効かなかった／設定でハマった過程を書く。
- 実行ログ（`biome check` の出力）とコードを貼る。CLIツールなのでスクショより**テキスト出力の貼り付け**が読みやすい。
- 公式ドキュメント（Roadmap 2026 / Linter Domains）へのリンクを入れる。
- 再現性のため Node・Biome・oxlint の**実バージョン**とOSを明記する。

## 想定リスク・注意点

- コスト（無料枠の範囲・課金トリガー）: なし。すべてOSS・ローカル実行。npm install 以外のネットワーク不要。
- ライセンス / 規約: Biome / oxlint / ESLint はいずれもOSS。検証で使う脆弱パッケージ等は不要。
- セキュリティ（APIキーの扱い等）: 認証情報なし。秘密情報を含めない。
- 型認識ルールの扱い: 型認識ルールの一部は **nursery（実験）扱い**の可能性がある。正式名・安定/実験の別・有効化方法は着手時に公式ドキュメントで最新を再確認する（本ファイルのルール名は要確認）。
- 撤退ライン: `types` ドメインを有効化しても検出が全く再現できない場合、（a）個別ルールを `"error"` で明示指定、（b）`tsconfig.json` とプロジェクトscanの前提を見直す。それでも30分で再現しなければ「現時点では新人環境で型認識lintを安定再現できなかった」という**詰まりログ自体を成果**として記事化し、比較（フェーズ4）へ進む。
