# 実践タスク: Node 26.5 の `--experimental-import-text` で `.txt` を import して詰まった記録

## このタスクの前提

- 出典レポート: `research/search-topic-20260713-0400.md`
- 元テーマ: テーマ#1（最優先）「Node 26.5の `--experimental-import-text` で `.txt` をimportしてみて、属性なしで詰まった記録」
- 対象技術: Node.js v26.5.0 の text imports（import attributes `with { type: 'text' }` / `--experimental-import-text` フラグ）
- 記事の方向性（記事タイプ）: 「試してみた」「詰まった点をまとめた」検証ログ型
- 想定筆者 / 想定読者: Web系の新人エンジニア / 新人〜実務2年目（ESM・importは書けるが import attributes は未経験）
- 検証に使える想定時間: 半日（約3〜4時間）
- 判断方針: 引数で対象レポートのみ指定。テーマは無指定のためレポートの「最初に試すべき1本」（=候補#1）を採用。その他はデフォルト前提を採用
- 実行環境の担保: Node のインストール・数行のファイル作成・CLI実行・エラーログ収集のみで完結。ネットワーク・課金・アカウント・ブラウザ操作すべて不要。AIエージェント（Claude Code）単独で最後まで実行・検証可能。テーマの置き換えは不要

> **裏取り済みの一次情報（2026-07-13 時点）**
> - Node.js **v26.5.0（2026-07-08 リリース）** で text imports が追加（SEMVER-MINOR, PR #62300）。同リリースで `blob.textStream()`（Blob UTF-8 ストリーミング, PR #64036）も追加。
> - `--experimental-import-text` フラグが**必須**。付けないとエラー。
> - 構文は `import message from './message.txt' with { type: 'text' };`。`with { type: 'text' }` は**省略不可**。
> - 公開されるのは **default export のみ**（named export 不可）。default の値はファイル内容の**文字列**。
> - Stability: **1.0 - Early development**（実験的。将来変更されうる）。
> - 比較用: JSON imports（`with { type: 'json' }`）は v23.1.0+ で**安定・フラグ不要**、default export のみ、値はパース済みオブジェクト。
> - 出典: Node.js 公式 ESM ドキュメント / v26.5.0 リリースノート（WebFetchで確認）。

## 完成イメージ（成果物）

- 作るもの: `node:` 標準機能だけで動く最小 ESM プロジェクト（`.txt` を text import する `main.mjs`、比較用の `fs.readFileSync` 版、JSON import 版）と、フラグ/属性の有無で挙動が変わることを示す**実行ログ集**。
- 「できた」と言える完了条件:
  1. `--experimental-import-text` **なし**で実行 → エラー全文が取れている
  2. `--experimental-import-text` **あり**で実行 → `.txt` の中身が文字列として出力される
  3. `with { type: 'text' }` を**省略/誤記**した失敗パターンのエラー全文が取れている
  4. `fs.readFileSync` 版・JSON import 版と、コード量・型・出力を比較した表ができている
- 完了確認の方法: CLI 出力（成功時の標準出力／失敗時のエラー全文）をログファイルに保存して判定。ブラウザ確認は不要なため Playwright は使わない。
- 記事タイトル案（そのまま使える形）:
  1. `node app.mjs` で `.txt` を import してみたら、まず属性で怒られた話（Node 26.5 text imports）
  2. 新人がNode 26.5の text imports を触って、フラグと `with { type:'text' }` で詰まった記録
  3. JSON importは知ってた。text importは何が違うのか小さく試してみた

## 詰まりそうなポイントと対処の指針

| # | 詰まりそうな点 | なぜ起きるか | 最初に試すこと | 記事での活かし方 |
|---|---|---|---|---|
| 1 | Node が 26.5 未満で機能自体が無い | 検証環境の既定が古い（例 v22.17.0）。text imports は v26.5.0 追加 | `node -v` を確認 → バージョンマネージャで 26 系を導入し切替 | 「まずランタイムのバージョンで詰まった」導入エピソードにする |
| 2 | `--experimental-import-text` を付け忘れて実行が失敗 | フラグが必須（安定版JSONと違いフラグ不要ではない） | フラグ付きで再実行。JSONの感覚で無フラグにしていないか疑う | フラグ有無のエラー/成功を並べ、実験機能の慎重さを示す |
| 3 | `with { type: 'text' }` を省略/誤記して失敗 | import attribute が必須。属性なし・`type` 違いは別モジュール種別扱い | 属性を正しく付ける。`json`と`text`の取り違えを確認 | 「属性で怒られた話」の核心。誤記別のエラー対応表にする |
| 4 | named import で受け取ろうとして失敗 | 公開は default export のみ | `import x from ...` の default 形に直す | 「default だけ」という仕様を実測で示す |
| 5 | `package.json` の `"type":"module"` 忘れ／`.mjs` 拡張子違い | ESM でないと import attributes 構文が使えない | `"type":"module"` を入れるか `.mjs` を使う | ESM 前提を明示し再現性を担保する |
| 6 | `ExperimentalWarning` を「エラー」と誤認 | 実験機能は警告を標準エラーに出す | 警告と致命エラーを区別し、成功出力があるか確認する | 「警告は出るが動く」を明記し読者の不安を減らす |
