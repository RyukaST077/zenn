## 完成イメージ（成果物）

- 作るもの: `node:ffi` で libc の `strlen` / `abs` を JS から呼ぶ最小スクリプト群（`step*.mjs`）＋「フラグ忘れ→型ミス→ポインタ渡し→Permission Model」の時系列実行ログ
- 「できた」と言える完了条件:
  1. `--experimental-ffi` 付きで `strlen('hello')` が `5n`（BigInt）を返す
  2. `abs(-42)` が `42`（number）を返す
  3. フラグ無し実行時のエラー全文、型を誤ったときの誤値/挙動、`--permission` で `--allow-ffi` 無しだと弾かれることの3つがログに残っている
- 完了確認の方法: 各スクリプトの標準出力/標準エラーをファイルにリダイレクトして保存（`node ... > logs/stepN.txt 2>&1`）。ブラウザ表示は無いため Playwright は使わない
- 記事タイトル案（そのまま使える形）:
  1. Node 26 の `node:ffi` で JS から libc の strlen を呼ぶまでに詰まったこと
  2. C++アドオンなしでネイティブ関数を叩く：Node 26 の実験的 FFI を新人が試してみた
  3. `--experimental-ffi` を知らずに時間を溶かした話（Node 26 node:ffi 入門）

### フェーズ3: 実装・検証【本編】（目安: 120分）

- [ ] `step1-noflag.mjs` を作り、`import { dlopen } from 'node:ffi'` を**フラグ無し**で実行してエラー全文を保存する: `node step1-noflag.mjs > logs/step1.txt 2>&1`（目安: 20分）
  - 記録すること: エラーメッセージの**全文**（`No such built-in module` 系か、experimental ゲートの案内か）。記事の「詰まった点：フラグ忘れ」の核
- [ ] `step2-strlen.mjs` で `dlopen(null, { strlen: { arguments: ['pointer'], return: 'u64' } })` を書き、`--experimental-ffi` 付きで `functions.strlen('hello')` を実行して `5n` を確認する（目安: 30分）
  - 記録すること: 成功時のコマンドと出力（`5n`）。`5` ではなく `5n`（BigInt）で返ることに気づいた瞬間。`arguments: ['string']` にした場合との違いも試したら記録
- [ ] `step3-abs.mjs` で `abs: { arguments: ['i32'], return: 'i32' }` を呼び、`abs(-42) === 42`（number）を確認する（目安: 15分）
  - 記録すること: 32bit 整数は number、64bit は BigInt という戻り値型の使い分け。実行ログ
- [ ] `step4-wrongtype.mjs` で戻り値/引数の型を**故意に誤る**（例: `strlen` の return を `i32` にする、`u64` を `number` で扱おうとする）→ 誤値・例外・（起きれば）クラッシュを安全に観察・保存する（目安: 35分）

- [ ] `step5-using.mjs` で `using handle = dlopen(...)` を使い、ブロックを抜けると自動 close されることと、明示 `close()` の二重呼び出しが no-op なことを確認する（目安: 20分）
  - 記録すること: `using` あり/なしのコード差分。`[Symbol.dispose]` / `close()` の挙動

## 詰まりそうなポイントと対処の指針

| # | 詰まりそうな点 | なぜ起きるか | 最初に試すこと | 記事での活かし方 |
|---|---|---|---|---|
| 1 | Node のバージョンが古く `node:ffi` が存在しない | ローカルは v22.17.0。`node:ffi` は 26.1 で追加 | `node -v` を確認し `nvm install 26 && nvm use 26` | 「そもそも自分の Node では `No such built-in module: node:ffi` だった」出発点として冒頭に置く |
| 2 | `--experimental-ffi` を付け忘れる | experimental ゲートの二重構造を知らない | 実行コマンドの先頭に `--experimental-ffi` を付ける | タイトルにもなる王道の詰まり。フラグ無し/有りのエラー対比を貼る |
| 3 | 戻り値 `u64` を number として扱いクラッシュ/誤値 | 64bit 整数は BigInt で返る仕様を知らない（`5` ではなく `5n`） | `return: 'u64'` の結果を `typeof` で確認し BigInt 前提で扱う | 「`5n` に驚いた」型ミスマッチの実演。number/BigInt の境界を図解 |
| 4 | 誤ったシグネチャでプロセスがクラッシュ | FFI はポインタ・型の正しさを検証しない（unsafe） | まず正しいシグネチャで動かし、誤りは1箇所ずつ変えて観察 | 「危険」明示でクラッシュ例を載せる。unsafe の実感が記事価値 |
| 5 | 導入した Node に FFI サポートが含まれない可能性 | 「builds with FFI support のみ」提供と明記あり（配布バイナリの有無は要確認） | `import('node:ffi')` が読めるか単体確認。ダメなら公式リリースノート/ビルドフラグを確認 | 「experimental は配布ビルド依存」という新人が見落とす前提として補足 |
| 6 | `--permission` 下で `--allow-ffi` 無しだと拒否される | Permission Model が FFI を既定で制限 | `--allow-ffi` を追加して比較 | セキュリティ設計の意図を体験として書ける |

## 想定リスク・注意点

- コスト: なし（完全ローカル・無料）
- ライセンス / 規約: なし（標準の Node.js のみ）
