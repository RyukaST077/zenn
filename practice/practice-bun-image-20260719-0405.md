## 完成イメージ（成果物）

- 作るもの: 同一の入力画像に対し「読み込み → リサイズ → WebP/AVIF 変換 → 保存」を **Bun.Image** と **sharp** の両方で実装した比較用スクリプトと、結果をまとめた比較表（依存差・出力サイズ・処理時間・コード行数）。
- 「できた」と言える完了条件:
  1. `bun run bun-image.ts` を実行すると、Bun.Image でリサイズ＋WebP変換した出力ファイルが生成される。
  2. `bun run sharp-image.ts`（または `node`）を実行すると、sharp で同処理の出力ファイルが生成され、両者の出力サイズ・処理時間がコンソールに出る。
  3. Bun.Image 版は追加の `node_modules`（native ビルド）なしで動くことを確認できる。
- 完了確認の方法: スクリプトの標準出力（処理時間・出力バイト数）＋ `ls -la` による出力ファイルの存在・サイズ確認（CLI出力で判定。ブラウザ確認は不要）。
- 記事タイトル案（そのまま使える形）:
  1. Bun.Imageを初めて触って、sharpと同じ画像処理を書き比べてみた
  2. sharpしか知らない新人がBun.Imageで画像リサイズを試してみた
  3. Bun 1.3.14のBun.Imageで、依存ゼロの画像変換がどこまでできるか試した

## 実践タスク（フェーズ別）

### フェーズ1: 事前調査（目安: 30分）

- [ ] Bun.Image 公式ドキュメント（https://bun.com/docs/runtime/image）で API の形（コンストラクタ・`.resize()`・`.webp()`/`.avif()`・terminal メソッド `.write()`/`.bytes()`）と対応フォーマット表を確認する（目安: 15分）
  - 記録すること: チェーンAPIの形（`Bun.file("x.jpg").image().resize(...).webp(...).write(...)`）／HEIC・AVIF が macOS/Windows 限定・Linux 非対応である点／`ERR_IMAGE_FORMAT_UNSUPPORTED` というエラーコードの存在をメモ
- [ ] Bun v1.3.14 リリースブログ（https://bun.com/blog/bun-v1.3.14）で公式ベンチ値（metadata 70x / resize→JPEG 1.38x / →WebP 1.33x、比較対象 sharp 0.34.5・Linux x64）を控える（目安: 10分）
  - 記録すること: 公式値は「metadata が突出して速い一方、実処理は1.3〜1.4倍」という温度感。記事で自分の実測と区別して引用するため、数値と条件（OS/対sharpバージョン）をそのまま記録
- [ ] 自分の実行環境を記録する（`bun --version`、`node --version`、OS）（目安: 5分）
  - 記録すること: バージョンとOS。特に **HEIC/AVIF の対応可否はOS依存**なので macOS か Linux かを明記

### フェーズ2: 環境構築（目安: 45分）

- [ ] 作業ディレクトリを作り `bun init --yes` で最小プロジェクトを初期化する（対話プロンプト回避に `--yes`）（目安: 10分）
  - 記録すること: 実行コマンドと生成物。`bun init` で何が作られるか（package.json / tsconfig / index.ts）
- [ ] 比較用のサンプル画像を用意する。外部DL不要にするため、まず Bun.Image 自体で単色/グラデーションのPNGを生成するか、リポジトリ内の既存画像を使う（目安: 15分）
  - 記録すること: 画像の作り方・寸法・元サイズ（バイト）。外部素材に頼らず再現できる形にした点
  - 補足: 生成が難しければ ImageMagick 等でローカル生成でも可。**認証・DLを伴う素材サイトは使わない**
- [ ] `Bun.file("input.png").image().metadata()` を実行して `{ width, height, format }` が取れることを確認する（Hello World 相当）（目安: 10分）
  - 記録すること: metadata の出力。ここで動けば Bun.Image が使える確証になる
- [ ] `bun add sharp` を実行し、導入にかかった時間・native ビルドの有無・追加された依存数を記録する（目安: 10分）
  - 記録すること: `bun add sharp` の全出力（ビルドログ・警告があれば全文）、`node_modules` のサイズ差。**sharp 側の導入コスト**が記事の核なので丁寧に残す

### フェーズ3: 実装・検証【本編】（目安: 120分）

- [ ] `bun-image.ts` を実装: 入力画像を `.resize(800, 800, { fit: "inside" })` → `.webp({ quality: 80 })` → `.write("out/bun.webp")` する（目安: 30分）
  - 記録すること: 実装コード全文。チェーンの書き味・型補完の効き具合。`fit: "inside"` と既定 `"fill"` の違いをメモ
- [ ] `sharp-image.ts` を実装: 同じ入力に対し sharp で `resize(800, 800, { fit: "inside" })` → `.webp({ quality: 80 })` → `.toFile("out/sharp.webp")` する（目安: 30分）
  - 記録すること: 実装コード全文。Bun.Image とのAPI差（terminal が `.write()` vs `.toFile()` 等）、書いていて詰まった点
- [ ] 両スクリプトに `performance.now()` を入れ、各処理を**ウォームアップ1回＋計測5回**して中央値を取り、出力バイト数とともにコンソール出力する（目安: 30分）

  - 記録すること: AVIF が通ったか／`ERR_IMAGE_FORMAT_UNSUPPORTED` 等が出たかのエラー全文。OS（macOS/Linux）と結果の対応。**ここが「詰まった点」の目玉**

## 詰まりそうなポイントと対処の指針

| # | 詰まりそうな点 | なぜ起きるか | 最初に試すこと | 記事での活かし方 |
|---|---|---|---|---|
| 1 | AVIF/HEIC 変換が失敗する | HEIC/AVIF は macOS/Windows 限定で **Linux は非対応**。Intel/M1/M2 Mac は AVIF **エンコード**非対応の注記もある | まず WebP で通す。AVIF は try/catch で `error.code === "ERR_IMAGE_FORMAT_UNSUPPORTED"` を捕捉し、OSと結果を記録 | 「依存ゼロでも OS依存の壁は残る」という新人目線の学び。表で OS×フォーマットの可否を示す |
| 2 | 処理時間が測るたびに大きくブレる | JIT ウォームアップ・OSキャッシュ・GC の影響。1回計測では実態を反映しない | ウォームアップ1回＋5回計測で中央値。入出力を毎回同条件にする | 「公式の1.3倍を鵜呑みにせず自分で測ると揺れる」体験そのものが記事価値 |
| 3 | sharp の `bun add` / 実行でつまずく | sharp は native prebuild 依存で、環境によりビルド・プラットフォームバイナリ解決で警告が出ることがある | `bun add sharp` の出力を全文確認。動かなければ Node.js で実行して切り分け | まさに「sharp のビルド地獄を Bun.Image が消せるか」という記事の主題に直結 |
| 4 | Bun.Image の入力にユーザー入力文字列を渡してしまう | 公式が「ユーザー制御文字列を直接コンストラクタに渡すと任意ファイル読み取りになる」と警告 | 入力パスはコード内の固定値にする。検証用途では問題ないが注意点として明記 | セキュリティ注意点として1段深い記事にできる |
| 5 | 出力サイズ比較で quality 条件がズレる | Bun.Image と sharp で quality 既定値や内部エンコーダが異なると不公平な比較になる | 両者とも `quality: 80` 等を**明示**して揃える。fit も両方 `inside` に統一 | 「比較は条件を揃えないと意味がない」という検証の作法を見せられる |

## 想定リスク・注意点

- コスト: すべてローカル・完全無料。課金トリガーなし
- ライセンス / 規約: サンプル画像は自前生成またはライセンス明確なものを使う（DL素材サイトの規約に注意。認証が要るサイトは使わない）
- セキュリティ: Bun.Image のコンストラクタに**ユーザー制御文字列を直接渡さない**（任意ファイル読み取りになる、と公式が警告）。検証では入力パスを固定値にする
