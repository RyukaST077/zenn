# 実践タスク: Node 26.1 の `randomUUIDv7()` と `randomUUID()`(v4) を書き比べ、SQLiteの並び順で違いを見る

- 実行環境の担保: すべてローカル・完全無料・認証不要。標準モジュール（`node:crypto` / `node:sqlite`）のみで完結し、外部API・課金キー・人手サインアップ・手動デプロイは一切不要。ブラウザ確認も不要なため Playwright は使わず、完了条件はすべて CLI 出力・生成ファイルで判定する。

> **裏取りメモ（一次情報で確認済み）**
> - `crypto.randomUUIDv7()` は **Node.js v26.1.0** で追加（PR #62553）。**Stable**（SEMVER-MINOR追加）で、**実行時警告は出ない**。v26.5 時点のドキュメントでは `randomUUIDv7([options])` として options 引数が追加されている（要確認: 実行環境のバージョンで options が使えるか）。
> - 公式ドキュメントに **「非単調（non-monotonic）なクロックに依存する」** という注意書きがある。連続呼び出しでタイムスタンプ部が単調増加する保証はない → **SQLiteの並び順検証の核心的な観測ポイント**になる（後述の詰まりポイント参照）。
> - `node:sqlite`（`DatabaseSync`）は **v25.7.0 で `--experimental-sqlite` フラグ不要**になり、Release Candidate（Stability 1.2）。**ExperimentalWarning は出ない**。Node 26 系ではフラグなしで `import { DatabaseSync } from 'node:sqlite'` が使える。

## 完成イメージ（成果物）

- 作るもの: 単一リポジトリ内の小さな検証スクリプト群（依存ゼロ、`node:crypto` / `node:sqlite` のみ）
  1. `gen-compare.mjs` … v4/v7 を各1000件生成し、辞書順ソート結果と生成順の一致率を出す
  2. `extract-ts.mjs` … v7 の先頭48bitから Unix時刻(ms) を復元し、生成時刻との誤差を出す
  3. `sqlite-order.mjs` … v4/v7 をPKとして連続INSERT→`ORDER BY id` と挿入順(rowid)の一致度を出す
  4. `results.md` … 上記の観測結果を表にまとめた検証ログ
- 「できた」と言える完了条件:
  - 3スクリプトが `node <file>.mjs` でエラーなく実行され、標準出力に比較結果（一致率・誤差ms・並び順の一致度）が出る
  - v4 は辞書順ソートで生成順とほぼ一致しない、v7 はほぼ一致する、という差が数値で観測できる
  - v7 の timestamp 復元値が生成時刻と数ms〜数十ms以内で一致する
  - `results.md` に3項目の結果表と「詰まった点」が埋まっている
- 完了確認の方法: CLI 出力（各スクリプトの標準出力ログ）と生成された `results.md` / `.sqlite` ファイルの中身。ブラウザ表示は無いため Playwright 不要。
- 記事タイトル案（そのまま使える形）:
  1. Node 26の`randomUUIDv7()`と`randomUUID()`を書き比べて、SQLiteの並び順で違いを見た
  2. UUIDv7とv4を実際に生成して、辞書順ソートとtimestamp復元で違いを確かめた
  3. 「v7はソートできる」を鵜呑みにせず`randomUUIDv7()`を手元で試した（非単調クロックの落とし穴つき）
