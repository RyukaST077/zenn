### 裏取りで判明した重要事項（実行前に必ず読む）

- **`using`/`Symbol.dispose` は「Node 26 の新機能」ではない**。Node 26.0.0 リリースノートには記載がなく（Node 26 の目玉は Temporal 既定化・V8 14.6 の Map.getOrInsert / Iterator.concat）、これは **ES2026 の Explicit Resource Management** で V8 側に以前から入っている機能。Node 26 は「現在の既定ランタイム」という位置づけで書く（「Node 26 で追加された」と断定しない）。
- **MDN は `Symbol.dispose` を「Limited availability（Baseline 未到達）」と表記**。ブラウザでは未対応が多い。記事では「ブラウザ標準というより、まず Node/TS で使う機能」と補足する。
- **ローカル実行環境は現在 `v22.17.0`**（`node -v` で確認済み）。レポート前提の Node 26 と食い違う。`using` は Node のバージョンによってはフラグが要る/未対応の可能性があるため、**フェーズ1で必ず実バージョンと機能有無を検証する**（下記 詰まりポイント#1）。
- `Symbol.dispose` の要件（MDN）: 例外を投げると「解放失敗」を意味する / 複数回呼ばれても例外を投げない（冪等）べき / **戻り値に Promise を返してはいけない（`using` は await しない）**→ 非同期解放は `await using` + `Symbol.asyncDispose` を使う。

## 完成イメージ（成果物）

- 作るもの: 1つの検証用ディレクトリに、`Symbol.dispose` / `Symbol.asyncDispose` を実装した小さな自作リソースクラスと、**同じ題材を「try/finally 版」と「using 版」で書いた比較スクリプト群**。実行すると解放ログが標準出力に並ぶ。
- 「できた」と言える完了条件:
  1. `using` 版と try/finally 版が同じ後始末結果になることを、標準出力ログで確認できる
  2. 複数 `using` の **LIFO（後入れ先出し）解放順**がログで確認できる
  3. **例外時・early return 時にも解放される**ことがログで確認できる
  4. `await using` + `Symbol.asyncDispose` の**非同期解放**がログで確認できる
- 完了確認の方法: **CLI 出力（`node xxx.mjs` の標準出力ログ）**。ブラウザ表示は伴わないため Playwright は不要（任意でログ画像を撮る程度）。
- 記事タイトル案（そのまま使える形）:
  1. Node 26 の `using` を try/finally と同じ処理で書き比べてみた
  2. `Symbol.dispose` の解放順・例外時の挙動を実際に動かして確かめた
  3. try/finally から `using` に乗り換えて分かった、後始末の書き味の違い

| 5 | 解放順が直感と逆に見える | `using` は LIFO 解放（後に確保したものから解放）。取得順に解放されると誤解しがち | a/b/c を並べて解放ログ順を実測し、LIFO を目で確認 | 実測ログを貼って「後入れ先出し」を体感で示す |
